# Start/finish platform fit against corridor walls

Date: 2026-06-30. Branch: `fix-start-platform-bug`.

**NOTE:** The code has gone through several rounds of additional polishing and cleanups. This may be partially stale. Much of the core content is still accurate.

## The bug

The start platform's right edge sometimes sat too close to, or through, the corridor ceiling. On a seeded batch, the gap between the platform's top edge and the ceiling went negative in about 3 of 20 seeds (as low as -94px), so the ball could not fit between the platform and the corridor wall.

Cause: the start platform was placed at a fixed height before the corridor was routed. The corridor's first leg descends steeply toward the lower finish, and nothing reserved headroom above the platform. The tightest point was the platform's right edge, where the descending ceiling met the fixed platform.

## The fix

Generate the corridor first, then build both platforms to fit inside with sufficient headroom for the ball above the platform surface and enough clearance to keep the platform body off the floor.

`generateLevel` now picks start and finish route targets, routes the corridor between them, then places the start and finish platforms against the corridor's actual inner wall faces. Previously, we didn't have the wall segments available yet and were trying to calculate using the corridor centerline.

### Geometry helpers

`src/game/geometry.ts` has level-agnostic segment and path helpers: path length, polyline-to-segments, y-at-x interpolation, x-stops within a span, and the y-range of a set of segments over a span.

There are two offset functions with subtle differences:

- `offsetPathByVertexNormal` offsets a routed path along smoothed vertex normals. Used to draw the wall centerlines.
- `offsetPolylineSegments` offsets each segment on its own and reconnects adjacent offsets. Used to build the inner wall faces that we check against when fitting the platforms.

### Corridor data

The private `Corridor` object carries both representations:

```ts
type Corridor = {
  endX: number
  faces: { ceiling: Segment[]; floor: Segment[] }  // inner faces, used to fit platforms
  halfHeight: number
  length: number
  path: Point[]
  startX: number
  walls: { ceiling: Point[]; floor: Point[] }       // offset polylines, emitted as terrain
}
```

`walls` are the polylines emitted as `kind: 'wall'` terrain and drawn. `faces` are the inner physical faces used to fit the platforms. The emitted level also carries `startPlatformIndex` and `finishPlatformIndex` so tests and other code can directly access those terrain pieces, instead of inferring them from shape and position.

### Platform fit rule

For a flat platform candidate spanning `[left, right]` (`platformFitRange`):

```ts
ceilingY = max y of the ceiling faces over [left, right]
floorY   = min y of the floor faces over [left, right]

minSurfaceY = ceilingY + topBuffer
maxSurfaceY = floorY - platformHeight

fits = minSurfaceY <= maxSurfaceY
```

It holds when the platform can sit at a height that clears the ceiling by `topBuffer` while its body stays above the floor.

The finish platform also carries the gameplay invariant that its actual surface must be at least `MIN_FINISH_DROP_FROM_START` below the fitted start platform surface. That invariant is enforced as a minimum finish surface y during fitting.

`flatSpanAroundAnchor` finds the widest span that still fits. It starts at the anchor (the start spawn x, or the finish marker x), grows toward the desired left and right edges using wall-face vertex x's as stops, and accepts each step only when `platformFitRange` reports a fit. For the finish platform, the candidate must also allow the required minimum surface y without pushing the platform body into the floor. When a step overruns, it binary-searches that segment for the farthest fitting cut and stops growing that side. The same fit predicate drives both the search and the final placement.

If no flat span fits (it does not occur in the sweep below, but the case is handled), the platform preserves its ceiling buffer and lets its body reach into the floor rather than clip the ceiling, and a `[level]` warning fires. A platform body merging into the floor is harmless; clipping the ceiling is the bug above.

### Design buffers

The buffers are sized for gameplay and geometry:

- `EDGE_CEILING_BUFFER` is 2.5x the ball height (100px) above the start platform — the ball itself plus gameplay headroom. This is the binding headroom at the start.
- `FINISH_TOP_BUFFER` is the finish post height plus 20px (240px), so the post stays inside the corridor.
- `FINISH_DROP_PLANNING_BUFFER` biases the finish route target lower by the maximum platform center drop. The later fit step still enforces the actual surface-to-surface finish drop, but this planning buffer makes the routed corridor naturally leave room for that invariant.

The wall faces match the drawn and simulated walls closely. Each face segment is exactly the inner edge of the matching Matter rectangle, and the canvas stroke (round joins and caps) follows the same centerlines. The two diverge only at polyline joins, where the connector segment cuts across the inner corner (likely by a few pixels) — well within the buffers above.

## Scope

This covers the start and finish platforms only. Scattered platforms and hazards placement was unchanged: uses corridor centerline model with margin and distance checks. The headroom bug was specific to the start and finish platforms.

## Validation

Run under Node `v24.13.0`. Typecheck, unit tests, lint, and build pass.

The headroom test (`tests/level/platform-fit.test.ts`) asserts the ball fits between the start and finish platforms and the corridor walls, and that the finish platform keeps the required drop below the start platform. The generation produces no warnings across 300 seeds.

The 5000-seed sweep (`tmp/sweep.ts`, gitignored) reports:

- `fits-failing 0/5000`
- `[level] warnings 0`
- min headroom: start 100, finish 240
- min finish drop: 1450
- min width: start 680, finish 413
