# Canvas renderer: Phase 2 stub

A thin placeholder. Phase 2's detailed design happens *after* Phase 1 is implemented — informed by what we measure, what surprises us, and what the Phase 1 visual baseline calls for. This doc captures only what we already know.

## What Phase 2 brings

- **First animation: heartbeat pulse around blob-seeds on each tick.** Single-timestamp pattern, not entity machinery — one `pulseStart: number | null` set on store notify; progress `t` computed each frame. Derives "which tiles pulse" from current store state (no per-instance entity array, no shadow map).

- **Offscreen `OffscreenCanvas` tile buffer.** Forced by the first animation — without it, the renderer would redraw 14k tiles every animation frame just to provide a background. Tile pass writes to the buffer (only on store dirty); frame body blits buffer to main canvas, then draws indicators + animations on top.

- **Continuous rAF while animation is alive.** Phase 1 is rAF-on-dirty (idle when nothing's pending). Phase 2 graduates to "stays alive while a pulse / future animation is in flight."

- **Possibly: split dirty flags.** Tile-dirty (store changed → rebuild buffer) vs. indicator-dirty (hover changed → re-composite without rebuilding buffer). Details depend on Phase 1 measurements.

- **Possibly bundled: exclusion-radius hover preview.** The visible-feature win — a circular halo on hovered placeable tiles during placing phase. May land in this phase or split into a follow-up.

## What Phase 1 will teach us

Phase 2's detailed design will pull from these:

- Real per-frame cost of `drawAllTiles` and the buffer-blit cost.
- Visual baseline — animation needs to land tastefully on top of whatever Phase 1 produces.
- Whether the deferred edge cases from Phase 1 (empty-cache first paint, indicator-only changes) need real handling.

## Out of scope (still, in Phase 2)

- Entity pass / entity array / shadow map. Only built when a per-instance effect (capture flare, projectile) needs per-instance state.
- Everything else in Phase 1's "Non-goals / out-of-scope" list.

## Open questions (resolved during Phase 2 detailed design)

- Pulse visual details: duration, color, expansion curve, behavior when ticks arrive faster than the pulse can finish.
- Exclusion preview: snap or fade? Same color as hover ring or distinct?
- Buffer dirty management: one flag for both buffer rebuild + frame composite, or two?
