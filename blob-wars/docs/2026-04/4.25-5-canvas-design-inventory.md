# Canvas renderer: design inventory

A consolidated catalog of every design consideration we have on the table for the canvas renderer swap. The intent is to capture everything before the focused design phase begins, so nothing important gets lost. Items can be evaluated, trimmed, or reordered later — this is the "comprehensive list" pass, not the "decided plan" pass.

The four prior docs in this date range each cover a slice in depth (`4.25-1` survey of options, `4.25-2` game design pressures, `4.25-3` two-layer hybrid renderer, `4.25-4` DPR / integer cell sizing). This doc references them rather than restating them, and adds the things that have come up in conversation but didn't land in any single doc — most importantly the recognition that hover UI belongs to a third indicator pass distinct from both tiles and entities.

## Direction we've landed on

Canvas is the right foundation for the renderer swap. The case is over-determined once even half of the design pressures in `4.25-2` are on the table as things we might actually explore — larger boards alone eliminates SVG, mid-simulation placement breaks the cheap DOM `canPlant` fix, animations require a frame loop the DOM can't host coherently, and stacking any two of these makes the alternatives unviable. Accessibility is a stated non-goal for the game, which removes the one remaining theoretical advantage DOM had over canvas.

The renderer itself is structured as **three composable render passes** drawing to a shared canvas surface — tiles, indicators, and entities — each with semantically distinct lifecycles and update triggers. We're skipping the dual-renderer migration phase; the swap will land as a single commit, with before/after comparison done by jumping commits in git.

## Key questions that have shaped the exploration

These are the framing questions we kept coming back to. Each one shifted the architecture in a specific way, and the answers are worth keeping in mind during the design phase because they're the "why" behind a lot of the smaller decisions downstream.

1. **How do most game engines model mouse hover UI?** — Across Unity, Unreal, Godot, Phaser, and the way real-time strategy games have traditionally structured this, there's a consistent split between an *indicator/overlay layer* (state-driven, single-instance, condition-based — hover rings, selection rings, range previews, build-placement ghosts) and a *VFX/effects layer* (event-driven, multi-instance, TTL-based — flashes, particles, projectiles). They are not the same abstraction, and trying to model hover UI as a long-lived entity with `duration: Infinity` works mechanically but stretches the entity abstraction past what it's good at. This recognition is what moved us from a two-layer (tile + entity) architecture to a three-pass (tile + indicator + entity) architecture.

2. **Can we design a home-grown canvas renderer that makes it easy to integrate a 3rd-party library like PixiJS later if needed?** — Yes, and the things that preserve that option are simple code-organization choices we'd make anyway. Entities as plain data with a `kind` discriminant rather than class instances with draw methods; positions in grid units rather than device pixels, converted at draw time; the tile pass and entity pass as separate function calls rather than an interleaved scene graph; the store completely decoupled from rendering. With those in place, the transition path is mechanical: bring in Pixi for the entity layer only, leave the tile layer as raw canvas, change only the body of `drawEntity` to construct/update Pixi display objects from the same entity data shapes.

3. **What game design changes might pressure the renderer, and how do they stack?** — This is the question doc `4.25-2` exists to answer. Larger boards, mid-simulation placement, faster tick rates, modest animations, and improved hover effects each pressure the renderer in different ways, but the more important observation is that the pressures *compound* — any one of them alone has reasonable workarounds, but stacking even two narrows the viable architectures sharply. This is the central argument for committing to canvas + the multi-pass structure now rather than waiting for a single specific pressure to force the issue.

4. **Would canvas be overkill if the current game design and visual aesthetics were mostly settled?** — Probably — but with the design pressures in `4.25-2` on the table as exploration candidates, that's not the situation we're in. Canvas will give us a strong foundation to explore different game design aspects with lots of perf headroom. There's also a learning motivation worth naming — getting hands-on with canvas as a rendering tech is its own reason.

5. **What's the cheapest possible perf fix for the placing-phase fan-out, independent of any renderer swap?** — Un-prop `canPlant` from `<Tile>` and gate hover at the board level via a `data-can-plant` CSS attribute on the container. The fan-out goes from ~140ms across 14k tiles to one parent re-render. Documented in `4.25-1` §"Cheap DOM fixes". We considered landing it as a baseline before the canvas swap so any later perf comparison would be apples-to-apples, but with the dual-renderer phase skipped and the case for canvas not gated on a perf comparison, this fix is no longer load-bearing and we'll skip it.

6. **Should effects, hover UI, and tiles all live in the same draw-call abstraction, or are they semantically different things?** — Semantically different, and conflating them creates awkwardness. Tiles are a stateless projection of store state; indicators are state-driven, single-instance, condition-based; effects are time-driven, multi-instance, TTL-based. Each gets its own pass. The differences in lifecycle, instance multiplicity, and trigger source are real, and a single unified abstraction would have to ignore those differences to work, which is the same cost paid every time you touch the system.

7. **Should the renderer keep its own previous-state copy ("shadow map" / `prevState`), and what for?** — Yes for change *classification* (so we can detect that a tile transitioned from `null → player1` versus `player1 → player2` and fire different entity effects accordingly), no for render dirty detection (the store already does this internally via `tilesEqual` and only updates `tileCache` when something genuinely changed). The principle is "shadow what the store forgets, never shadow what the store already remembers." The store remembers *whether* things changed; it forgets *what they were before*. The renderer's local copy fills only the second gap.

8. **How do we get React out of the per-tick path with the smallest defensible change to the existing architecture?** — Subscribe once to the whole store rather than per-tile, set a dirty flag on each notify, do a full-board redraw inside an rAF loop reading `tileCache` directly via `getTileData`. This is the "B1" pattern from `4.25-1`. It's the smallest possible diff against the existing store — `subscribeTile` becomes unused by the renderer (we keep the API in place gated on `tileSubs.size > 0` for future flexibility), and everything else in the store/session/hooks split survives untouched. The store isn't inviolable; store-level changes like B3's dirty-coord set, an event/causality channel for animation triggers, or typed-array backing for very large boards are all reachable when a specific feature or measurement justifies them. The renderer swap itself doesn't require any of those, so we don't pre-emptively make them.

9. **How do we make the renderer DPR-correct on fractional-DPR systems (Windows scaling, browser zoom)?** — Compute cell size in device pixels first, as an integer, derive backing-store size from that, derive CSS size from backing-store size, accept a small gutter where the canvas doesn't quite fill the container. Documented in full in `4.25-4`. The naive approaches both fail at fractional DPR — one with blurriness on retina, the other with visible seams between tiles at any DPR that isn't a clean integer.

10. **What's load-bearing about the existing architecture that any renderer swap must preserve?** — The store/session/hooks split that landed in `4.23-2`, the snapshot-driven model where `applySnapshot` runs synchronously and populates `tileCache` before any subscriber is notified, and the contract that the renderer is just one consumer of `(subscribe, getTileData, version, state, derived)`. The renderer doesn't get to assume anything else, and nothing else has to assume anything about the renderer. This is what made the swap tractable in the first place; preserving it is what keeps future renderer changes (worker, WebGL, framework) tractable too.

11. **When should the renderer infer something itself versus ask the store or protocol for help?** — Renderer-side inference is free (no upstream cost, no protocol changes, no cross-team coordination), so default there. Escalate to store-level or protocol-level only when a specific feature genuinely demands information the renderer can't reconstruct from before/after state — for example, a particle burst that needs to know *which blob* caused a capture, not just that a capture happened. The "renderer-side classification first, escalate only when forced" principle keeps the wire format and store internals stable, and makes it explicit when we're actually paying for new information.

12. **Where does HMR fit, and how much complexity is it worth?** — React-effect-driven init/teardown via `useEffect` is enough. Fast Refresh handles the component lifecycle for free, edits to renderer code cause `BoardCanvas` to re-mount, the effect tears down and re-runs against the current store state. The cost is a sub-frame visual flash on edit, which is acceptable. The fancier alternative (`import.meta.hot.dispose+accept` to preserve in-flight state across HMR) preserves animations through edits but adds non-trivial plumbing for a DX gain that doesn't matter until we have animations worth preserving. We'll take the simple path and revisit only if it ever becomes annoying.

13. **Should we build the cached `OffscreenCanvas` tile buffer on day one, or defer it?** — Architecturally yes, in v1 code no. The buffer is only load-bearing once we have continuous animations running at 60fps and don't want to redraw 14k tiles per frame just to provide a background. Without entities, the tile pass redraws to the main canvas on dirty and the indicator/entity passes are no-ops, which is fine. Adding the buffer later is mechanical (about ten lines: create the offscreen surface, redirect the tile-pass `ctx` reference, blit the buffer at the start of each frame), and we'll have a real measurement to justify it by then.

## Architectural shape

The renderer is structured as a thin React wrapper around an imperative module, with three render passes composed each frame onto a shared canvas surface.

The React wrapper (`board-canvas.tsx`) owns the canvas DOM element and the lifecycle. Its `useEffect` runs `init(canvas, store)` on mount and `destroy()` on unmount, keyed by `matchId` so a new match remounts cleanly. All renderer state — the rAF id, hover state, entity array, shadow maps, layout numbers — lives inside that effect's closure or a single ref. There is no module-level singleton state in the renderer, which is what lets HMR work cleanly without `import.meta.hot` plumbing.

The imperative module (`renderer.ts`) owns initialization, the rAF loop, and teardown. It subscribes once to the whole store, sets a dirty flag on each notify, and runs a frame function that composes the three passes. The frame loop is rAF-on-dirty with a one-shot scheduling pattern — when the store changes, schedule a frame; when nothing's happening and no entities are in flight, no rAF is scheduled at all and the renderer is fully idle. Once continuous animations land, the loop will need to run continuously while entities are alive; we'll switch from one-shot to continuous-gated then.

The synchronous first paint on mount is load-bearing and easy to skip. The order is: apply layout (compute `cellDevicePx` and size the canvas), synchronously paint the current cache state by walking all coords and reading `getTileData` directly, *then* subscribe to the store and start the rAF loop, *then* attach pointer handlers. If subscription happens before the first paint, you race the store; if first paint happens after subscription, you waste a frame. The synchronous paint is what guarantees the canvas isn't blank for ~1 second on `key={matchId}` remount during the simulating phase.

This synchronous-paint guarantee depends on `applySnapshot` running synchronously with respect to the React render that mounts the renderer — the WebSocket message handler runs `applySnapshot` synchronously, which runs `runPipeline` and populates `tileCache`, then notifies subscribers, including the `useMatchId` hook that triggers the chain that mounts the new `BoardCanvas`. By the time the new component's `useEffect` fires, the cache reflects the new match. If `applySnapshot` ever becomes asynchronous (microtask-batched, transferred to a worker, etc.) this guarantee breaks and the renderer would need to handle "cache is empty on first read" by re-painting on the first store notify. Worth a contract comment in `renderer.ts` so a future refactor doesn't silently invalidate this.

The proposed module shape:

```
board-canvas/
  board-canvas.tsx     thin React wrapper, owns useEffect lifecycle and canvas ref
  renderer.ts          init(canvas, store) / destroy(); rAF loop; pass composition
  draw.ts              drawTile, drawIndicators, drawEntity — pure functions
  theme.ts             colors (CSS-var reads at init + seat lookup table)
  layout.ts            cellDevicePx, applyLayout, ResizeObserver, matchMedia listener, eventToCoord
```

## The three render passes

Each pass has its own update trigger, lifecycle model, and conceptual role. Treating them as three semantically distinct things rather than one unified scene graph makes the code read like what it actually does, and makes future additions (selection rings, drag ghosts, blob power glow, capture flares) drop cleanly into the pass that matches their semantics.

**Tile pass.** A pure projection of store state onto the grid. Every visual element maps 1:1 to a grid cell. The draw logic is a function of current store state with no memory of previous frames, no continuous position, no interpolation. It redraws when the store says something changed (`dirty = true` set by the subscription callback, cleared after the redraw completes). At 14k tiles a full redraw is roughly 3–5ms; at 40k tiles roughly 10–15ms, still within budget. If full-redraw becomes a bottleneck on larger boards, the path forward is the B3 variant — `runPipeline` exposes the dirty-coord set it already computes, the renderer redraws only those tiles. We'd add that when profiling forces it, not before.

**Indicator pass.** State-driven, single-instance, computed each frame from current input and store state. Hover state is just `{ coord: Coord | null }` mutated by pointer handlers. The pass reads the hover coord plus the store data for that tile and draws whatever indicators apply: hover ring on placeable tiles, exclusion-radius halo during placing phase, plant-cursor styling, disabled-cell affordance for walls/owned/excluded, future growth-direction arrows. Single-instance is enforced by construction — there's one hovered coord, not a list — and cleanup on pointer-leave is just setting the coord to null. This is the pass that handles things every game engine treats as overlay/HUD work, not as effects or scene-graph entities.

**Entity pass.** Time-driven, free-positioned, lifecycle-managed, multi-instance. Entities are plain data with a discriminated union on `kind`: capture flares, projectiles, floating text, pulse rings, future particle bursts. Lifetimes are independent of tick rate; entities can exist at fractional grid positions and animate continuously between ticks. The pass walks the entity array each frame in reverse order, computes progress `t = (now - startTime) / duration`, splices expired entries, and draws the rest by switching on `kind`. Drawing positions are in grid units, converted to device pixels at draw time using `cellDevicePx` from the layout module. Effects are triggered from a renderer-side change classification step: compare the previous tile's relevant fields (the shadow map) against the new ones, infer the change kind, push a new entity onto the array. This entire pass is a stub in v1 — entities exist as a typed array but the array is empty until we ship the first effect.

## Visual reproduction targets

The tile pass needs to reproduce everything `board.css` does today. The current visual is more than just colors, and a flat `fillRect` per tile would undersell it. The renderer needs to handle empty tiles (rgba background fill), walls (fill plus 1px stroke for the border), player owner color (fill from the seat lookup), origin=seed tiles (a three-layer composition of owner fill, white center arc, and an optional sheen overlay), origin=spread tiles (flat owner fill, no extra layers), exclusion zones for player1/player2/both (alpha overlay tint over the base color, layered correctly when multiple players' zones overlap), and a 1 device-pixel gutter between tiles to match today's CSS `gap: 1px`.

The indicator pass adds the hover states: a seat-tinted border with an inset glow on placeable tiles during the local player's turn, the exclusion-radius halo during placing phase, and the disabled cursor / no-hover behavior for walls, owned tiles, and tiles inside exclusion zones. All of these are pure functions of `(hoverState, store, theme)` and run each frame.

Two compositions need real care: origin=seed (three-layer stack — owner fill, then arc for the white center dot, then optional gradient sheen) and exclusion zones (alpha overlays that need to composite correctly when both players' zones overlap on the same tile). Worth implementing as a single `drawTile(ctx, coord, tile, theme)` function that handles all combinations rather than scattering the cases across the codebase, and verifying the result with side-by-side screenshots against the current DOM renderer before deleting the DOM code.

A sprite atlas pre-baked at init time (one `OffscreenCanvas` holding every "tile species" — player1+seed, player1+spread, exclusion-p1+player1, exclusion-both, exclusion-p1+empty, and so on) would make per-tile draw a single `drawImage` call and could be measurably faster than fill-plus-arc combos. We'll defer this until full-redraw measures slow; B1 with explicit fill calls is the right starting point.

## Cross-cutting implementation details

**DPR and integer cell sizing.** The cell-size-first pattern documented in `4.25-4`. The layout module owns `cellDevicePx` and every pass consumes it; nothing else does its own DPR math. Integer stroke widths for borders and rings (`lineWidth = Math.max(1, Math.round(dpr))`) so the indicator pass doesn't introduce its own anti-aliasing artifacts.

**Re-layout triggers.** `ResizeObserver` on the container handles window resize and parent layout shifts. A `matchMedia('(resolution: ${dpr}dppx)')` listener handles browser zoom and DPR changes from monitor switches; it must re-arm at the new DPR after each fire because the media query is parameterized by the current DPR. Both triggers re-run `applyLayout` and mark the renderer dirty so the next rAF redraws everything at the new size.

**HMR.** React-effect-driven init/teardown. The `useEffect` cleanup must actually clean up — cancel the rAF, remove pointer listeners, drop the canvas ref, clear the entity array. Incomplete cleanup leaves stale rAF callbacks holding references to old store closures, and the symptoms (gradual jank after several edits) are quiet enough to miss until they accumulate.

**Theme and seat colors.** Theme-level CSS custom properties (`--accent-bg`, `--text-soft`, etc.) are read once at init via `getComputedStyle(document.documentElement)`. Seat-scoped colors (the hover ring color and glow that today live in `.board-seat-player1` / `.board-seat-player2`) move to a hardcoded lookup table in `theme.ts` keyed by `PlayerSeat`, populated from the same values currently in CSS. This loses the "single source of truth" property for those specific values, but they're conceptually owned by gameplay rather than by theme; the simpler approach is the right one.

**Shadow map for change classification.** `Map<CoordKey, T>` where `T` carries only the fields needed to classify transitions — for capture detection, just the owner. The map is populated on first snapshot after match mount, with effect-firing suppressed during this initial population so the board doesn't flash 100 simultaneous "growth" effects at match start. Subsequent passes compare `prev = shadow.get(key)` against the current tile, classify the transition, fire effects, and update the shadow. The map is cleared on `matchId` change. Never used for render-dirty detection; that's the store's job.

**Hit-testing.** Coord conversion happens in CSS pixels: read `getBoundingClientRect()` fresh on each event, divide event coords by `rect.width / GRID_WIDTH`, floor to get the integer tile coord. The cell-CSS-px width can be fractional and the math still produces correct tile coords. No need to keep this in sync with `cellDevicePx`.

## Renderer ↔ existing system contract

The loose coupling between the store and the React UI was a deliberate architectural bet and an exploration of state↔UI interfaces — making a canvas swap easier was one of the explicit goals. The bet has paid off enough that the current swap is tractable, but that doesn't make it a commitment to this exact store shape forever. The store contract today — `subscribe`, `subscribeTile`, `getTileData`, `version`, `state`, `derived` — is the working boundary between the renderer and the rest of the system, and the v1 canvas renderer will likely keep most of it unchanged, as it looks like it meets the needs. We'll confirm during the detailed design session.

For the swap itself, the default is to reuse what's working. Sensible store changes are on the table, especially small ones, and the deferred items in §J (B3's dirty-coord set, an event/causality channel, typed-array backing) are the obvious candidates when a specific measurement or feature pulls them in. We just don't bundle store changes into the swap pre-emptively. Take what's working, iterate where iteration is cheap, redesign where redesign is genuinely warranted, and lean toward reuse for v1.

The renderer subscribes once to the whole store (not per-tile) and reads `tileCache` directly via `getTileData`. The `subscribeTile` API stays in place — its only in-tree consumer (`use-tile-data.ts`) goes away, but the API is kept and the dispatch in `runPipeline` is gated on `tileSubs.size > 0` for one extra `if` outside the iteration. This costs effectively nothing and preserves future flexibility for things like a debug overlay or a HUD subscriber that wants to react to single tiles. The seam comment at the top of `board-store.ts` explicitly anticipates this.

After the swap, the following get deleted: `Board.tsx` and `Tile`, `board.css`, `BoardProfiler` and the `reactCommit` perf event it exclusively emits, `use-tile-data.ts`, `use-can-plant.ts` (the canvas reads `session.canPlant()` directly rather than via a hook). `MatchDetails` remains the only React commit on the per-tick critical path — it subscribes to store version and re-renders every tick. The work is trivial today (a handful of text nodes) and not worth optimizing, but worth revisiting if `MatchDetails` ever grows to display anything expensive to compute.

## Perf instrumentation refactor

The renderer module emits a new `canvasDraw` timed event, scoped at minimum to the tile-pass `drawAll` call. We may want a separate event for the full composite (tiles + indicators + entities) once the latter passes do real work, but starting with just `canvasDraw` for tiles is enough for v1.

The `paint` close-event source moves from the store (`board-store.ts:143`'s `rAFEvent('paint', tick)` inside the `onChange` callback) onto the renderer (emit at end of `drawAll`). With the dual-renderer phase skipped, this move can land in the same commit as the canvas swap rather than as a paired refactor. The store-side `rAFEvent` call gets deleted along with the rest of the DOM-mode plumbing.

`formatTickLine` and the `TickRecord` type in `perf-log.ts` need updating to surface `canvasDraw` and to remove the now-dead `commitMs`, `commitBase`, `pipeEnd→commit`, and `commit→paint` fields that depended on the `reactCommit` event. Important sequencing: the `reactCommit` event source (`BoardProfiler`) only gets deleted *after* `canvasDraw` is emitting, otherwise the perf dump silently nulls those fields with no replacement.

## Animation strategy (architected for, deferred from v1)

There are two qualitatively different kinds of animation, and they have different upstream requirements. They're worth distinguishing now so we don't promise more than we can deliver from a single architectural choice.

**Snapshot-driven tweens** are free in the sense that they're purely a renderer concern. The renderer detects a state change by diffing its shadow map against the new tile data, kicks off a 200ms tween over `from-color → to-color`, lerps on each rAF, and idles when all tweens finish. Late snapshots cleanly cancel in-flight tweens by re-anchoring `from = currentInterpolated, to = newTarget`. This handles color fades on owner change, hover ring fade-in, and the exclusion-radius halo fade-in. No store changes, no protocol changes, no upstream cost.

**Event-driven effects** are not free. They require knowing the *kind* of change, not just the before/after state. A capture flash should fire on `seed→capture` but not on `empty→seed`; a particle burst on capture differs from a fade-in on growth. The current snapshot diff doesn't carry this — `runPipeline` knows tile N's data changed but not why. Two paths forward: renderer-side classification (compare shadow against current to infer the change kind, works for transitions deducible from before/after, fails for things that depend on which blob caused a capture), or store/protocol-level events (`runPipeline` exposes a per-tile change kind alongside the dirty set, or the protocol grows a `tileEvents` channel on each `MatchSnapshot`). Start with classification; promote to events only when a specific effect demands information classification can't recover.

Event-driven effects also need a different lifecycle model than tweens: an effect queue with TTL, max-concurrent caps, explicit "let it finish" versus "cancel and re-fire" rules. The cleanly-cancel-late-snapshots pattern that works for tweens doesn't apply — a particle burst that's mid-flight when an opponent recaptures the same tile shouldn't smoothly redirect; it should either finish on its own timeline or drop entirely. This is a separate v1+1 design than the tween model.

There's an open question to settle before shipping animation: when a tile flips owner twice within a single tween window (empty → player1 → player2 within 200ms), does the lerp animate the path (briefly tinting toward player1's color before redirecting toward player2's) or snap to the latest target? Both are defensible; "snap to latest" is the saner default and matches what the cancel-and-re-anchor pattern does naturally.

## Indicator features (v1 candidates)

The indicator pass is where the visible motivation for the swap lives. The DOM renderer can't host most of these without paying for them per-tile.

**Hover ring** replaces the current CSS `:hover` rule. Drawn as a seat-tinted border with an inset glow on the hovered coord when it's placeable during the local player's turn.

**Disabled-cell affordance** turns off the hover ring and changes the cursor for walls, owned tiles, and tiles inside an exclusion zone. The current CSS handles this via the `.tile-disabled` class; the canvas version is one branch in the indicator pass that suppresses the ring and updates the canvas element's `cursor` style.

**Exclusion-radius preview** during placing phase is the headline feature — when the local player hovers a placeable tile, light up the 7-radius circular halo where future seeds would be excluded. This is the moment where the swap visibly pays for itself; it's awkward in CSS and trivial in a renderer that already knows tile coords and exclusion radius.

Future indicators that fit the same pass without adding new machinery: growth-direction arrows on hover (computed from tile adjacency on pointer move, removed on pointer-leave), blob-power visualization (color or border intensity scaled by blob size), drag-to-place ghost (preview where the seed would land while the user drags). All of these are pure functions of `(hoverState, store, theme)`.

## Game design pressures driving the work

The five pressures from `4.25-2`, included here for completeness so the inventory is self-contained.

**Larger boards.** 200×200 starts to thin out DOM's margin; 500×500 is firmly canvas territory; 250k tiles starts to make the store's `Map<CoordKey, TileData>` with string keys matter. Eliminates SVG outright (40k+ DOM nodes break down on Elements panel, memory, compositing). Eventually forces the B3 variant where `runPipeline` exposes a dirty-coord set.

**Mid-simulation placement.** Per-tile placeability instead of phase-level. Breaks the cheap DOM `canPlant` fix (you can't gate 14k tiles' hover affordance with one CSS attribute when each tile's placeability is independent). Canvas handles this naturally via one `getTileData` lookup per pointer move. Also creates the case for the entity pass — a "seed landing" effect needs to overlap with concurrent spread animations.

**Faster tick rate.** At 4Hz the frame budget tightens; at 10Hz the rAF loop must coalesce multiple ticks that arrive within a single frame. Higher rates may also push the wire protocol toward deltas instead of full snapshots. The renderer contract stays the same — store still exposes `getTileData` and `subscribe`, renderer still redraws on dirty. The internal state machinery changes, not the API.

**Modest animations.** Forces continuous rAF while effects are in flight, motivates the cached tile buffer (otherwise we redraw 14k tiles at 60fps just to provide a background for a few animated effects), and validates the tile/entity split.

**Improved hover effects.** Exercises the indicator pass. The exclusion-radius preview is the v1 example; growth arrows and blob-power glow are easy follow-ons. These scale with the visual complexity of each effect, not with board size.

## Deferred / preserved-but-not-built

These are reachable from the architecture without rework, but we're not building them in v1. Each has a concrete trigger that would justify it later.

The B3 variant of the tile pass (`runPipeline` exposes the dirty-coord set, renderer redraws only those tiles) — when full-redraw measurably slows on larger boards. The cached `OffscreenCanvas` tile buffer for the entity pass — when we ship the first continuous animation. A sprite atlas pre-baked at init for per-tile-species draw calls — when full-redraw measurably slows even on 14k tiles. `OffscreenCanvas` plus a Web Worker to run the renderer off the main thread — when main-thread contention shows up in profiles. WebGL with instanced quads — for game #2 territory or 500×500+ boards. PixiJS or another canvas framework for the entity pass — when entity complexity (dozens of simultaneous sprites with textures, blend modes, particle systems) grows past what raw canvas2D handles cleanly. Delta-based wire updates instead of full snapshots — when tick rates make the wire bandwidth or pipeline cost matter. Typed-array store backing keyed by `y * width + x` — at 250k+ tiles where the `Map<CoordKey>` overhead matters. An event/causality channel from the store or protocol — when a specific visual effect demands information renderer-side classification can't recover.

None of these should be built speculatively. Each has a measurable trigger; the architecture leaves the door open for each one.

## Explicit non-goals

These are stated commitments to *not* do something, captured here so they don't drift mid-implementation.

**Accessibility.** No keyboard navigation, no ARIA labels, no focus management, no screen-reader announcements. The 14k-button DOM grid wasn't actually accessible to begin with, and the game's visual nature plus the tile count put it firmly outside the design space where a11y is workable. Stated commitment, not a drift-able default.

**Dual-renderer migration phase.** The DOM renderer and the canvas renderer will not coexist behind a feature flag. The swap lands as a single commit; before/after comparison is done by jumping commits in git. This skips the perf-event sourcing refactor and removes a category of cross-mode bugs from the migration.

**Pre-canvas hedges (cheap DOM `canPlant` fix, G2 SVG spike).** Both were considered as ways to validate or hedge the canvas decision. Neither is load-bearing given that the dual-renderer phase is skipped and the canvas case is over-determined by the design pressures in `4.25-2`. See Q4 and Q5.

**The `import.meta.hot.dispose+accept` HMR pattern.** Preserves in-flight renderer state across edits. We don't have in-flight state worth preserving (no animations yet), and Fast Refresh handles the simpler React-effect pattern for free.

**A CSS probe for seat colors.** Considered as a way to keep CSS as the single source of truth for the seat hover colors. A hardcoded lookup table in `theme.ts` is simpler, and the values are conceptually owned by gameplay anyway.

**A renderer-side shadow map for render-dirty detection.** The store already does this internally via `tilesEqual`. A renderer-side version is a pointless duplicate.

## Small things worth keeping during the swap

A console hook for tile inspection: `window.__board.dumpTile(x, y)` that returns the current `TileData` plus any renderer-side state (active entities at that tile, shadow map value, etc.). This replaces the "open Elements panel and click a tile" debug move that the DOM renderer gave us implicitly. It's not for users — it's for us, during debugging — and it costs about ten lines wired during the renderer's `init`. Plausible follow-ons in the same hook: `window.__board.dumpEntities()` to inspect the live entity array, `window.__board.recordFrame()` for ad-hoc visual debugging.

## Open questions for the design phase

These are the calls that aren't made yet and will need to be made when the design phase lands.

1. **Effect triggering location.** Does the tile pass's draw loop classify changes inline (compare shadow, fire effect, draw tile, all in one walk), or is classification a separate pass that runs before drawing and just queues effects? Probably separate — keeps the tile pass purely about projection and the change-classification logic in one place.

2. **Tile-buffer dirty management** (when we add the buffer). One dirty flag for both the buffer and the next frame, or separate flags? Probably one — store changes mark both dirty simultaneously, and the buffer redraw runs at the start of the frame before the blit.

3. **Persistent indicators with fade in/out.** If the hover ring fades in over 150ms when the cursor enters a tile and out over 150ms on leave, does that lifecycle live in the indicator pass (with a small `fadeT` field on the hover state) or escalate to the entity pass? Probably the indicator pass — fade lifecycle is small enough not to need entity machinery, and re-entering during fade-out should reverse the fade rather than stack a second instance.

4. **Tween conflict resolution.** Owner flips A → B → A within tween window. Default to "snap to latest target, no path animation" for sanity, or animate the path? The snap behavior matches what cancel-and-re-anchor produces naturally; the path behavior would require extra machinery to track tween history.

5. **`canvasDraw` event scope.** Just the tile pass, or the full composite? They're different signals — tile-pass cost is the dominant per-tick number, full-composite cost matters once entity work is real. We probably want both as separate events eventually.

## Risks to watch during implementation

The **synchronous-first-paint contract being silently broken** by a future refactor that makes `applySnapshot` async. Symptom: blank canvas for ~1 second on match remount during simulating phase. Mitigation: top-of-file contract comment in `renderer.ts` calling out the dependency, plus possibly an assertion in dev that `getTileData` returns non-empty cache state during the initial paint.

**HMR cleanup leaks.** Incomplete `useEffect` cleanup leaves a stale rAF callback holding references to the old store closure. New renderer mounts and works; old renderer is also still running. Symptom: gradual jank after several edits. Mitigation: explicit teardown function returned by `init`, reviewed for completeness.

**Tile-buffer staleness** (once we add the buffer). If a code path mutates store state but doesn't notify, the buffer drifts. The current store contract handles this correctly, but a future "imperative tile setter" or any path that bypasses `runPipeline` would break the invariant.

**DPR re-arm forgotten.** The `matchMedia` listener is parameterized by the current DPR; forgetting to re-register at the new DPR after a change means subsequent zooms don't trigger relayout. Symptom: first browser zoom relayouts correctly, second doesn't. Mitigation: the re-arm pattern in `4.25-4`.

**Pixel-space state cached without relayout invalidation.** The architecture reads pixel-space state fresh today (hit-test `getBoundingClientRect`, `cellDevicePx`); any future cache layer would need to flush on `ResizeObserver` / DPR change.

**Shadow-map repopulation race on match remount.** Need to clear the shadow on `matchId` change, repopulate from the first post-mount snapshot, and suppress effect firing during this initial population. Symptom (if missed): hundreds of "capture" or "growth" effects firing simultaneously at match start.

**Entity array growing unbounded.** If creation outpaces TTL expiration (unlikely at human action rates, but possible with rapid-fire ticks during simulation), the array grows without bound. Mitigation: a max-concurrent cap that drops the oldest entity when exceeded.
