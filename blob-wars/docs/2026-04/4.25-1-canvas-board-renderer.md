# Canvas board renderer: exploration

Design exploration for replacing the DOM-based board with a canvas-rendered one. UI chrome (status bar, match details, lobby, etc.) stays in React. Not a decision yet — this is the option-mapping pass before we commit.

## TL;DR

The store/session/hooks split shipped in the perf refactor (`4.23-2-ui-perf-refactor.md`) made the renderer a swappable layer; canvas is the most natural next renderer. Per-tick perf is *not* the reason to switch — we're at p99 32ms on 14,400 tiles, well under the 1s tick budget. The reasons that matter are the visual ceiling (no headroom for animation, organic blob outlines, gradients, hover halos), the DX ceiling (DevTools Elements panel breaks at 14k DOM nodes), and the placing-phase fan-out (~140ms commit when `canPlant` flips), which canvas eliminates as a free side-effect rather than as the primary lever.

The recommended starting point is the smallest swap that delivers the architectural payoff: **a single canvas, full-board redraw on rAF only when the store version changes, reading directly from the existing `tileCache`**. No per-tile subscriptions on the renderer side; no layered canvases yet; no worker; no WebGL. Add complexity only when we have a concrete reason — animation pipeline, larger boards, or a measured frame cost we can't redraw through.

## Current architecture (one paragraph)

The store at `ui/src/board-store/board-store.ts` owns all game state, runs `runPipeline` on every action to refresh a `Map<CoordKey, TileData>` cache, and notifies per-tile subscribers (`tileSubs`) for the tiles whose cached data actually changed. A `<Tile>` React component (`ui/src/components/board.tsx`) subscribes to its own coord via `useSyncExternalStore` and re-renders when its cache entry changes; rendering is purely CSS classes (`tile-player1`, `tile-origin-seed`, `tile-in-exclusion-zone-both`, etc.) plus a `:hover` rule. The board is a CSS grid of 14,400 such divs, keyed by `matchId` so a new match remounts cleanly. Hover, click handling, and accessibility roles all flow through the per-tile DOM nodes today.

## Why consider canvas now

Per-tick rendering is fine. The actual pressure points are:

1. **Visual ceiling.** Every visual flourish today (color, exclusion tint, origin marker) is a CSS class on a div, plus optional pseudo-element. Anti-aliased blob outlines, soft borders, animated growth between ticks, exclusion-radius preview on hover, particle bursts on capture, gradients keyed off blob power — every one of these compounds the per-tile DOM cost. Canvas treats them as drawing, not as reified objects.

2. **Animation between ticks.** Server ticks at 1Hz. Tiles snap from one state to the next; this looks choppy and undersells the simulation. Tweening a captured tile from old color → new color over ~200ms in canvas needs an rAF loop and zero React work. Doing it via CSS transitions on 14k divs isn't free, and CSS transitions don't compose well with React's mount-on-key reset.

3. **DX ceiling.** Chrome's Elements panel synchronously walks the DOM when opened. On 14,400 nodes that's seconds of jank — already burned us once during perf chase (`4.24-1-ui-perf-followup.md`). Canvas reduces the board to a single node.

4. **Placing-phase fan-out.** When `canPlant` flips (every placement, every turn boundary, the placing→simulating transition), all 14,400 `<Tile>` instances re-render because `canPlant` is a prop that breaks `React.memo`. ~140ms commit in dev, noticeable. Canvas doesn't propagate "props" through tiles, so this disappears for free. The same fix exists DOM-side (read `canPlant` per Tile via `useSyncExternalStore`), but we'd be paying 14k subscriptions instead of one — worth knowing the canvas swap dissolves the problem rather than displacing it.

5. **Headroom for bigger boards.** 200×200 (40k tiles) starts to tip over the DOM. 500×500 is firmly canvas territory. We don't need this today, but we shouldn't paint ourselves into a corner.

What we are *not* claiming as motivation: per-tick speed. We're already fast enough.

## What changes vs. what survives

The store's interface (`subscribe`, `subscribeTile`, `getTileData`, `version`, `state`, `derived`) is the contract. Everything that currently consumes that contract will keep consuming it; the renderer is one consumer among many.

| Layer | Survives unchanged | Changes |
|---|---|---|
| Server / protocol | ✅ | — |
| `board-store/*` | ✅ store, derived, tileCache, runPipeline | likely retire `subscribeTile` use from the renderer (keep the API) |
| `session/*` | ✅ | — |
| `hooks/use-status`, `use-current-user`, `use-match-id`, `use-phase`, `use-can-plant` | ✅ | — |
| `hooks/use-tile-data` | ❌ | unused after canvas swap; delete or mark unused |
| `App.tsx`, `MatchContent`, `MatchDetails`, `StatusBar`, `HomeContent` | ✅ | — |
| `Board` component | ❌ | replaced by `BoardCanvas` (thin React wrapper around imperative renderer) |
| `Tile` component | ❌ | deleted |
| `board.css` | ❌ | replaced by JS color constants in renderer module |
| `BoardProfiler` | ❌ | obsolete (no React commits to profile) |
| `perfLog` instrumentation | ✅ | `tileRenders` becomes a no-op or repurposed; tick log otherwise unchanged |

The interesting line is `subscribeTile`. It exists because each `<Tile>` is its own React component; the per-tile sub is what lets one tile's flip avoid a re-render of its 14,399 neighbors. A canvas renderer doesn't have that problem natively — it has one consumer, one subscription, one redraw. We don't need `subscribeTile` for canvas. We do need it if there's ever a non-canvas thing that wants to react to a single tile (HUD overlay, accessibility cursor, debug panel). The honest answer is *keep the API, stop using it from the renderer*. Cheap to keep; awkward to bring back if we delete and need it later.

## Architectural options

I'll lay out six concrete options across three rendering technologies and a few orthogonal patterns. Each option below is a complete, shippable v1 — they're not stages of a single plan.

### Option A — One canvas, per-tile subs drive a dirty set

Keep `subscribeTile` usage. On mount, the renderer subscribes to all 14,400 coords; each callback adds the coord to a `Set<CoordKey>`. An rAF loop drains the set, computes pixel rects, and `clearRect` + `fillRect` (or `drawImage` from a sprite atlas) per dirty coord.

```ts
// sketch
const dirty = new Set<CoordKey>();
const unsubs = coords.map(c => store.subscribeTile(c, () => dirty.add(serializeCoord(c))));
function frame() {
  if (dirty.size > 0) flush(dirty);
  dirty.clear();
  requestAnimationFrame(frame);
}
```

**Pros.** Smallest diff to the store (zero changes). The per-tile diff already does the dirty detection upstream; we're just bridging it to canvas. Plays nicely with future per-tile consumers (HUD overlay, debug picker).

**Cons.** Pays for 14,400 subscription objects (Set entries) for no benefit — the canvas renderer is the single subscriber that matters. The renderer carries a coord-iteration ceremony on init/teardown that buys nothing. Resubscribing on `matchId` change is a Set-of-Sets dance.

### Option B — One canvas, version-driven, reads `tileCache` directly  *(recommended starting point)*

Subscribe once to the whole store. On every notify, mark `dirty = true`. An rAF loop checks `dirty`, then walks the cache and draws. Two sub-variants:

- **B1 (full-redraw).** On `dirty`, redraw all 14,400 cells from the cache. Canvas2D fillRects over 14k tiles is ~3–5ms — well within budget at 1Hz, and trivially correct.
- **B2 (cache-diff).** Keep a `Map<CoordKey, TileVisualKey>` shadow on the renderer side. On redraw, walk the cache, compare each tile's "visual key" (a tiny string or packed int encoding owner|origin|terrain|exclusionSource|isPlaceable) against the shadow, and only draw cells whose key changed.

```ts
// sketch (B2)
let dirty = true;
store.subscribe(() => { dirty = true; });
function frame() {
  if (dirty) {
    for (const coord of allCoords) {
      const tile = store.getTileData(coord);
      const key = visualKeyOf(tile);
      if (shadow.get(coord) !== key) {
        drawTile(ctx, coord, tile);
        shadow.set(coord, key);
      }
    }
    dirty = false;
  }
  requestAnimationFrame(frame);
}
```

**Pros.** Drops `subscribeTile` from the renderer entirely. One subscription, one rAF, simple lifecycle. B1 is dead-simple and probably fast enough. B2 trades simplicity for asymptotic perf if we ever scale past 14,400.

**Cons.** Renderer holds a shadow cache (B2) that's a parallel structure to the store's `tileCache`. Two diff layers feels redundant — though the shadow's key is a *visual* projection (color identity), not the structural `TileData`, so they aren't quite the same thing. Still: there's a temptation to dirty-track in the store and skip the shadow.

**Variant B3 — runPipeline emits a dirty set.** Modify `runPipeline` to expose the changed coords (it already counts them; exposing the keys is a few lines). Renderer reads the dirty set off the most recent emission. Eliminates the shadow cache entirely. Cleanest version of B; mild contract leak from store to renderer (the store now broadcasts dirty coords, not just "something changed"). I think B3 is the right end-state, but B1 is the right *starting* state — ship the simple version, measure, only specialize if measurement says so.

### Option C — Layered canvases + sprite atlas + DOM hover overlay

Stack three or four `<canvas>` elements in a positioned container:

1. **Static layer.** Walls and the board grid lines. Drawn once on mount; redrawn only on dimension/match change.
2. **Owner layer.** Tile fills, origin marks, exclusion tints. This is what the per-tick redraw operates on.
3. **Effects layer.** Hover ring, plant-cursor, future animations (growth particles, capture flashes).
4. **Optional DOM overlay.** A single transparent div with `pointermove` / `pointerdown`, mapping screen coords → tile coords math, dispatching to renderer state.

Pre-bake "tile species" sprites (player1+seed, player2+spread, exclusion-p1, exclusion-both, etc.) onto an offscreen atlas at mount time. Per-tile draw is then a single `drawImage(atlas, sx, sy, …, dx, dy, …)` — measurably faster than fill + radial-gradient combos.

**Pros.** Each layer redraws on its own cadence. Effects layer can run rAF at 60Hz for animation while owner layer only redraws on tick. Sprite atlas is the standard 2D-game approach and unlocks animation cells trivially. Cleanest path to "interesting visuals".

**Cons.** More code: layer composition, sprite-atlas generation, layer-specific dirty tracking. Compositing N canvases costs something in the GPU compositor (small but non-zero). DPI scaling has to be reapplied per layer. Overkill if we never want animation.

### Option D — OffscreenCanvas + Web Worker

The renderer module runs in a worker. Main thread postMessages snapshot diffs (or whole snapshots; they're <1MB JSON for 14k tiles). Worker derives, draws to its `OffscreenCanvas`, and the main thread sees pixels appear. Main thread keeps `pointermove` / `click` and forwards as messages.

**Pros.** Removes *all* render work from the main thread. Future-proofs against animation, larger boards, and any main-thread contention. The store's purity makes this almost a pure transport problem — the worker hosts an instance of `createBoardStore` and applies snapshots locally.

**Cons.** Real complexity: two store instances (or one in the worker only), serialized message protocol, `transfer` semantics for `OffscreenCanvas`. Debugging is harder. Hot reload is harder. We don't have a measured main-thread bottleneck that justifies this. OffscreenCanvas is supported across modern browsers but adds another portability axis to remember.

Worth keeping in our back pocket; not worth doing now.

### Option E — WebGL with instanced quads

One quad mesh, instanced 14,400 times. Per-instance attributes: tile index (→ position), color, origin marker flag, exclusion flag. Single draw call per frame; updates are `bufferSubData` patches into the attribute buffer.

**Pros.** Fastest possible. Scales to 500×500+. Zoom/pan/effects/particles are native. Future-proofs everything.

**Cons.** Order-of-magnitude more complexity. Shader code, GL state machine, attribute layouts, framebuffer fiddling. Adds a tech to the stack that nobody else here is using. We don't need it. The "blobby organic" visual direction (gradient borders, soft outlines) is actually *easier* in canvas2D than in shader land, despite WebGL's raw speed.

A WebGL renderer is the right answer for game #2 if it's a real-time RTS or a 200×200+ board. For blob-wars at 1Hz, it's the wrong tool.

### Option F — Hybrid: canvas for visuals, single DOM overlay for input

This is more of a *coding pattern* than a separate option — it can be combined with A/B/C. Worth naming because the alternative (input handling on the canvas itself) is also valid and the choice has UX implications.

**F1: input on canvas.** Attach `pointermove` / `pointerdown` directly to the `<canvas>`. Coord math: `(event.offsetX / cellSizePx, event.offsetY / cellSizePx)`. Hover state is renderer-side; redraw the hover ring on rAF.

**F2: input on DOM overlay.** A single transparent `<div>` over the canvas with `pointer-events: auto`. Same coord math. Lets us put accessibility roles (`role="application"`, ARIA labels, keyboard focus) on a real DOM element without the 14k-divs cost.

I'd default to F1 for v1 (simpler) and graduate to F2 if accessibility or focus management warrants it. Either way the renderer doesn't change.

## Cross-cutting decisions

These apply to whichever option we pick.

### Frame loop: rAF-on-dirty, not always-on

Always-running rAF burns power and shows up in CPU profiles even when nothing's happening. The right pattern is `dirty = true` on store change; `if (dirty) draw(); dirty = false;` inside an rAF that only runs when needed. Two ways to "only run when needed":

- **Continuous rAF, gated on flag.** Simplest; the rAF loop runs forever but is a no-op when clean. Costs ~0 CPU.
- **One-shot rAF on dirty.** Schedule rAF only when a change arrives; renderer is fully idle otherwise. Cleanest; slightly more code.

Once we have animations, the loop must run continuously anyway. So: start with one-shot on dirty (today's reality has no animation); switch to continuous-gated when animation lands.

### Hit-testing: math, not pixel readback

Tile size is fixed (board is a CSS grid in CSS px). Map pointer event to coord with floor division. No need for `getImageData` tricks or hit regions.

```ts
const x = Math.floor(event.offsetX / cellSizePx);
const y = Math.floor(event.offsetY / cellSizePx);
```

The two gotchas are (a) DPR (the canvas backing-store is `cellSizePx * dpr`, but `event.offsetX` is in CSS px, so the math uses CSS px — easy to get wrong if you scale wrong) and (b) sub-pixel rounding on resize.

### DPR (devicePixelRatio) handling

This is where canvas renderers most often look bad on retina. The pattern:

- Backing-store size = CSS size × DPR. Set via `canvas.width` / `canvas.height` (the attributes, not CSS).
- Display size = CSS px. Set via `canvas.style.width` / `style.height` (or via container layout).
- Apply `ctx.setTransform(dpr, 0, 0, dpr, 0, 0)` once after sizing so all draw commands speak CSS px.
- On `ResizeObserver` change *and* on `window.devicePixelRatio` change (zoom), re-apply.

`matchMedia(`(resolution: ${dpr}dppx)`)` events fire on DPR change; observe and re-init.

### Sizing: ResizeObserver-driven, container-bounded

The current board uses `aspect-ratio: 1; flex: 1 1 0` and lives inside a flex column. Canvas mode preserves the same outer container; only the inner element changes. ResizeObserver on the container, recompute `cellSizePx = floor(containerWidth / GRID_WIDTH)`, redraw. Round to integer to avoid sub-pixel seams between tiles.

### Hover, exclusion preview, and the click-affordance

DOM gives us `:hover` for free. Canvas requires us to track hover state and redraw it. Two upgrades fall out of doing it ourselves:

- **Exclusion-radius preview.** When the local player hovers a placeable tile during placing phase, light up the 7-radius circular halo where future seeds *would* be excluded. Trivial in canvas; awkward in CSS. Concrete win for the swap.
- **Disabled-cell affordance.** The current CSS turns off hover for disabled tiles; canvas does the same with one branch in the hover renderer.

I'd treat exclusion preview as a v1 nice-to-have (cost: ~1 evening) since it directly motivates the swap.

### Animation strategy (when we want it)

Most natural pattern: per-tile *transition state* held in the renderer (not in the store). When the renderer detects a tile changed `owner` from `null → player1`, it kicks off a 200ms tween (start time, from-color, to-color). On each rAF, lerp and draw. When all transitions are done, mark clean and stop animating.

Critical property: animations live entirely in the renderer. The store remains snapshot-truth; the renderer just smooths the path between snapshots. This means:

- Animations don't touch the protocol.
- Animations don't add per-tick store work.
- Late-arriving snapshots cleanly cancel in-flight tweens (`from = currentInterpolated, to = newTarget`).

Animation isn't v1. But the architecture should leave the door open by making the renderer module the obvious place to add it later.

### Color & theme

CSS custom properties (`--accent-bg`, `--text-soft`, etc.) drive the rest of the app. The canvas renderer needs equivalent values in JS. Two options:

- **Read from CSS at init.** `getComputedStyle(document.documentElement).getPropertyValue('--accent-bg')`. Stays linked to theme.
- **Hardcode constants.** Faster, but theme drift becomes a possibility.

Pick "read from CSS at init" — keeps a single source of truth, costs nothing, and supports a future light/dark toggle.

### Dimension changes mid-session

Today the grid size is fixed for a session (architectural assumption documented in `board-store.ts`). The renderer can rely on this — capture `width`/`height` once on init, never recheck. If we ever support resize-mid-session, it'd be a remount via React `key`, same as `matchId`.

## What we lose

Honest list:

- **Free `:hover`.** Reimplemented in code; not free, but bounded.
- **React DevTools Profiler view of the board.** Already noisy, already deleted by the perf-debug gate. Not a real loss.
- **Per-tile RTL tests.** The renderer module is harder to test than the DOM tile. Unit testing stays at the store / pure-function layer (where the actual game logic lives); the renderer becomes a "verify by playing" thing or a pixel-diff thing if we ever want it.
- **Implicit DOM accessibility.** A 14k-button DOM grid wasn't actually accessible — no screen-reader user is navigating that. Canvas with explicit keyboard support (arrow-keys to move a focus cursor, Enter to plant, ARIA live region for state changes) is *better* a11y than the current implementation, but it's now explicit work we own.
- **The "open Elements panel and inspect a tile" debugging move.** Replaced by `window.__board.dumpTile(x, y)` style debug hooks if we want them. Probably worth wiring during the swap.

## What we gain

- **Visual headroom.** Animation, gradients, organic outlines, hover effects, particle bursts — each is local to the renderer module instead of compounding the DOM cost.
- **Placing-phase fan-out goes to zero** without further work, as a free side-effect.
- **DevTools Elements panel works again.**
- **Larger boards become tractable.** If we ever want 200×200, we don't have to redo the renderer.
- **Bandwidth for "second opinions"** — exclusion-radius preview, blob-power visualization, growth direction arrows. Things we can't add to DOM without paying again.

## Recommendation

Start with **Option B1** (one canvas, full-board redraw on dirty rAF, reads `tileCache` directly), pattern **F1** (input on canvas), with cross-cutting decisions as listed. Concretely:

```
ui/src/board-canvas/
  board-canvas.tsx      thin React wrapper: <canvas ref=…> + useEffect that wires the renderer
  renderer.ts           imperative module: init(canvas, store), destroy(), the rAF loop
  draw.ts               pure draw functions: drawTile(ctx, coord, tile, theme), drawHover(...)
  theme.ts              read CSS vars at init, expose typed colors
  layout.ts             cellSizePx, screen→coord math, DPR setup
  hit-test.ts           pointer event → coord
```

`MatchContent` swaps `<Board>` for `<BoardCanvas>` keyed by `matchId`. `Board.tsx`, `Tile`, `board.css`, `use-tile-data.ts`, `BoardProfiler` get deleted (or marked unused) once the swap proves out.

Migration plan:

1. **Land B1 alongside the DOM board behind a flag** (`?renderer=canvas` query param or a build-time const). Both renderers running off the same store proves the contract.
2. **Verify perf parity or better.** `perfLog` now counts no `tileRenders`; `wsRecv→paint` should be flat or improved. Sanity check: a manual board redraw from the cache should land in <5ms. Spot-check on placing phase: commit cost should drop materially when canPlant flips since no React tree depends on it.
3. **Add exclusion-radius hover preview.** This is the visible-win moment for users; it's also a real exercise of the renderer's hover-state path.
4. **Delete the DOM renderer.** Drop `Board.tsx`, `Tile`, `board.css`, `use-tile-data.ts`. Audit `subscribeTile` callers; if none remain, leave the API in place but note in `board-store.ts` that no in-tree consumer uses it.
5. **(Later, optional)** Graduate B1 → B3 (`runPipeline` exposes a dirty-coord set) if profiling on larger boards calls for it. Graduate to layered canvases (Option C) when animation lands. Consider OffscreenCanvas (Option D) only if main-thread contention shows up.

Why B1 over B3 as the *starting* point: B3 leaks store internals (a "dirty coords since last drain" channel) before we know we need them. B1 stays inside the existing contract and leaves room for B3 as a clean extension. Forward door open, no premature commitments.

Why not C from the start: layered canvases shine *with* animation. Without animation (where we are today), the second canvas is overhead with no payoff. Right tool, wrong day.

Why not D or E ever, until proven: D adds a worker boundary we don't need; E adds a rendering tech the rest of the project isn't paying for. Both are reachable from B1.

## Open questions / things to validate before committing

1. **What's the actual visual direction?** If "we want growth animations and organic blobs" is real, Option C is closer to right than B. If the answer is "v1 looks like today, just on canvas", B1 is unambiguous. Worth a brief design conversation before coding.
2. **Do we want to keep `subscribeTile` in the public store API?** Cheap to keep; no in-tree consumer once the swap completes. I lean keep-and-document.
3. **DPR on weird displays.** Need to spot-check on a non-1×, non-2× display (1.5×, 2.5× — common on Windows). Most canvas perf reports come from retina-or-FHD; the wonky ratios are where seam artifacts hide.
4. **Will `perfLog` instrumentation still tell us what we want?** `tileRenders` is dead; `wsRecv→paint` still meaningful (paint event still fires). `commit` from `BoardProfiler` is gone. Probably want to add a `canvasDraw` timed event so we keep visibility.
5. **Accessibility floor.** What's the minimum we want? My guess: "renders a label/state announcement for the local player's turn", nothing fancier. Worth confirming we're not signing up for full keyboard navigation as a v1 deliverable.
6. **Match remount cost.** Today, `key={matchId}` cleanly remounts the board. With canvas, "remount" means tearing down the renderer (cancel rAF, clear hover state, optionally `clearRect`) and re-initing. The React key idiom still works since `<BoardCanvas>`'s `useEffect` cleanup runs on unmount. Verify: no flash, no leaked rAF.
7. **Hot-reload behavior.** Vite HMR replacing `renderer.ts` should tear down the old loop and start the new one. Document in code; HMR is a real DX feature here.
