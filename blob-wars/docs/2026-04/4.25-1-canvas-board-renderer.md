# Canvas board renderer: exploration

Design exploration for replacing the DOM-based board with a canvas-rendered one. UI chrome (status bar, match details, lobby, etc.) stays in React. Not a decision yet — this is the option-mapping pass before we commit.

## TL;DR

The store/session/hooks split shipped in the perf refactor (`4.23-2-ui-perf-refactor.md`) made the renderer a swappable layer; canvas is the most natural next renderer. **Perf is not the reason to switch.** Per-tick steady-state is fine (p99 32ms on 14,400 tiles); the only painful perf number — the placing-phase fan-out where `canPlant` flips and re-renders all 14,400 Tiles — has a cheap DOM-side fix that should land regardless (see §"Cheap DOM fixes" below). The honest case for canvas is two things only:

- **Visual headroom that's render-only** — color tweens on owner change, hover halos, exclusion-radius preview, organic blob outlines via boundary paths, gradients keyed off blob power. All achievable purely in the renderer.
- **DX** — DevTools Elements panel breaks at 14k DOM nodes.

Other visual ambitions (particle bursts on capture, change-cause-specific effects) require upstream work in the store or protocol — see §"Animation strategy" — and aren't free side-effects of the renderer swap.

Both depend on a visual-direction question that has not been answered yet — do we actually want animation/organic visuals, or is "v1 looks like today, just on canvas" the goal? The recommendation below is conditional on that answer.

The recommendation is two-step:

1. **Land the cheap DOM fixes** (un-prop `canPlant` via a board-level CSS attribute) — independent of any renderer choice; lands on its own merit; ~1 hour.
2. **Spike Option G2 (imperative SVG)** for ~1 hour to test whether SVG's free `:hover` and free CSS color transitions deliver the visual direction at lower complexity than canvas. **Decisive exit criterion: does Elements-panel-on-G2 behave materially better than today's 14k-div tree?** If yes, G2 is the answer and canvas is over-engineering. If no (the more likely outcome — Chrome serializes any DOM tree on Elements expansion), canvas is the answer and the SVG case collapses to "free color transitions," which doesn't outweigh the lost DX win.

If the spike points to canvas, the right starting point is the smallest one that delivers the architectural payoff: **a single canvas, full-board redraw on rAF only when the store version changes, reading directly from the existing `tileCache`** (Option B1 below). No per-tile subscriptions on the renderer side; no layered canvases yet; no worker; no WebGL. Add complexity only when we have a concrete reason — animation pipeline, larger boards, or a measured frame cost we can't redraw through.

## Current architecture (one paragraph)

The store at `ui/src/board-store/board-store.ts` owns all game state, runs `runPipeline` on every action to refresh a `Map<CoordKey, TileData>` cache, and notifies per-tile subscribers (`tileSubs`) for the tiles whose cached data actually changed. A `<Tile>` React component (`ui/src/components/board.tsx`) subscribes to its own coord via `useSyncExternalStore` and re-renders when its cache entry changes; rendering is purely CSS classes (`tile-player1`, `tile-origin-seed`, `tile-in-exclusion-zone-both`, etc.) plus a `:hover` rule. The board is a CSS grid of 14,400 such divs, keyed by `matchId` so a new match remounts cleanly. Hover, click handling, and accessibility roles all flow through the per-tile DOM nodes today.

## Why consider canvas now

Per-tick rendering is fine. The actual pressure points are:

1. **Visual ceiling.** Every visual flourish today (color, exclusion tint, origin marker) is a CSS class on a div, plus optional pseudo-element. Anti-aliased blob outlines, soft borders, animated growth between ticks, exclusion-radius preview on hover, particle bursts on capture, gradients keyed off blob power — every one of these compounds the per-tile DOM cost. Canvas (or SVG; see Option G) treats them as drawing, not as reified objects.

2. **Animation between ticks.** Server ticks at 1Hz. Tiles snap from one state to the next; this looks choppy and undersells the simulation. Tweening a captured tile from old color → new color over ~200ms in canvas needs an rAF loop and zero React work. Doing it via CSS transitions on 14k divs isn't free, and CSS transitions don't compose well with React's mount-on-key reset.

3. **DX ceiling.** Chrome's Elements panel synchronously walks the DOM when opened. On 14,400 nodes that's seconds of jank — already burned us once during perf chase (`4.24-1-ui-perf-followup.md`). Canvas reduces the board to a single node. (SVG has the same Elements-panel risk in principle but in practice the panel collapses SVG children differently — worth measuring rather than assuming.)

4. **Headroom for bigger boards.** 200×200 (40k tiles) starts to tip over the DOM. 500×500 is firmly canvas territory. We don't need this today, but we shouldn't paint ourselves into a corner.

What we are *not* claiming as motivation:
- **Per-tick steady-state speed.** We're already fast enough.
- **Placing-phase fan-out** (~140ms commit when `canPlant` flips). The follow-up doc costed this at half a day to fix in DOM by un-propping `canPlant`; on closer look it's even cheaper than that — see "Cheap DOM fixes" below.
- **Placing→simulating boundary tick** (~190ms `wsRecv→paint` in clean baseline; up to ~460ms with DevTools open). Same root cause as #2 and same DOM-side fix.

In other words: every perf concern in the current data is either fixable in DOM cheaply or is a measurement artifact (DevTools-on multiplier). Once those land, the case for canvas reduces to the visual-direction question and the Elements-panel DX argument. The recommendation here is conditional on that.

## Cheap DOM fixes that obviate part of the case

Before committing to a renderer swap, two small DOM-side changes erase most of the "perf" arguments. They should land first regardless of whether canvas happens.

**1. Un-prop `canPlant`.** Today, `<Board>` reads `canPlant` and passes it as a prop to every `<Tile>`. The flip on every placement / turn boundary / phase transition invalidates `React.memo` for all 14,400 tiles → ~140ms commit. But `canPlant` only affects one thing on the rendered output: it gates the `.tile-disabled` class, which suppresses the `:hover` CSS rule and changes the cursor. Two ways to detach it from per-tile renders:

- **Option 1 — gate hover at the board level via CSS attribute.** Put `data-can-plant="true|false"` on the `.board` container. Add CSS rules like `.board[data-can-plant="false"] .tile:hover { ... no-op styling }`. The `<Board>` renders when the attribute changes (still calls `useCanPlant`), but `<Tile>` no longer reads `canPlant` and its prop set goes fully stable, so `React.memo` short-circuits the 14k re-renders. Concrete Tile diff:
  ```diff
  -const Tile = memo(function Tile({ store, x, y, canPlant, plant }: TileProps) {
  +const Tile = memo(function Tile({ store, x, y, plant }: TileProps) {
     const tile = useTileData(store, { x, y });
     const isWall = tile.terrain === "wall";
  -  const disabled = isWall || tile.owner !== null || tile.insideExclusion || !canPlant;
  +  const disabled = isWall || tile.owner !== null || tile.insideExclusion;
     // ... className logic unchanged for tile-disabled (cursor + no-hover for walls/owned/excluded)
     onClick={disabled ? undefined : () => plant(x, y)}
  ```
  `disabled` still gates `onClick` for walls/owned/excluded (the cases that survive across turns). The `canPlant` flip is now a CSS-only effect at the board level, no React commit fan-out. (Load-bearing: `session.plant` is referentially stable from the perf refactor — closure created once inside `createSession` against the module-level singleton — so the remaining Tile props are fully stable across parent re-renders.) ~1 hour.
- **Option 2 — drop the `disabled`/canPlant interaction entirely; check at click time.** `session.plant()` already short-circuits when `canPlant` is false (`session.ts:74–82`). Same Tile diff as Option 1 plus drop the `disabled ? undefined : ...` click gate (always wire `onClick` to `plant`). The `tile-disabled` class still applies for walls/owned/excluded (those gate via `tile` data, not `canPlant`). Trade-off: hover affordance shows on placeable tiles even on the opponent's turn, which is slightly less informative UX. ~30 minutes.

I'd take Option 1 — it preserves UX and has no fan-out cost. Either way, the fan-out goes to zero.

**2. Move the `wsRecv→paint` boundary outlier to its real source.** The 190ms boundary tick is the same `canPlant` flip on the player who held `canPlant=true` at the moment simulating started. Fixing #1 fixes this. No additional work.

After both: the only remaining perf number worth caring about is DevTools-open dev-mode behavior, which is a tooling concern (and a real one), not a renderer concern. Canvas helps it but isn't the only path; deleting the 14k-node DOM via canvas is the most direct response.

## What changes vs. what survives

The store's interface (`subscribe`, `subscribeTile`, `getTileData`, `version`, `state`, `derived`) is the contract. Everything that currently consumes that contract will keep consuming it; the renderer is one consumer among many.

| Layer | Survives unchanged | Changes |
|---|---|---|
| Server / protocol | ✅ | — |
| `board-store/*` | ✅ store, derived, tileCache, runPipeline | likely retire `subscribeTile` use from the renderer (keep the API) |
| `session/*` | ✅ | — |
| `hooks/use-status`, `use-current-user`, `use-match-id`, `use-phase` | ✅ | — |
| `hooks/use-can-plant` | ❌ | only consumer is `<Board>`; canvas reads `session.canPlant()` directly. After swap, no in-tree consumer — delete |
| `hooks/use-tile-data` | ❌ | unused after canvas swap; delete |
| `App.tsx`, `MatchContent`, `MatchDetails`, `StatusBar`, `HomeContent` | ✅ | — |
| `Board` component | ❌ | replaced by `BoardCanvas` (thin React wrapper around imperative renderer) |
| `Tile` component | ❌ | deleted |
| `board.css` | ❌ | replaced by JS color constants in renderer module |
| `BoardProfiler` | ❌ | obsolete (no React commits to profile) — **only delete after `canvasDraw` event lands** (see migration plan); deleting earlier silently nulls `commitMs`/`commitBase`/`pipeEnd→commit`/`commit→paint` in the perf dump (these are derived from the `reactCommit` event that `BoardProfiler` exclusively emits) |
| `perfLog` instrumentation | ✅ contract | `tileRenders` becomes a no-op; gain a new `canvasDraw` timed event; the `paint` close-event source moves from `board-store.ts:143`'s `rAFEvent('paint', tick)` to the renderer (canvas emits after `drawAll` completes; DOM emits from a `useLayoutEffect` if we keep DOM around in dual-renderer phase). See migration step 4 for sequencing. |

The interesting line is `subscribeTile`. It exists because each `<Tile>` is its own React component; the per-tile sub is what lets one tile's flip avoid a re-render of its 14,399 neighbors. A canvas renderer doesn't have that problem natively — it has one consumer, one subscription, one redraw.

There's a small cost to keeping `subscribeTile` "alive but unused": `runPipeline` calls `tileSubs.get(key)` for every *changed* tile (`board-store.ts:113–128`; the dispatch is inside the `!tilesEqual(prev, next)` branch, so only changed tiles incur it — typically 6–170 per tick during simulating). With zero subscribers, those Map lookups return `undefined` and accomplish nothing — small, but unprincipled. So this isn't quite "keep the API for free" — it's "keep the API and the dispatch path for free." Two coherent options:

- **Delete the API.** Drop `subscribeTile`, `tileSubs`, the dispatch in `runPipeline`, and `use-tile-data.ts`. The architecture survives — bringing it back later is mechanical (the seam comment at `board-store.ts:1–11` explicitly anticipates this).
- **Keep the API, gate the dispatch on `tileSubs.size > 0`.** Hoist the size check outside the iteration; one extra `if`. Avoids the per-changed-tile lookup when no one's listening.

Default: keep the API, gate the dispatch. The arithmetic is small (typically 6–170 saved Map lookups per tick), but the gate is principled — paid for in one extra `if`. Future flexibility is real (a future "render only this tile to the HUD" or debug-overlay wouldn't require touching the store). Delete only if a second renderer arrives that doesn't want it either.

## Architectural options

I'll lay out six concrete options across three rendering technologies and a few orthogonal patterns. Each option below is a complete, shippable v1 — they're not stages of a single plan.

### Option A — One canvas, per-tile subs drive a dirty set

Keep `subscribeTile` usage. On mount, the renderer subscribes to all 14,400 coords; each callback adds the coord to a `Set<CoordKey>`. An rAF loop drains the set, computes pixel rects, and `clearRect` + `fillRect` (or `drawImage` from a sprite atlas) per dirty coord.

```ts
// sketch
const dirty = new Set<CoordKey>();
const unsubs = coords.map(c => store.subscribeTile(c, () => dirty.add(serializeCoord(c))));
function frame() {
  if (dirty.size > 0) flush(dirty);
  dirty.clear();
  requestAnimationFrame(frame);
}
```

**Pros.** Smallest diff to the store (zero changes). The per-tile diff already does the dirty detection upstream; we're just bridging it to canvas. Plays nicely with future per-tile consumers (HUD overlay, debug picker).

**Cons.** Pays for 14,400 subscription objects (Set entries) for no benefit — the canvas renderer is the single subscriber that matters. The renderer carries a coord-iteration ceremony on init/teardown that buys nothing. Resubscribing on `matchId` change is a Set-of-Sets dance.

### Option B — One canvas, version-driven, reads `tileCache` directly  *(recommended starting point)*

Subscribe once to the whole store. On every notify, mark `dirty = true`. An rAF loop checks `dirty`, then walks the cache and draws.

```ts
// sketch (B1)
let dirty = true;
store.subscribe(() => { dirty = true; });
function frame() {
  if (dirty) {
    for (const coord of allCoords) {
      drawTile(ctx, coord, store.getTileData(coord), theme);
    }
    dirty = false;
  }
  requestAnimationFrame(frame);
}
```

**Pros.** Drops `subscribeTile` from the renderer entirely. One subscription, one rAF, simple lifecycle. Full-board fillRect over 14k tiles is ~3–5ms — well within budget at 1Hz, trivially correct.

**Cons.** Doesn't take advantage of the diff that `runPipeline` already computes. Fine at 14,400 tiles; less fine at 200×200 = 40k.

**A note on a shadow-cache variant.** A naive instinct is to add a per-renderer `Map<CoordKey, TileVisualKey>` shadow that diffs which tiles changed visually before drawing. *Don't.* The store's `tilesEqual` (`tile-data.ts`) already compares every field that affects rendering (owner, origin, terrain, isPlaceable, insideExclusion, exclusionSource); `runPipeline` only updates the cache when tiles actually changed. A renderer-side shadow recomputes the same diff one layer downstream. The clean asymptotic improvement is B3 below, not a parallel diff.

**Variant B3 — runPipeline emits a dirty set.** Modify `runPipeline` to expose the changed coords it already computes (`board-store.ts:113–128`). Renderer reads the dirty set off the most recent emission and draws only those tiles. Mild contract addition (the store now broadcasts a dirty list, not just "something changed"); ~10 lines of impl. Right end-state if profiling on larger boards calls for it; B1 is the right *starting* state — ship simple, measure, only specialize if measurement says so.

**Variant B0 — `putImageData` from a typed array (mentioned for completeness).** Floor case: encode the board as a `Uint8ClampedArray`-backed `ImageData` at one pixel per tile, `putImageData` once per tick, scale to display size with `image-rendering: pixelated` on the canvas (or via a CSS `transform: scale`). This is the cheapest possible redraw path. It does not work for blob-wars's actual visual (origin-seed dot, exclusion-tint composition, hover ring) without significant compositing work on top — those need post-`putImageData` overdraw, at which point you've reinvented B1 with extra steps. Useful only as a perf reference point: B1's per-tile fillRect should be within ~2× of putImageData's cost, and if it isn't, something's wrong with the draw code. Not a serious shipping option.

### Option C — Layered canvases + sprite atlas + DOM hover overlay

Stack three or four `<canvas>` elements in a positioned container:

1. **Static layer.** Grid lines and gutters; walls if (and only if) we accept "no terrain mutation mid-match" as a renderer contract. Drawn once on mount; redrawn only on match remount. **Assumption to declare**: terrain doesn't change after `applySnapshot` for the current match. Today this holds (no game mechanic mutates terrain). If destructible walls ever ship, walls move to the owner layer or a "terrain layer" that redraws on terrain dirty-flag.
2. **Owner layer.** Tile fills, origin marks, exclusion tints. This is what the per-tick redraw operates on.
3. **Effects layer.** Hover ring, plant-cursor, future animations (growth particles, capture flashes).
4. **Optional DOM overlay.** A single transparent div with `pointermove` / `pointerdown`, mapping screen coords → tile coords math, dispatching to renderer state.

Pre-bake "tile species" sprites (player1+seed, player2+spread, exclusion-p1, exclusion-both, etc.) onto an offscreen atlas at mount time. Per-tile draw is then a single `drawImage(atlas, sx, sy, …, dx, dy, …)` — measurably faster than fill + radial-gradient combos.

**Pros.** Each layer redraws on its own cadence. Effects layer can run rAF at 60Hz for animation while owner layer only redraws on tick. Sprite atlas is the standard 2D-game approach and unlocks animation cells trivially. Cleanest path to "interesting visuals".

**Cons.** More code: layer composition, sprite-atlas generation, layer-specific dirty tracking. Compositing N canvases costs something in the GPU compositor (small but non-zero). DPI scaling has to be reapplied per layer. Overkill if we never want animation.

### Option D — OffscreenCanvas + Web Worker

The renderer module runs in a worker. Main thread postMessages snapshot diffs (or whole snapshots; they're <1MB JSON for 14k tiles). Worker derives, draws to its `OffscreenCanvas`, and the main thread sees pixels appear. Main thread keeps `pointermove` / `click` and forwards as messages.

**Pros.** Removes *all* render work from the main thread. Future-proofs against animation, larger boards, and any main-thread contention. The store's purity makes this almost a pure transport problem — the worker hosts an instance of `createBoardStore` and applies snapshots locally.

**Cons.** Real complexity: two store instances (or one in the worker only), serialized message protocol, `transfer` semantics for `OffscreenCanvas`. Debugging is harder. Hot reload is harder. We don't have a measured main-thread bottleneck that justifies this. OffscreenCanvas is supported across modern browsers but adds another portability axis to remember.

Worth keeping in our back pocket; not worth doing now.

### Option E — WebGL with instanced quads

One quad mesh, instanced 14,400 times. Per-instance attributes: tile index (→ position), color, origin marker flag, exclusion flag. Single draw call per frame; updates are `bufferSubData` patches into the attribute buffer.

**Pros.** Fastest possible. Scales to 500×500+. Zoom/pan/effects/particles are native. Future-proofs everything.

**Cons.** Order-of-magnitude more complexity. Shader code, GL state machine, attribute layouts, framebuffer fiddling. Adds a tech to the stack that nobody else here is using. We don't need it. The "blobby organic" visual direction (gradient borders, soft outlines) is actually *easier* in canvas2D than in shader land, despite WebGL's raw speed.

A WebGL renderer is the right answer for game #2 if it's a real-time RTS or a 200×200+ board. For blob-wars at 1Hz, it's the wrong tool.

### Option F — Hybrid: canvas for visuals, single DOM overlay for input

This is more of a *coding pattern* than a separate option — it can be combined with A/B/C. Worth naming because the alternative (input handling on the canvas itself) is also valid and the choice has UX implications.

**F1: input on canvas.** Attach `pointermove` / `pointerdown` directly to the `<canvas>`. Coord math: `(event.offsetX / cellSizePx, event.offsetY / cellSizePx)`. Hover state is renderer-side; redraw the hover ring on rAF.

**F2: input on DOM overlay.** A single transparent `<div>` over the canvas with `pointer-events: auto`. Same coord math. Lets us put accessibility roles (`role="application"`, ARIA labels, keyboard focus) on a real DOM element without the 14k-divs cost.

I'd default to F1 for v1 (simpler) and graduate to F2 if accessibility or focus management warrants it. Either way the renderer doesn't change.

### Option G — Single `<svg>` with `<rect>` per tile (two sub-variants)

Render the board as one `<svg>` element with 14,400 `<rect>` children. Hover via CSS `:hover` on `<rect>`, transitions via CSS `transition: fill 200ms`, theme variables work, layout engine handles DPR/sizing. The interesting question is whether React owns the rect tree or not.

**G1 — React renders the rects.** `<svg>` with `{coords.map(c => <rect key={c} fill={fillFor(tile)} />)}`. Naive but obvious. **This has the same React reconciliation cost as today's DOM tree** — every render walks 14,400 children. It does buy us SVG's drawing primitives (`<filter>`, `<path>`, gradients, etc.) without giving back React's per-tile diff cost. Probably not worth shipping; mentioned only because it's the obvious port.

**G2 — Imperative SVG.** React renders an empty `<svg ref={...}>` shell. A renderer module mounts the rect children via direct DOM ops (`createElementNS` + `appendChild`); on store changes, mutates `fill` / `data-state` attributes on existing rects rather than re-rendering. CSS still drives `:hover` and `transition`. Same architectural shape as the canvas options — React out of the per-tick path, single ref-mounted shell, imperative module owns updates — but with SVG's free hover/transition/theme-var/layout-engine support.

This is the genuinely strong SVG case. The pattern parallels canvas Option B almost exactly: subscribe once, mark dirty, on rAF flush attribute updates to changed rects. With B3-style (`runPipeline` exposes dirty coords), only the changed rects get attribute updates — comparable per-tick work to canvas, with better hover/transition for free.

```ts
// sketch (G2)
const rects = new Map<CoordKey, SVGRectElement>();
function init(svg: SVGSVGElement, store: BoardStoreInstance) {
  // Mount once: 14,400 rects, positioned via CSS Grid on SVG (or x/y attrs).
  for (const coord of allCoords) {
    const r = document.createElementNS('http://www.w3.org/2000/svg', 'rect');
    r.setAttribute('x', String(coord.x));
    r.setAttribute('y', String(coord.y));
    r.setAttribute('width', '1');
    r.setAttribute('height', '1');
    setRectFromTile(r, store.getTileData(coord));
    svg.appendChild(r);
    rects.set(serializeCoord(coord), r);
  }
  store.subscribe(() => { /* mark dirty, rAF flush */ });
}
```

**Pros (G2).** Free `:hover`, free CSS transitions (single line: `rect { transition: fill 200ms; }`). Organic-looking transitions for free. Theme variables work as-is. Layout engine handles container fit and DPR. No rAF loop required if we let CSS transitions drive the animation between ticks; can opt in to rAF later.

**Cons (G2).** SVG with 14k child elements is still a real DOM tree — Elements panel performance, memory footprint, scroll perf all unknown. The "blob outlines via `<filter>`" claim deserves caveats (see below). Coord-to-pixel math on a viewBox-fit SVG is slightly different from a sized canvas (use CTM or viewBox math).

**About `<filter>` for organic blob outlines — read this carefully.**

The classic "goo filter" recipe (`feGaussianBlur` + `feColorMatrix` to alpha-threshold) was designed for **sparse**, **separated** shapes — chat bubbles merging, loading-spinner dots — where each shape's blurred halo overlaps a neighbor's halo and the threshold restores a hard edge that *bridges* them. On a 120×120 grid where most of a player's blob is *adjacent same-color rects*, the filter's main effect is to soften the blob's outer boundary, not to add organic merging — adjacent rects don't need merging because they already touch. A useful blur is ~1.5–3 viewBox units (with `1 unit = 1 tile`), retuned alpha threshold; smaller values produce no visible effect. Even with correct parameters, the effect on a dense grid is "softened blob outline," not "organic blob shape" — we should not promise more than that.

If we want true organic shapes (irregular, non-grid-aligned outlines), the right approach is a **boundary path**: marching-squares over the tile grid to generate a `<path>` per blob, smoothed with bezier curves, animated via `pathLength` / `stroke-dashoffset`. That's not free in either canvas or SVG — it's actual geometry code — but it's the same cost in both, and SVG's path animation is materially nicer than canvas's.

**The case for G2 vs canvas:** strongest if our visual ambitions are **CSS-shaped** (color transitions on tick, hover affordances, theme integration). Canvas wins if we want **frame-loop animation** (per-pixel particles, capture flashes, anything varying per-frame at 60Hz). SVG can do frame-loop animation but it's awkward; canvas can do CSS-shaped transitions but you reimplement the easing yourself.

**Recommendation:** spike G2 for ~1 hour before committing to canvas. Concrete spike scope:

1. **Decisive Elements-panel test first (5 minutes).** Render 14,400 placeholder `<rect>` children (no rendering wiring yet, no store subscription, just static rects). Open Chrome DevTools Elements panel. If the panel hangs or blocks the main thread the same way today's 14k-div tree does (likely — Chrome serializes any DOM tree on Elements expansion regardless of node type), **G2 fails the spike here**. The DX motivation collapses to "free CSS transitions," which is not enough on its own to outweigh canvas's clean win on the same metric. Stop and go canvas.
2. If step 1 passes, mount 14,400 rects via direct DOM ops (no React per child).
3. Wire `store.subscribe` → flush attribute changes on rAF.
4. Add `rect { transition: fill 200ms; }` and verify color transitions on owner change look right.
5. *Skip the goo filter* in the spike — measure baseline G2 perf first. Filter perf is a separate question; conflating the two muddies the result.

If G2 passes step 1 *and* comes back with comparable perf to canvas, free hover, and free color transitions — take it. Canvas is over-engineering for the *visual* case, and G2 is closer to the actual code we already have. If step 1 fails (likely), we know early and cheaply that canvas is the answer.

## Cross-cutting decisions (canvas-flavored)

These apply to canvas options (A/B/C/D). Most fall away under SVG (Option G) — SVG inherits the layout engine, `:hover`, transitions, theme variables, and HMR-via-React-Fast-Refresh — but the animation and hover sections still apply if we want effects beyond CSS transitions.

### Frame loop: rAF-on-dirty, not always-on

Always-running rAF burns power and shows up in CPU profiles even when nothing's happening. The right pattern is `dirty = true` on store change; `if (dirty) draw(); dirty = false;` inside an rAF that only runs when needed. Two ways to "only run when needed":

- **Continuous rAF, gated on flag.** Simplest; the rAF loop runs forever but is a no-op when clean. Costs ~0 CPU.
- **One-shot rAF on dirty.** Schedule rAF only when a change arrives; renderer is fully idle otherwise. Cleanest; slightly more code.

Once we have animations, the loop must run continuously anyway. So: start with one-shot on dirty (today's reality has no animation); switch to continuous-gated when animation lands.

### Sizing and DPR: integer device-pixel cell size

These are normally separate sections; they shouldn't be. The right approach couples them, otherwise you get seams at fractional DPR (1.5×, 2.5× — common on Windows). The naive approach — "round CSS-px cell size to an integer" — actively *creates* seams at fractional DPR because integer CSS px doesn't map to integer device px.

**The right pattern:**

```ts
function applyLayout(canvas: HTMLCanvasElement, container: HTMLElement) {
  const dpr = window.devicePixelRatio || 1;
  const containerCssPx = container.getBoundingClientRect().width;

  // 1. Cell size in DEVICE pixels, integer. Avoids seams at any DPR.
  const cellDevicePx = Math.floor((containerCssPx * dpr) / GRID_WIDTH);

  // 2. Backing-store size derived from cell size (so total = cells × integer pixels).
  const backingSize = cellDevicePx * GRID_WIDTH;

  // 3. CSS size derived from backing size, may be fractional CSS px.
  //    Container space ≥ this; the difference is unused gutter (centered or padded).
  const cssSize = backingSize / dpr;

  canvas.width = backingSize;          // attribute, not style
  canvas.height = backingSize;
  canvas.style.width = `${cssSize}px`;
  canvas.style.height = `${cssSize}px`;

  // All draw calls operate in DEVICE pixels (unscaled context). Coord math:
  // pixelX = coord.x * cellDevicePx, etc. No setTransform(dpr, ...) needed.
}
```

Triggers for re-applying layout:
- `ResizeObserver` on the container (CSS size changes).
- `window.matchMedia(`(resolution: ${dpr}dppx)`)` change event (DPR / zoom changes).

**Hit-testing math, given the above:**

```ts
function eventToCoord(canvas: HTMLCanvasElement, event: PointerEvent): Coord {
  const rect = canvas.getBoundingClientRect();
  const cssX = event.clientX - rect.left;        // CSS px relative to canvas
  const cssY = event.clientY - rect.top;
  const cellCssPx = rect.width / GRID_WIDTH;     // fractional, that's fine
  return { x: Math.floor(cssX / cellCssPx), y: Math.floor(cssY / cellCssPx) };
}
```

Note we pull `rect.width` rather than caching `cellCssPx` — keeps the math accurate across any later layout shift.

**The current container.** `.board` uses `aspect-ratio: 1; flex: 1 1 0` inside a flex column. Canvas preserves the same outer container; the canvas element is the inner element. The CSS size from `applyLayout` will be ≤ container size; remaining space becomes gutter (center the canvas in the container, optionally with a thin debug border during dev).

### Tile draw operations

A flat `fillRect` per tile undersells the existing visual. The current `board.css` does more than colors — let's enumerate what the renderer actually needs to reproduce:

| Visual | Today (CSS) | In renderer |
|---|---|---|
| Empty tile | `background: rgba(71,75,81,0.59)` | `fillRect` with that color |
| Wall | `background: #1f2937; border-color: #111827` | `fillRect` + 1px stroke or inset rect |
| Player1 / Player2 owner | `background: var(--tile-color)` | `fillRect` with seat color |
| Origin = spread | flat owner color | `fillRect`, same as above |
| Origin = seed | radial-gradient (white dot center 30%) + linear-gradient sheen + owner color | `fillRect` (owner) + `arc` filled white at center + optional linear-gradient sheen |
| Exclusion zone (player1/player2/both) | linear-gradient overlay tint over base | `fillRect` (base) + `fillRect` with tint color and alpha |
| Hover (placeable) | seat-tinted border + inset glow | `strokeRect` border + alpha-blended inset fill |
| Tile-disabled | `cursor: default`, no hover | branch in hover handler / cursor CSS on canvas element |
| Inter-tile gap | `gap: 1px` in CSS grid | leave 1 device-px gutter when computing draw rects |

Two tiles need real composition: **origin=seed** (3-layer stack) and **exclusion zone** (alpha overlay). The renderer scope is "fillRect + fillRect overlays + one `arc` for seed center + one `strokeRect` for hover" — small but not trivially "fillRect of a constant." Worth implementing as `drawTile(ctx, coord, tile, theme)` that handles all combinations, and verifying by side-by-side screenshots.

**Pre-baking via sprite atlas (optional, B-only optimization).** Generate one `OffscreenCanvas` at init holding every "tile species" (player1+seed, player1+spread, exclusion-p1+player1, exclusion-both, exclusion-p1+empty, …). Per-tile draw becomes one `drawImage(atlas, sx, sy, sw, sh, dx, dy, dw, dh)`. ~10× faster than fill+gradient combos per tile. Defer to v2 unless full-redraw measures slow.

### Hover, exclusion preview, and the click-affordance

DOM gives us `:hover` for free. Canvas requires us to track hover state and redraw it. Two upgrades fall out of doing it ourselves:

- **Exclusion-radius preview.** When the local player hovers a placeable tile during placing phase, light up the 7-radius circular halo where future seeds *would* be excluded. Trivial in canvas; awkward in CSS. Concrete win for the swap.
- **Disabled-cell affordance.** The current CSS turns off hover for disabled tiles; canvas does the same with one branch in the hover renderer.

I'd treat exclusion preview as a v1 nice-to-have (cost: ~1 evening) since it directly motivates the swap.

### Animation strategy (when we want it)

There are two qualitatively different kinds of animation, and they have different upstream requirements. "Particle bursts on capture" and "color tween on owner change" sound similar but aren't equivalent.

**Snapshot-driven tweens (free).** Smooth the path between two store states. Per-tile transition state held in the renderer: when the renderer detects a tile changed `owner` from `null → player1`, it kicks off a 200ms tween over `from-color → to-color`. Each rAF, lerp and draw. When all tweens finish, idle. This is purely a renderer concern — store is snapshot-truth, renderer interpolates between snapshots. Late snapshots cancel in-flight tweens cleanly (`from = currentInterpolated, to = newTarget`). Examples: color fades on owner change, hover ring fade-in, exclusion-radius halo fade-in.

**Event-driven effects (NOT free).** Visual flourishes that depend on knowing the *kind* of change, not just the before/after. A "capture flash" should fire on `seed→capture` but not on `empty→seed`. A particle burst on capture differs from a fade-in on growth. The current snapshot diff doesn't carry this — `runPipeline` knows a tile changed but not why. Two paths:

- **Renderer-side change classification.** The renderer compares its own previous shadow of `TileData` against the new one and infers the change kind (`empty → owned`, `owned[A] → owned[B]`, etc.). Works for transitions deducible from before/after; fails for things like "this growth came from blob #7" without that info in `TileData`.
- **Store/protocol surfaces change events.** `runPipeline` exposes a per-tile change kind alongside the dirty set; or the protocol grows a `tileEvents: TileEvent[]` channel on each `MatchSnapshot`. Real upstream cost — touches store internals or wire format.

Event-driven effects also need a different *lifecycle* model than tweens: an effect queue with TTL, max-concurrent caps, and explicit "let it finish" / "cancel and re-fire" rules. The "cleanly cancel late snapshots" pattern doesn't apply (a particle burst that's mid-flight when an opponent recaptures shouldn't smoothly redirect; it should either finish on its own timeline or drop). This is a separate v1+1 design than the tween model.

For v1+1 (when animation lands): start with renderer-side classification + a simple "effect queue" with TTL. It's enough for color tweens, fade-ins, and most of the visual repertoire. Promote to store/protocol-level events only when we hit a flourish that demands it.

**Critical properties** (still hold for the snapshot-driven path):

- Animations don't touch the protocol.
- Animations don't add per-tick store work.
- Late-arriving snapshots cleanly cancel in-flight tweens.

**Open question for v1+1.** When a tile flips owner twice within one tween window (empty → p1 → p2 within 200ms), does the lerp animate the path (briefly tinting toward purple before redirecting toward green) or snap to the latest target? Both are defensible; pick a default before shipping animation.

Animation isn't v1. But the architecture leaves the door open by making the renderer module the obvious place to add it later.

### HMR for the renderer module

Vite has React Fast Refresh, which works well for components but does not preserve module-level state (rAF id, hover state, mounted DOM children for G2). The renderer module has both. Worth being clear about what `session-instance.ts` actually does as a reference point: it uses `import.meta.hot.dispose` to *tear down and let HMR cascade into a full module reload*, not `hot.accept` to swap-and-keep-state. There's no in-tree precedent for live module swap of long-running state.

Two coherent patterns for the renderer:

- **React-effect-driven init/teardown.** `BoardCanvas.tsx`'s `useEffect` runs `init(canvas, store)` on mount and `destroy()` on unmount. Renderer state lives inside that effect's closure (or a single ref). Edits to `draw.ts` / `theme.ts` / etc. trigger Fast Refresh; the BoardCanvas component is re-mounted by Fast Refresh's component-swap behavior; the effect tears down and re-runs. Loses any in-flight tween mid-edit, but we don't have tweens yet, and the price of "tweak a color, lose 200ms of animation" is acceptable.
- **Module-level singleton with `import.meta.hot.dispose` + `accept`.** Renderer module owns its rAF/state at module scope. On HMR, `dispose` tears it down; `accept` re-imports and re-initializes against the existing canvas + store. Preserves DX feature parity with `session-instance.ts` (which uses dispose-and-cascade-reload but in dev that's fine because `session-instance.ts` is rarely edited). Adds non-trivial complexity: passing the canvas ref through a module-level singleton, handling the case where the module re-evaluates before the ref resolves, etc.

Pick the first pattern. The DX benefit of in-place state preservation across HMR is small (no tweens yet to preserve), and the React-effect path requires no `import.meta.hot` plumbing in the renderer module — Fast Refresh handles the lifecycle for free. Document in the renderer module's top-of-file contract comment that "HMR is owned by the React component shell; the renderer module is a pure init/destroy pair."

If we land animation later and HMR-eats-tweens becomes annoying, the `dispose+accept` upgrade is a localized refactor.

### Color & theme

CSS custom properties (`--accent-bg`, `--text-soft`, etc.) drive the rest of the app. The canvas renderer needs equivalent values in JS. Two flavors:

- **Theme-level (seat-independent).** `--accent-bg`, `--text-soft`, `--border`. Read once at init via `getComputedStyle(document.documentElement).getPropertyValue('--accent-bg')`. Stays linked to theme; supports a future light/dark toggle.
- **Seat-scoped (runtime-known).** Today's `.board-seat-player1` / `.board-seat-player2` define `--tile-hover-color` and `--tile-hover-glow` for the local player's hover ring (`board.css:12–20`). The local seat is unknown until the first snapshot arrives, so this can't be read at init. Two options:
  - **Look up by seat key.** Hardcode a `seatColors: Record<PlayerSeat, { hover: string; glow: string }>` table in `theme.ts`, populated from the same color values currently in CSS. Renderer keys into it once `currentUser.seat` resolves. Loses the "single source of truth" benefit for these specific values; gains stable runtime access.
  - **Read from a hidden DOM probe.** Render an off-screen `<div class="board-seat-player1">` and `<div class="board-seat-player2">`, read their computed `--tile-hover-color` once, store the lookup. Preserves CSS-as-source-of-truth at the cost of a probe element.

Pick the lookup table for v1 — simpler, ~6 lines, theme drift on these two specific values is unlikely (they're seat colors, conceptually owned by gameplay rather than theme). Revisit if we add light/dark.

### Dimension changes mid-session

Today the grid size is fixed for a session (architectural assumption documented in `board-store.ts`). The renderer can rely on this — capture `width`/`height` once on init, never recheck. If we ever support resize-mid-session, it'd be a remount via React `key`, same as `matchId`.

## What we lose

Honest list:

- **Free `:hover`.** Reimplemented in code; not free, but bounded.
- **React DevTools Profiler view of the board.** Already noisy, already deleted by the perf-debug gate. Not a real loss.
- **Per-tile RTL tests.** The renderer module is harder to test than the DOM tile. Unit testing stays at the store / pure-function layer (where the actual game logic lives); the renderer becomes a "verify by playing" thing or a pixel-diff thing if we ever want it.
- **Implicit DOM accessibility.** A 14k-button DOM grid wasn't actually accessible — no screen-reader user is navigating that. Canvas with explicit keyboard support (arrow-keys to move a focus cursor, Enter to plant, ARIA live region for state changes) is *better* a11y than the current implementation, but it's now explicit work we own.
- **The "open Elements panel and inspect a tile" debugging move.** Replaced by `window.__board.dumpTile(x, y)` style debug hooks if we want them. Probably worth wiring during the swap.

## What we gain

- **Visual headroom.** Animation, gradients, organic outlines, hover effects, particle bursts — each is local to the renderer module instead of compounding the DOM cost. (SVG-with-filters covers the static-but-organic subset more cheaply; canvas is needed for frame-loop animation.)
- **DevTools Elements panel works again** in canvas mode. SVG depends on browser collapsing behavior — measure during the spike.
- **Larger boards become tractable.** If we ever want 200×200, we don't have to redo the renderer.
- **Bandwidth for "second opinions"** — exclusion-radius preview, blob-power visualization, growth direction arrows. Things we can't add to DOM without paying again.

Not on this list: placing-phase fan-out elimination. That's a cheap DOM-side fix (see §"Cheap DOM fixes") and shouldn't be credited to the canvas swap.

## Recommendation

Start with **Option B1** (one canvas, full-board redraw on dirty rAF, reads `tileCache` directly), pattern **F1** (input on canvas), with cross-cutting decisions as listed. Concretely:

```
ui/src/board-canvas/
  board-canvas.tsx      thin React wrapper: <canvas ref=…> + useEffect that wires the renderer
  renderer.ts           imperative module: init(canvas, store), destroy(), the rAF loop
  draw.ts               pure draw functions: drawTile(ctx, coord, tile, theme), drawHover(...)
  theme.ts              read CSS vars at init, expose typed colors
  layout.ts             cellSizePx, screen→coord math, DPR setup
  hit-test.ts           pointer event → coord
```

`MatchContent` becomes the renderer-flag site, branching on `?renderer=canvas` (or a const) between the DOM and canvas branches; both keyed by `matchId`. `Board.tsx`, `Tile`, `board.css`, `use-tile-data.ts`, `BoardProfiler` get deleted only after the canvas instrumentation lands.

**Initialization ordering (important).** A canvas renderer that subscribes lazily can paint a blank first frame on match remount. The order in `init(canvas, store)` must be:

1. Apply layout (`applyLayout` above).
2. Paint the current cache state synchronously (`drawAll(ctx, store)` reading `store.getTileData` for every coord).
3. Subscribe to the store; on every notify, set `dirty = true` and schedule rAF (one-shot pattern; switch to continuous-rAF when animation lands).
4. Attach pointer handlers.

If we subscribe before step 2 we race the store; if we paint after subscribing we're fine but waste a frame. Step 2 is the synchronous "first paint" guarantee; without it, on `key={matchId}` remount, `applySnapshot` has already run before `useEffect` fires, and the canvas mounts blank until the next tick (~1s later in simulating).

**Load-bearing assumption.** Step 2 reads from `store.getTileData` and assumes the cache is already populated. The mechanism: the WebSocket message handler runs `applySnapshot` synchronously, which runs `runPipeline` and populates `tileCache`, then notifies subscribers. `useMatchId` (read by `MatchContent`) is one of those subscribers; it re-renders the chain and mounts a fresh `<BoardCanvas key={matchId}>`. By the time the new component's `useEffect` fires (post-commit, post-paint), the cache reflects the new match. So step 2's synchronous read works.

The renderer relies on `applySnapshot` being **synchronous w.r.t. the React render that mounted us**. If `applySnapshot` ever becomes asynchronous (microtask, batched commit, transferred to a worker, etc.), the synchronous-paint guarantee breaks; the renderer would need to handle "cache is empty / stale on first read" and re-paint on the first store notify. Worth a note in the renderer module's contract comment so a future refactor doesn't silently invalidate this.

Migration plan:

1. **Land the cheap DOM fixes first (un-prop `canPlant`).** Independent of the canvas swap. Lands on its own merit and removes the canPlant fan-out as a confound from any later perf comparison. Without this landing first, canvas will look artificially good against DOM in the placing phase, and the team will draw the wrong conclusion. ~1 hour.
2. **Add the `canvasDraw` perf event and refactor `paint` sourcing.** Two paired changes:
   - Add `perfLog.timed('canvasDraw', tick, () => drawAll(...))` to the renderer module. Update `formatTickLine` and the `TickRecord` type in `perf-log.ts` to surface it.
   - Move `paint` event emission off the store and onto the renderer. Today `paint` is emitted from `perfLog.rAFEvent('paint', tick)` inside `board-store.ts:143` (the `onChange` callback), which is rAF-deferred relative to React's commit — fine for DOM mode, **wrong for canvas** because it fires before the rAF-on-dirty loop redraws. If we leave it as-is during dual-renderer mode, canvas's `wsRecv→paint` *under-counts* draw latency (the recorder closes on the first `paint` event per tick — `perf-recorder.ts:37–40`).
   
   Two clean options for the paint-source refactor: (a) make `paint` a renderer responsibility unconditionally — DOM emits from `useLayoutEffect` post-commit, canvas emits at end of `drawAll`; (b) gate the store-side `rAFEvent` on "no canvas renderer mounted" via a module flag. Prefer (a) — fewer cross-module flags.
3. **Land B1 alongside the DOM board behind a renderer flag.** Branch in `match-content.tsx`:
   ```tsx
   {useRenderer() === 'canvas'
     ? <BoardCanvas key={matchId} session={session} width={GRID_WIDTH} height={GRID_HEIGHT} />
     : <BoardProfiler><Board key={matchId} session={session} width={GRID_WIDTH} height={GRID_HEIGHT} /></BoardProfiler>}
   ```
   Both renderers run against the same store. `useRenderer` reads the flag (URL param or `localStorage` for sticky dev preference). This is the contract-proves-out step.
4. **Verify perf parity or better — apples-to-apples.** With step 1 in place (DOM canPlant fixed) and step 2 in place (paint sourced from the active renderer), compare:
   - `wsRecv → drawAll start` (canvas) vs. `wsRecv → reactCommit start` (DOM) — both measure scheduling latency.
   - `drawAll duration` (canvas) vs. `commitMs` (DOM) — both measure the per-tick render work.
   - `wsRecv → paint` — sourced from the renderer in both modes, so apples-to-apples.

   Sanity check: a manual board redraw from the cache should land in <5ms. Placing-phase outliers should be similar across renderers; the big deltas should be DevTools-open behavior and the absence of a 14k-node DOM.

   Worth naming explicitly: post-swap, the only React commit on the per-tick critical path is `<MatchDetails>`, which subscribes to store version and re-renders every tick. It's small but real, and it's the floor on `wsRecv→paint` regardless of renderer. If we ever need to drive that to zero, `MatchDetails` would need to swap to imperative DOM updates too.
5. **Add exclusion-radius hover preview.** This is the visible-win moment for users; it's also a real exercise of the renderer's hover-state path. Land canvas-only (the DOM version isn't getting this).
6. **Delete the DOM renderer.** Drop `Board.tsx`, `Tile`, `board.css`, `use-tile-data.ts`, `use-can-plant.ts` (no remaining consumer once canvas reads `session.canPlant()` directly), `BoardProfiler` (and its `reactCommit` event from `perf-log.ts`). Resolve the `subscribeTile` decision (gate the `runPipeline` dispatch on `tileSubs.size > 0`, per the §"What changes" discussion).
7. **(Later, optional)** Graduate B1 → B3 (`runPipeline` exposes a dirty-coord set) if profiling on larger boards calls for it. Graduate to layered canvases (Option C) when animation lands. Consider OffscreenCanvas (Option D) only if main-thread contention shows up.

Why B1 over B3 as the *starting* point: B3 leaks store internals (a "dirty coords since last drain" channel) before we know we need them. B1 stays inside the existing contract and leaves room for B3 as a clean extension. Forward door open, no premature commitments.

Why not C from the start: layered canvases shine *with* animation. Without animation (where we are today), the second canvas is overhead with no payoff. Right tool, wrong day.

Why not D or E ever, until proven: D adds a worker boundary we don't need; E adds a rendering tech the rest of the project isn't paying for. Both are reachable from B1.

Why this might be the *wrong* answer entirely: if the visual direction is "organic blobs + soft hover affordances + color transitions on tick" — and not "frame-loop animation" — then **Option G2 (imperative SVG)** is probably the right choice and canvas is over-engineering. The ~1 hour G2 spike (see Open question 1 below) should run before any canvas code lands. G2 buys free hover and free CSS transitions while keeping React out of the per-tick path; canvas wins decisively only if frame-loop animation is a stated goal.

## Open questions / things to validate before committing

1. **What's the actual visual direction?** This is the question that gates everything else. Three candidate answers: (a) "v1 looks like today, just on canvas" — go with Option B1; (b) "organic blob outlines, soft borders, hover halos, color transitions on tick" — strongly favors Option G2 (imperative SVG); (c) "frame-loop animation, particle bursts on capture, growth direction arrows" — favors Option B/C with a real animation pipeline. Worth a brief design conversation, then a ~1 hour G2 spike, before coding anything else.
2. **Accessibility floor.** What's the minimum we want? Best guess: "renders a label/state announcement for the local player's turn", nothing fancier. Worth confirming we're not signing up for full keyboard navigation as a v1 deliverable.
3. **DevTools Elements panel cost in SVG mode.** If we go SVG, the 14k-rect tree is a real DOM tree. The panel might or might not collapse it acceptably — depends on browser version. Spike answer is part of #1.
