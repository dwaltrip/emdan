# UI perf refactor: session layer + thin React hooks

Design note for decoupling game/network logic from React so per-tick renders scale with *changed tiles*, not with *grid size*. Not implemented yet.

## Motivation

Per-tick perf logging (committed in `5946a39`, see `perfLog` in `ui/src/lib/perf-log.ts`) showed a clear bottleneck on a 50×40 = 2000-tile grid during simulating phase:

| Zone | Cost |
|---|---|
| `derive` | ~0.0–0.5 ms |
| `diff` (per-tile pipeline) | ~0.5–1.0 ms |
| `pipeEnd → paint` | ~5–10 ms |
| **`wsRecv → pipeStart`** | **~100–125 ms** |
| `tileRenders` | ~4000 (2× the grid, every tick) |
| `changed` | 6–170 per tick |

At 14,400 tiles the `wsRecv→pipeStart` gap was ~600 ms. The store/diff work and browser paint are fine. **The bottleneck is React re-rendering every `<Tile>` on every WS message.**

## Root cause

`latestSnapshot` lives in `useState` inside `useGameSocket`. Every WS message calls `setLatestSnapshot`, which re-renders `App`. App re-renders `<Board>`, which builds a 2000-entry `cells` array and creates a `<Tile>` element for each. Tile's function body then runs because (a) no `React.memo` wrapper is applied and (b) even if the Compiler attempts auto-memoization, `onPlant` is a fresh closure per parent render and defeats it. `useSyncExternalStore` prevents DOM changes when tile data is unchanged, but not the React element walk. The ~4000 count is 2× 2000 due to StrictMode dev double-invoke.

**Primary fix**: take the snapshot out of App's React state. Route it directly into a pure "session" layer that owns the store. With the snapshot no longer triggering App re-renders, `<Board>` stops re-rendering per tick and the fan-out disappears. `React.memo(<Tile>)` is added as defense-in-depth, but it is not the primary lever.

## Target state (who re-renders when)

| Component | Re-renders when |
|---|---|
| `App` | `connectionStatus` or `currentUser.seat` changes (rare) |
| `<Board>` | `canPlant` flips (turn transitions) or `matchId` (remount on new match) |
| `<Tile>` | its own per-tile cache entry changes |
| `<MatchDetails>` | per tick (small component; accept it) |
| `<StatusBar>` | `connectionStatus` changes |

`tileRenders` scales with `changed` tiles, not with grid size.

- **Simulating, steady-state**: `tileRenders ≤ 2 × changed` (StrictMode in dev; ~1× in prod).
- **Placing, turn transition**: all tiles re-render once (`canPlant` flipped) — one-shot, not per-tick.
- **Idle (no messages)**: zero tile renders.

## Protocol changes (shared + server)

Atomic refactor — the app is a prototype, no back-compat plumbing is needed.

- `MatchSnapshot` gains required field `currentUser: { seat: PlayerSeat | null }`. Null reserved for future spectators.
- `WelcomeMessage.seat` kept as-is in the protocol; client stops acting on it. Authoritative seat comes from snapshots only.
- `isMatchSnapshot` validator in `shared/protocol.ts` updated to require `currentUser`.
- Server's `getSnapshot()` becomes per-recipient (can no longer share a single serialized payload between the two players). Grep for `getSnapshot` in `server/src/game/`.

## Client state shape

`BlobWarsSourceState`, `UIState`, and `applySnapshot` are existing names from `board-store/`; `connectionStatus` + its default are new additions.

```ts
interface BlobWarsInputState {
  game: BlobWarsSourceState;       // mirrors MatchSnapshot verbatim, incl. currentUser
  ui: UIState;                     // transient display state (unchanged)
  connectionStatus: ConnectionStatus;  // 'connecting' | 'connected' | 'disconnected'
}
```

Default `game.currentUser = { seat: null }` so the field is always present; `seat` becomes non-null when the first snapshot arrives. Renamed top-level field from `status` to `connectionStatus` to avoid collision with `players[seat].connected` inside `game`. No special seat slot. No `setSeat` action. `applySnapshot` handles everything; `setConnectionStatus` is a trivial separate action called from socket lifecycle callbacks.

## Session layer (pure, no React imports)

Two interfaces the whole section refers to:

```ts
interface BlobWarsSession {
  store: BlobWarsBoardStoreInstance;
  handleMessage(msg: ServerMessage): void;
  handleOpen(): void;
  handleClose(): void;
  plant(x: number, y: number): void;
  canPlant(): boolean;
}

interface SessionHandle {
  session: BlobWarsSession;
  end(): void;
}
```

### `ui/src/blob-wars/session.ts`

Factory `createBlobWarsSession({ store, send }): BlobWarsSession` — orchestration on top of the existing `board-store`:

- Owns the store instance (created upstream via `createBlobWarsBoardStore` and injected).
- `handleMessage(msg: ServerMessage)` — single entry point. Dispatches welcome (client-side no-op) / lobbyUpdate / matchStarted / stateUpdate / matchEnded / error.
- On `matchId` change (detected inside an incoming snapshot): `store.reset(width, height)`, then `applySnapshot`.
- `handleOpen()` / `handleClose()` — set `connectionStatus` in the store.
- `plant(x, y)` — reads `store.state` at call time for rules check, calls injected `send`. Stable identity for the life of the session.
- `canPlant()` — pure selector over `store.state` (reads phase, currentTurn, currentUser.seat, seedsRemaining).
- Exposes `session.store` for subscriptions.

### `ui/src/blob-wars/game-socket.ts`

Pure WebSocket wrapper:

- `createGameSocket(url, { onOpen, onClose, onMessage })` → `{ send, close }`.
- No React imports; testable with a fake WebSocket constructor.
- **Fires the `wsRecv` perfLog mark** when a message is received (currently in `use-game-socket.ts:90`; must move here, otherwise the perf log breaks silently).

### `ui/src/blob-wars/session-lifecycle.ts`

Composition root — resolves the circular dependency between session (needs `send`) and socket (needs `session.handleMessage`) via a forward-referenced `getSession()`:

```ts
export function initSession(url: string): SessionHandle {
  const store = createBlobWarsBoardStore(/* default dims */);

  // Forward-referenced so socket callbacks can dispatch to the session.
  // Safe because WS callbacks only fire after `session` is assigned below.
  let session: BlobWarsSession | null = null;
  const getSession = (): BlobWarsSession => {
    if (!session) throw new Error('[session-lifecycle] session used before init');
    return session;
  };

  const socket = createGameSocket(url, {
    onOpen:    () => getSession().handleOpen(),
    onClose:   () => getSession().handleClose(),
    onMessage: (msg) => getSession().handleMessage(msg),
  });

  session = createBlobWarsSession({ store, send: socket.send });

  return { session, end: () => socket.close() };
}
```

### `ui/src/blob-wars/session-instance.ts`

Module-level singleton. Session is created once at module load, torn down on Vite HMR dispose. React never owns its lifecycle:

```ts
import { initSession } from './session-lifecycle';

const handle = initSession(WS_URL);

if (import.meta.hot) {
  import.meta.hot.dispose(() => handle.end());
}

export const session = handle.session;
```

**HMR dispose is required**, not optional — without it, every hot reload opens a fresh socket while the old one stays open. The server will see phantom connections accumulate in dev.

## React layer (thin hooks)

`ui/src/hooks/` — each ~5–10 lines, all `useSyncExternalStore` selectors:

- `use-status.ts` → `ConnectionStatus`.
- `use-current-user.ts` → `{ seat: PlayerSeat | null }` (never outer-null; `seat` is null until the first snapshot arrives).
- `use-match-id.ts` → `string | null`. Used as React `key` on `<Board>`.
- `use-phase.ts` → `MatchPhase`.
- `use-can-plant.ts` → `boolean`.
- `use-tile-data.ts` → existing, moved here from `board-store/hooks.ts`.

Selectors return primitives (or stable objects); React bails via `Object.is` on unchanged ticks. `<MatchDetails>` is the one exception — its selector returns an object and re-renders per tick. That's fine: the component is small and its content actually changes each tick.

**No `useBlobWarsSession` hook.** Consumers import `session` directly from `session-instance.ts`; selector hooks take `session` as a parameter.

**No React Context.** Props are stable enough for prop-drilling:

- `session` — stable for the module's lifetime.
- `plant` — stable reference (closes over nothing reactive).
- `canPlant` — flips only at turn transitions.

## Components

- **`App`**: imports `session` directly. Reads `connectionStatus` and `currentUser` via hooks, branches on them. No snapshot awareness, no lifecycle logic.
- **`<Board>`**: props `{ session, width, height }` plus `key={matchId}` (see "Match resets" below). Calls `useCanPlant(session)` once. Passes `{ store: session.store, x, y, canPlant, plant: session.plant }` to each `<Tile>`. Wrapped in `React.memo`.
- **`<Tile>`**: props `{ store, x, y, canPlant, plant }`. Reads tile data via `useTileData`. All props stable except `canPlant` (flips at turn transitions). Wrapped in `React.memo`.
- **`<MatchDetails>`**: subscribes via `useSyncExternalStore` selector that returns an object (tick, phase, scores, turn). Re-renders per tick; acceptable.
- **`<StatusBar>`**: reads `useStatus(session)`.
- **`<DebugPanel>` is deleted** (component + CSS). The current `latestMessage` / `logs` state in `useGameSocket` has no home in the new architecture and the panel was a dev aid we no longer need.

## Match resets (new contract)

`store.reset()` clears both `tileCache` and `tileSubs`. Mounted Tile components hold references into `tileSubs` via their `useSyncExternalStore` subscriptions — after reset, those references point to orphan Sets, and `runPipeline` notifications don't reach them. Tiles stay frozen until unmount.

Today this is latent (reset happens on remount via React `key` — which is, in fact, the mechanism we're formalizing here). In the new world `session.handleMessage` calls `store.reset` mid-mount, so we need an explicit remount signal.

**Required**: key `<Board>` by matchId.

```tsx
import { GRID_WIDTH, GRID_HEIGHT } from '@shared/protocol';
// ...
const matchId = useMatchId(session);
<Board key={matchId} session={session} width={GRID_WIDTH} height={GRID_HEIGHT} />
```

Flow on matchId change (all within a single React commit):
1. `session.handleMessage(matchStarted)` — synchronous: `store.reset(width, height)` → `applySnapshot` → `tileCache` re-populated.
2. React sees new `matchId`, unmounts old Board + all Tiles, mounts fresh instances.
3. New Tiles subscribe into fresh `tileSubs` and read from the populated `tileCache`. No flash (all within one commit).

Update the comment at `board-store/board-store.ts:171-172` to describe the new contract: *reset is safe only when Board is keyed by matchId; do not call reset on a live, un-keyed board.*

## Implementation order

Atomic refactor — the client can't boot in a half-migrated state. Land together.

1. **Protocol + server**: add `currentUser` to `MatchSnapshot`, update `isMatchSnapshot`, update server `getSnapshot` to be per-recipient.
2. **State shape**: add `connectionStatus` to top-level store state and `currentUser` to `BlobWarsSourceState`; update types; update `applySnapshot` to read `currentUser` from the snapshot. (Must land before step 3 because `session.ts` reads these fields.)
3. **Session layer** (pure): write `session.ts`, `game-socket.ts`, `session-lifecycle.ts`, `session-instance.ts`. Move the `perfLog` `wsRecv` mark from `use-game-socket.ts` to `game-socket.ts`. Testable headlessly — no React involved yet.
4. **React hooks**: write selector hooks.
5. **Components**: refactor `App`, `<Board>`, `<Tile>`, `<MatchDetails>`; add `React.memo` to Board and Tile; add `key={matchId}` to Board.
6. **Delete**: `use-game-socket.ts`, `board-store/hooks.ts`, `debug-panel.*`.
7. **Verify**: run a match, read `perfLog` output.

Steps 1–3 are self-contained (nothing consumes them yet); 4–6 must go together.

## Verification criteria

Existing `perfLog` instrumentation is the scoreboard. On the 50×40 grid after refactor:

- **Simulating tick (steady-state)**: `tileRenders ≤ 2 × changed` — ~100–340 in dev, ~50–170 in prod. Was ~4000 before.
- **Placing turn transition** (one-shot, not per-tick): ≤ 2 × grid (~4000). Expected, not a regression.
- **Idle (no messages)**: zero tile renders.
- **`wsRecv → pipeStart` gap**: dominated by store work (~1 ms), not React commit. Was ~100–600 ms before.

## Non-goals

- Canvas rendering. Kept as a known escape hatch for future mechanics that flip large fractions of the grid per tick, but out of scope here. The store/session split is designed to survive that swap (a canvas render loop would subscribe to the same store).
- Performance micro-tuning of `computeTileData`, `deriveBlobWarsState`, or the per-tile diff loop. They're already <2 ms combined; not the bottleneck.
- Removing `perfLog` instrumentation. It stays; we'll want it again next time.
