# UI perf refactor: session layer + thin React hooks

Design note for decoupling game/network logic from React so per-tick renders scale with *changed tiles*, not with *grid size*. Not implemented yet.

## Motivation

Per-tick perf logging (committed in `5946a39`, see `perfLog` in `ui/src/lib/perf-log.ts`) showed a clear bottleneck on a 50×40 = 2000-tile grid during simulating phase:

| Zone | Cost |
|---|---|
| `derive` | ~0.0–0.5 ms |
| `diff` (per-tile pipeline) | ~0.5–1.0 ms |
| `pipeEnd → paint` | ~5–10 ms |
| **`wsRecv → pipeStart`** | **~100–125 ms** |
| `tileRenders` | ~4000 (2× the grid, every tick) |
| `changed` | 6–170 per tick |

At 14,400 tiles the `wsRecv→pipeStart` gap was ~600 ms. The store/diff work and browser paint are fine. The bottleneck is React re-rendering every `<Tile>` on every WS message, regardless of whether its data changed.

Mechanically: `latestSnapshot` lives in `useState` inside `useGameSocket`, so every WS message re-renders `App`. `App` re-renders `<Board>`, which creates 2000 `<Tile>` elements. `onPlant` is a fresh closure each render (reads `latestSnapshot`), so even with React Compiler, Tile can't bail. Every Tile's function body runs; `useSyncExternalStore` prevents DOM changes but not the component walk. The ~4000 count is 2× 2000 due to StrictMode dev double-invoke.

## Target state (who re-renders when)

| Component | Re-renders when |
|---|---|
| `App` | `status` or `currentUser.seat` changes (rare) |
| `<Board>` | `canPlant` flips (turn transitions) |
| `<Tile>` | its own per-tile cache entry changes |
| `<MatchDetails>` | tick/phase/scores change |
| `<StatusBar>` | `status` changes |

`tileRenders` should scale with `changed` tiles, not with grid size. Between ticks: zero. During a simulating tick: ~2× changed (StrictMode in dev), ~1× changed in prod.

## Protocol changes (shared + server)

- `MatchSnapshot` gains `currentUser: { seat: PlayerSeat | null }`. Null reserved for future spectators.
- `WelcomeMessage.seat` kept as-is; client stops acting on it. Authoritative seat comes from snapshots only.
- Server updates `getSnapshot()` to include `currentUser` per recipient. `parseServerMessage` validator updated.

## Client state shape

```ts
interface BlobWarsInputState {
  game: BlobWarsSourceState;   // mirrors MatchSnapshot verbatim, incl. currentUser
  ui: UIState;                 // transient display state (unchanged)
}
```

No special seat slot. No `setSeat` action. `applySnapshot` handles everything. `status` (connection state) also lives in the store, mutated by an action the session calls from socket lifecycle callbacks.

## Session layer (pure, no React imports)

**`ui/src/blob-wars/session.ts`** — orchestration on top of the existing `board-store`:

- Owns the store instance (created via `createBlobWarsBoardStore`).
- `handleMessage(msg: ServerMessage)` — single entry point. Dispatches welcome / lobbyUpdate / matchStarted / stateUpdate / matchEnded / error.
- On `matchId` change (inside snapshot): `store.reset(width, height)`, then `applySnapshot`.
- `plant(x, y)` — reads `store.state` at call time for rules check, calls injected `send`. Stable identity for the life of the session.
- `canPlant()` — pure selector over `store.state` (reads phase, currentTurn, currentUser.seat, seedsRemaining).
- Exposes `session.store` for subscriptions.

**`ui/src/blob-wars/game-socket.ts`** — pure WS wrapper:

- `createGameSocket(url, { onOpen, onClose, onMessage })` → `{ send, close }`.
- No React imports; testable with a fake WebSocket constructor.

**`ui/src/blob-wars/session-lifecycle.ts`** — composition root:

- `initSession(url)` → `{ session, end }`. Wires store + socket + session callbacks together.
- `session.handleMessage` is passed as the socket's `onMessage`.
- Socket's `onOpen`/`onClose` call actions that mutate `store.state.status`.

## React layer (thin hooks)

`ui/src/hooks/`:

- `use-blob-wars-session.ts` — ~10-line mount/unmount bridge. `useRef` lazy-inits `initSession`; `useEffect` cleanup calls `end()`. Returns the session synchronously.
- `use-status.ts`, `use-current-user.ts`, `use-phase.ts`, `use-can-plant.ts` — all `useSyncExternalStore` selector hooks returning primitives. React bails via `Object.is`.
- `use-tile-data.ts` — existing, moved here from `board-store/hooks.ts`.

No React Context. Prop drilling is fine because the props are stable:

- `session` — stable for the match's lifetime.
- `plant` — stable reference (closes over nothing reactive).
- `canPlant` — flips only at turn transitions.

## Components

- **`App`**: `const session = useBlobWarsSession(url); const status = useStatus(session); const currentUser = useCurrentUser(session);` — branches on those. No snapshot awareness.
- **`<Board>`**: props `{ session, width, height }`. Calls `useCanPlant(session)` once. Passes `{ store: session.store, x, y, canPlant, plant: session.plant }` to each `<Tile>`. Wrapped in `React.memo`.
- **`<Tile>`**: props `{ store, x, y, canPlant, plant }`. Reads tile data via `useTileData`. All props stable except `canPlant` (flips at turn transitions). Wrapped in `React.memo`.
- **`<MatchDetails>`**: subscribes to its own slice (tick, phase, scores) via dedicated hook.

## Suggested implementation order

1. **Protocol**: add `currentUser` to `MatchSnapshot`, update server `getSnapshot`, update `parseServerMessage` validator.
2. **Session layer** (pure): write `session.ts`, `game-socket.ts`, `session-lifecycle.ts`. Test headlessly if desired (no React involved).
3. **State shape**: add `status` slice to store; update types; update `deriveBlobWarsState` if needed.
4. **React hooks**: write the thin hooks. Each is ~5–10 lines.
5. **Components**: refactor `App`, `Board`, `Tile`, `MatchDetails` to use hooks + new prop shapes. Add `React.memo`.
6. **Delete**: `use-game-socket.ts`, `board-store/hooks.ts`.
7. **Verify**: run a match, read `perfLog` output. Expect `tileRenders ≈ 2 × changed`.

Steps 1–3 can land independently; 4–6 need to go together.

## Verification criteria

Existing `perfLog` instrumentation is the scoreboard. After refactor, on the 50×40 grid:

- Steady-state simulating tick: `tileRenders ≤ 2 × changed` (~100–340 in dev, ~50–170 in prod).
- Between ticks (idle, no WS messages): zero tile renders.
- `wsRecv → pipeStart` gap: dominated by store work (~1 ms) rather than React commit; should drop by at least 10×.
- Turn transitions during placing: all tiles re-render once (expected — `canPlant` flipped).

## Non-goals

- Canvas rendering. Kept as a known escape hatch for future mechanics that flip large fractions of the grid per tick, but out of scope here. The store/session split is designed to survive that swap (canvas render loop would subscribe to the same store).
- Performance micro-tuning of `computeTileData`, `deriveBlobWarsState`, or the per-tile diff loop. They're already <2 ms combined; not the bottleneck.
- Removing `perfLog` instrumentation. It stays; we'll want it again next time.
