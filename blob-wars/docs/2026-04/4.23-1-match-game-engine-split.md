# Match ↔ GameEngine split

Design note for refactoring `server/src/game/match.ts` into a thin orchestrator over a pure-ish `GameEngine`. Not implemented yet.

## Motivation

`match.ts` currently does four things in one class:

1. Socket I/O (send/receive, error messages, matchStarted/stateUpdate/matchEnded)
2. Timer / tick scheduling
3. Turn & phase state machine
4. Placement validation + simulation (expansion) logic

(1) and (2) are transport/lifecycle. (3) and (4) are game rules. They're entangled in one class, so:

- Rules can't be exercised without a WebSocket.
- Adding a new rule (new phase, new placement constraint, scoring variant) touches transport code.
- Error handling is a side effect (`sendError`) rather than a return value, which is harder to test and harder to localize later.

## Classification of current members

| Member | Category |
|---|---|
| `board`, `seedsRemaining`, `phase`, `currentTurn`, `tickNumber` | game state |
| `applyPlacement`, `expandSeeds`, `violatesExclusion`, `determineWinner`, `beginSimulation` | game rules |
| `id`, `startedAt`, `clientIds`, `timer`, `send`, `isConnected`, `onEnded`, `ended` | orchestration |
| `handleClientMessage`, `handleDisconnect`, `tick`, `finish`, `start`, `sendToAll`, `sendError`, `getSeatByClientId` | orchestration |
| `getSnapshot` | straddles — reads game state AND transport state (`connected`, `serverTimeMs`) |

Two observations worth naming:

- **"`tick`" is misleading.** It's only a simulation step — placing phase has zero ticks. `GameEngine` owns `tickNumber` (because `tile.plantedTick`/`lastGrowthTick` record it); `Match` owns the *timer* that drives the step.
- **Phase transitions are already a readable signal.** `placing → simulating` happens when both players hit 0 seeds. `* → ended` happens when the board fills. `Match` can poll `engine.phase` after each mutation instead of wiring event callbacks.

## The split

### `server/src/game/engine.ts` (new, ~180 lines)

Owns board, phase, turn, seeds, tickNumber. No sockets, no timers, no identity, no `Date.now()`.

```ts
export type PlaceResult =
  | { ok: true }
  | { ok: false; code: string; message: string };

export interface GameStateSnapshot {
  tick: number;
  phase: MatchPhase;
  currentTurn: PlayerSeat | null;
  board: BoardState;
  seats: Record<PlayerSeat, { occupiedTiles: number; seedsRemaining: number }>;
}

export class GameEngine {
  get phase(): MatchPhase;

  placeSeed(seat: PlayerSeat, x: number, y: number): PlaceResult;
  step(): void;          // no-op unless phase === "simulating"
  markEnded(): void;     // idempotent; used by Match on disconnect + finish

  getState(): GameStateSnapshot;
  determineWinner(): MatchWinner;
}
```

All rule checks live here: `wrong_phase`, `not_your_turn`, out-of-bounds, non-integer, wall, occupied, radius. `violatesExclusion` becomes a private helper or module-local function inside `engine.ts`.

### `server/src/game/match.ts` (trimmed to ~130 lines)

Transport + orchestration only. No rule logic.

```ts
handleClientMessage(clientId, message) {
  if (this.ended) return;
  const seat = this.seatOf(clientId);
  if (!seat || message.type === "ping") return;

  const result = this.engine.placeSeed(seat, message.x, message.y);
  if (!result.ok) { this.sendError(seat, result.code, result.message); return; }

  this.broadcastState();
  if (this.engine.phase === "simulating" && !this.timer) this.startSimulationTimer();
}

private tick() {
  if (this.ended) return;
  this.engine.step();
  this.broadcastState();
  if (this.engine.phase === "ended") this.finish("boardFull", this.engine.determineWinner());
}

private snapshot(): MatchSnapshot {
  const gs = this.engine.getState();
  return {
    matchId: this.id,
    serverTimeMs: Date.now() - this.startedAt,
    tickIntervalMs: TICK_INTERVAL_MS,
    growthEveryTicks: GROWTH_EVERY_TICKS,
    tick: gs.tick, phase: gs.phase, currentTurn: gs.currentTurn, board: gs.board,
    players: {
      player1: { connected: this.isConnected("player1"), ...gs.seats.player1 },
      player2: { connected: this.isConnected("player2"), ...gs.seats.player2 },
    },
  };
}
```

`board.ts` and `blob.ts` stay untouched — they're primitives the engine composes.

## Key design decisions

- **Engine owns `tickNumber`, not Match.** It's recorded on every tile via `plantedTick`/`lastGrowthTick`. Match owns the *timer* that calls `step()`.

- **All rule validation moves into the engine, including bounds + integer checks.** Rule of thumb: malformed wire messages are Match's problem (and `parseClientMessage` already rejects most of that); anything from "parsed successfully" onward is the engine's domain. Integer-ness is ambiguous — putting it in the engine keeps all "valid move?" logic in one place.

- **Match polls `engine.phase` after each mutation.** Simpler than event callbacks or richer return shapes. `"simulating"` → start timer. `"ended"` → run `finish()`.

- **`PlaceResult` carries `{ code, message }` as strings.** The codes already travel over the wire as strings — Engine producing them directly means one source of truth. Upgrade to a discriminated union is cheap if it ever bites us.

- **Engine never knows about disconnects.** Disconnect is a transport event. Match calls `engine.markEnded()` to lock the engine, then sends `matchEnded` with `reason: "disconnect"`. `markEnded` is idempotent and also called from `finish()`.

- **`getState()` returns pure game state; Match wraps it into a `MatchSnapshot`.** Match adds `matchId`, `serverTimeMs`, the two timing constants, and per-seat `connected` status. Engine stays free of identity and wall-clock time — big win for tests.

- **Rule constants (`GROWTH_EVERY_TICKS`, `SEED_EXCLUSION_RADIUS`, `STARTING_SEEDS`) stay in `@shared/protocol`.** The client needs them (e.g. for future exclusion-halo rendering), so they're wire-level config, not engine-internal.

- **File name is `engine.ts`, class is `GameEngine`.** `engine` pairs naturally with `match`: Match orchestrates, GameEngine enforces rules. `GameEngine` over bare `Engine` because `this.engine = new GameEngine()` reads clearer at construction sites.

Match becomes a pure forwarder: `if (!result.ok) sendError(seat, result.code, result.message)`.

## Testability payoff

The point of the split. With GameEngine isolated, tests look like:

```ts
const e = new GameEngine();
expect(e.placeSeed("player2", 10, 10)).toEqual({
  ok: false, code: "not_your_turn", message: expect.any(String),
});
expect(e.placeSeed("player1", 10, 10)).toEqual({ ok: true });
expect(e.phase).toBe("placing");
// ...exhaust seeds, expect phase === "simulating"
```

No WebSocket, no timer, no fake transport. Rule changes get unit tests for free.

## Open questions

- **Constructor config override.** Should `new GameEngine(opts?: { width?; height?; startingSeeds?; exclusionRadius? })` be supported? Not needed for correctness, but makes it trivial to test on small boards (e.g. 5×5 with radius 1). Low cost, high future payoff.
- **Early win detection.** If a player's seeds go to 0 mid-simulation and the opponent clearly dominates, we still wait for board-fill. Out of scope here but worth a note for later.
- **Error codes as discriminated union.** Current plan uses stringly-typed codes. A `type PlaceError = { code: "too_close_to_seed"; radius: number } | ...` would let Match localize messages without string inspection. Defer until there's a reason.
- **Does `start()` stay on Match?** Yes — it sends `matchStarted`, which is a transport-layer lifecycle event. Engine's constructor handles initial state.
