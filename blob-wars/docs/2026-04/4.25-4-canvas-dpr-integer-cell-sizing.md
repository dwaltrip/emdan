# Canvas DPR and integer device-pixel cell sizing

A self-contained explanation of why a uniform grid drawn on a canvas needs its cells to be integer device pixels, why the obvious approaches don't get you there, and the layout pattern that does. Written generically; the motivating context is a board-renderer-style grid (large tile count, alpha overlays, mixed monitor configurations), but the principle applies to any canvas drawing a uniform grid of small repeating elements.

## TL;DR

A grid of small tiles drawn on a canvas will have visible seams between tiles unless each tile occupies an integer number of **device** pixels. The naive approaches ("fit the canvas to its CSS box," or "scale the context by DPR so I can draw in CSS units") both fail at fractional DPR (1.5×, 1.25×, 1.75× — common on Windows display scaling and on any browser zoom level). The right pattern is to fix cell size in device pixels first, derive backing-store size from that, derive CSS size from the backing-store size, and accept a small gutter where the canvas doesn't quite fill its container.

## Three coordinate systems

Canvas drawing involves three different pixel concepts, and getting the wrong one to be "the integer" is what causes seams.

- **CSS pixels.** What `style.width: 600px` and `event.clientX` use. The unit you usually think in.
- **Device pixels.** What the physical screen has. The actual rasterization grid the GPU eventually paints.
- **Canvas backing-store pixels.** `canvas.width` and `canvas.height` (the *attributes*, not the styles). The internal drawing surface the 2D context rasterizes into.

The relationships:

```
device_px       = css_px × devicePixelRatio
backing_store   = canvas.width × canvas.height (in its own units)
displayed_size  = canvas.style.width × canvas.style.height (in CSS px)
```

The browser maps the backing store onto the CSS-sized box (linear scaling), and the OS maps CSS pixels onto device pixels (also linear). If `canvas.width === canvas.style.width × dpr` exactly, one backing pixel maps to exactly one device pixel — crisp. If not, the browser interpolates, and you see blur.

## DPR is not always integer

Common DPR values:

- `1.0` — standard monitor.
- `2.0` — Mac retina, most modern phones.
- `3.0` — some high-density phones.
- `1.5, 1.25, 1.75, 2.5` — Windows display scaling (125%, 150%, 175%), browser zoom, dragging a window between monitors with different scaling. **These are where the failure modes live.**

You can't decide "we'll only support DPR ∈ {1, 2, 3}" — Ctrl-+ in any browser produces fractional DPR on the fly, and Windows users routinely run at 125% or 150% by default.

## What goes wrong: the two naive approaches

### Naive 1: backing store equals CSS size

```ts
canvas.width = 600;
canvas.style.width = '600px';
const cellSize = 600 / GRID_WIDTH;  // CSS units
ctx.fillRect(x * cellSize, y * cellSize, cellSize, cellSize);
```

On a retina display (DPR=2), the backing store is 600 px but the device has 1200. The browser stretches the 600-pixel image across 1200 device pixels — every drawn pixel becomes ~2 device pixels, the whole canvas looks blurry. Bad.

### Naive 2: scale the backing store by DPR, draw in CSS units

The standard "DPR-aware" recipe:

```ts
canvas.width = containerCssPx * dpr;
canvas.style.width = containerCssPx + 'px';
ctx.scale(dpr, dpr);
const cellSize = containerCssPx / GRID_WIDTH;  // CSS units
ctx.fillRect(x * cellSize, y * cellSize, cellSize, cellSize);
```

Crisp on DPR=2 (cell becomes a clean integer number of device px). But work it out at fractional DPR. With `containerCssPx = 600`, `GRID_WIDTH = 120`, `dpr = 1.5`:

- backing store = 900 device px
- cellSize = 5 CSS px = **7.5 device px**

Each tile is 7.5 device pixels wide. The rasterizer can only land on integer device-pixel boundaries, so it has to do *something* with the half-pixel. What you see:

- Tile 0 occupies device columns 0–7 (integer).
- Tile 1 starts at 7.5 — its left edge is anti-aliased across columns 7 and 8, blending with tile 0's right edge.
- Tile 2 starts at 15 (integer again).
- Tile 3 starts at 22.5 — anti-aliased again.
- …

The result: visible seams between every other tile, irregular widths, and on dense grids a faint shimmer/moiré pattern across the whole board. `image-rendering: crisp-edges` doesn't save you — the damage is done at draw time, not at display time. Subpixel positioning is fundamentally lossy when the renderer is integer-only.

This is what bites on Windows machines and on any user who has zoomed in/out.

## The right pattern: cell size first, in device pixels, integer

Reverse the dependency. Instead of "container size determines CSS cell size determines device cell size (whatever falls out)," compute cell size in **device pixels as an integer**, and let the rest follow.

```ts
function applyLayout(canvas: HTMLCanvasElement, container: HTMLElement) {
  const dpr = window.devicePixelRatio || 1;
  const containerCssPx = container.getBoundingClientRect().width;

  // 1. Cell size in DEVICE pixels, integer. Avoids seams at any DPR.
  const cellDevicePx = Math.floor((containerCssPx * dpr) / GRID_WIDTH);

  // 2. Backing-store size = integer × integer. Total board is exactly cells × cell size.
  const backingSize = cellDevicePx * GRID_WIDTH;

  // 3. CSS size derived from backing size. May be fractional CSS px (browser handles it).
  //    Container space ≥ cssSize; the difference is unused gutter (centered).
  const cssSize = backingSize / dpr;

  canvas.width        = backingSize;        // attribute, in device px
  canvas.height       = backingSize;
  canvas.style.width  = `${cssSize}px`;     // style, in CSS px
  canvas.style.height = `${cssSize}px`;

  // All draw calls operate in DEVICE pixels (unscaled context).
  // No ctx.setTransform(dpr, 0, 0, dpr, 0, 0) needed.
}
```

Worked example, same numbers as before (`containerCssPx = 600, GRID_WIDTH = 120, dpr = 1.5`):

- `cellDevicePx = floor(600 × 1.5 / 120) = floor(7.5) = 7` — every tile is exactly 7 device pixels.
- `backingSize  = 7 × 120 = 840` device px.
- `cssSize      = 840 / 1.5 = 560` CSS px.

The canvas is now 560 CSS px in a 600 CSS px container. Roughly 40 CSS px of unused space — centered, this becomes a thin gutter on each side. Visually invisible; pixel-wise every tile is identical and seam-free.

Drawing happens in device pixels directly, no scale call:

```ts
ctx.fillRect(
  coord.x * cellDevicePx,
  coord.y * cellDevicePx,
  cellDevicePx,
  cellDevicePx,
);
```

Simpler, faster, no rounding surprises hidden inside the context's transform.

The trade-off is *"canvas exactly fills its container"* in exchange for *"every tile is pixel-perfect."* For dense grids of small repeating elements, the latter wins easily — the gutter is at most `dpr - 1` device pixels per side, the seams would show up across thousands of cells.

## Hit-testing is independent

Pointer events arrive in CSS pixels (`event.clientX / clientY`), and you can do hit-testing math in CSS units regardless of how the drawing surface is laid out. The cell can be a *fractional* CSS-px width here and the math still produces correct integer tile coords:

```ts
function eventToCoord(canvas: HTMLCanvasElement, event: PointerEvent): Coord {
  const rect = canvas.getBoundingClientRect();
  const cssX = event.clientX - rect.left;
  const cssY = event.clientY - rect.top;
  const cellCssPx = rect.width / GRID_WIDTH;     // fractional, fine
  return {
    x: Math.floor(cssX / cellCssPx),
    y: Math.floor(cssY / cellCssPx),
  };
}
```

Pull `rect.width` fresh on each event rather than caching `cellCssPx` — keeps the math correct across any later layout shift, and it's cheap.

## Re-running layout: when and why

The layout numbers depend on `containerCssPx` and `dpr`. Both can change at runtime; the layout function must re-run (and the canvas redraw) when either does.

**Container size changes** — window resize, sidebar toggle, any layout shift that changes the available CSS width. Use a `ResizeObserver` on the parent container.

```ts
const observer = new ResizeObserver(() => {
  applyLayout(canvas, container);
  markDirty();   // schedule a redraw via the renderer's normal rAF path
});
observer.observe(container);
```

**DPR changes** — browser zoom (Ctrl-+/-), dragging the window between monitors with different scaling, OS-level display-scaling changes. Use a `matchMedia` listener parameterized by the *current* DPR:

```ts
function listenForDprChange(onChange: () => void) {
  const dpr = window.devicePixelRatio;
  const mql = window.matchMedia(`(resolution: ${dpr}dppx)`);
  mql.addEventListener('change', () => {
    onChange();
    listenForDprChange(onChange);  // re-arm at the new DPR
  }, { once: true });
}
```

The re-arm step is necessary because the media query is parameterized by the DPR you registered at — it only fires when DPR moves *away from* that value. After the change, you need a new query at the new DPR.

Both triggers should run `applyLayout` and then mark the renderer dirty so the next rAF redraws the full board with the new dimensions.

## Why this matters more for grid renderers than for typical canvas apps

Most canvas demos draw a few large shapes — a fractional pixel here or there is invisible because anti-aliasing on a single shape's edge looks like an intended soft edge. Grid renderers are different:

- **Many small repeating elements.** Thousands of tiles means thousands of chances to see a seam. The seams are highly visible because they cross the whole board in straight lines, and the human eye picks up regular patterns.
- **Adjacent same-color regions.** When tile N and tile N+1 share an owner, the seam between them should be invisible — they're conceptually one continuous color region. With sub-pixel positioning, that "join" gets anti-aliased into a faint dark line, and the player's blob looks like it has visible internal grid lines.
- **Stacked alpha overlays.** Exclusion zones, hover tints, and similar effects are alpha-composited on top of base color. Any anti-aliasing artifact at tile boundaries gets amplified by the second layer's alpha, making seams that were faint in the base layer become obvious.
- **Mixed user environments.** Anything visible to real users will be viewed at DPR ∈ {1, 1.25, 1.5, 1.75, 2, 2.5, 3} across the user base. You don't get to pick.

Get the layout pattern right once in a single function, never think about it again. The cost of the right approach is identical to the cost of the wrong approach, but only one of them is correct everywhere.

## Architecture notes (specific to the renderer we're building toward)

A few specifics that drop out of this for a multi-layer canvas renderer (tile layer + indicator layer + entity layer composing onto a shared surface):

- **All layers must use the same `cellDevicePx`.** If the tile layer draws in device pixels but the entity/indicator layers do their own DPR scaling, layers won't register. Compute `cellDevicePx` in the layout module, expose it to all draw passes.
- **Offscreen tile buffer matches the same backing-store size.** If the tile layer caches into an `OffscreenCanvas`, that offscreen surface needs `width = height = backingSize` exactly, and gets blitted 1:1 onto the main canvas. Any scaling at blit time defeats the integer-cell guarantee.
- **Entity positions in grid units, converted at draw time.** Entities (effects, projectiles, etc.) carry coords in grid space (possibly fractional — a projectile mid-flight at `{ x: 4.3, y: 7.8 }`). The draw function multiplies by `cellDevicePx` to get device coords. This keeps entity data resolution-independent and means a re-layout (resize, DPR change) doesn't invalidate any in-flight entity state.
- **Indicator pass also draws in device pixels.** The hover ring, exclusion-radius preview, etc. all use `cellDevicePx` for sizing. Borders should be drawn with integer device-pixel widths (e.g., `lineWidth = Math.max(1, Math.round(dpr))`) for the same reason — fractional stroke widths get anti-aliased into fuzzy lines.

The layout module is a small file that owns all of this:

```
layout.ts
  applyLayout(canvas, container) → { cellDevicePx, backingSize, cssSize }
  eventToCoord(canvas, event)    → Coord
  observeContainer(container, onChange)
  observeDpr(onChange)
```

Every other module reads `cellDevicePx` from layout state. Nothing else does its own DPR math.
