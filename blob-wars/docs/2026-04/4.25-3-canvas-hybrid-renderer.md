# Canvas hybrid renderer: two-layer architecture

## Overview

The canvas renderer is structured as two composable layers drawing to a shared surface. The **tile layer** handles the grid — a pure projection of store state, redrawn on store changes. The **entity layer** handles everything else — effects, animations, and any future free-positioned visuals that aren't locked to the grid. Both layers are raw canvas2D today, but the separation creates a seam that lets them evolve independently.

This design avoids the complexity of canvas frameworks (PixiJS, Konva, Fabric) while preserving a clean migration path toward one if the entity layer grows complex enough to justify it. The tile layer will likely never need a framework — it's a uniform grid loop, too regular and too tightly coupled to the domain model for a generic sprite system to add value.

## Key ideas

**Tile layer: store-driven, grid-locked, stateless per frame.** Every visual element maps 1:1 to a grid cell. The draw logic is a pure function of current store state — no memory of previous frames, no continuous position, no interpolation. It redraws when the store says something changed. This is the "B1" pattern from the canvas exploration doc: one subscription, dirty flag, full-board redraw.

**Entity layer: time-driven, free-positioned, lifecycle-managed.** Things here have their own lifetime independent of the tick rate. They animate continuously between ticks, they can exist at fractional grid positions, and they expire after a duration. Today this means capture flares. Tomorrow it could mean seed projectiles, floating score text, blob pulse rings, growth-direction indicators.

**Plain data, not objects.** Entities are described as plain typed objects with a `kind` discriminant. Drawing is a separate function that switches on `kind`. This keeps the entity data renderer-agnostic — if the draw backend changes, only the draw functions change; the entity lifecycle, creation triggers, and data shapes stay the same.

**Positions in game-space.** Entity coordinates are in grid units, not pixels. The draw functions convert to device pixels at render time. This makes entity data resolution-independent and decouples it from layout/DPR concerns.

**Cached tile buffer.** When only entities are animating (no store changes), redrawing 14k+ tiles as "background" every frame at 60fps is wasteful. An `OffscreenCanvas` caches the tile layer; entity frames blit the cached image and draw entities on top. The tile buffer only re-renders on store dirty. This is the practical equivalent of layered canvases (Option C from the exploration doc) without stacking DOM elements.

## Tile layer

A pure loop over the grid, reading from the store's tile cache:

```ts
function drawTiles(ctx: CanvasRenderingContext2D, store, theme) {
  for (let y = 0; y < height; y++) {
    for (let x = 0; x < width; x++) {
      drawTile(ctx, x, y, store.getTileData({ x, y }), theme);
    }
  }
}
```

`drawTile` handles the visual cases: empty tile fill, wall fill + stroke, owner color fill, origin-seed dot (fill + arc), exclusion zone (base fill + alpha overlay), hover border. Each case is a few canvas draw calls — `fillRect`, `arc`, `strokeRect`, `globalAlpha`. No sprites, no textures, no scene graph.

The tile layer redraws only when the store version bumps (dirty flag set by the store subscription). At 14k tiles this is ~3–5ms. At 40k tiles (200×200 board), probably ~10–15ms — still within budget even at higher tick rates. If it becomes a bottleneck on larger boards, the path is B3 (store exposes a dirty-coord set, renderer only redraws changed tiles), not a framework.

## Entity layer

### Data model

Entities are plain objects with a discriminated union on `kind`:

```ts
interface CaptureFlare {
  kind: 'capture';
  coord: Coord;                       // grid-locked
  startTime: number;                  // performance.now()
  duration: number;                   // ms
  color: string;
}

interface SeedProjectile {
  kind: 'projectile';
  from: { x: number; y: number };     // continuous position (grid units)
  to: { x: number; y: number };
  startTime: number;
  duration: number;
  color: string;
}

interface FloatingText {
  kind: 'text';
  position: { x: number; y: number };
  text: string;
  startTime: number;
  duration: number;
  velocity: { x: number; y: number }; // drift direction (grid units/ms)
}

interface BlobPulse {
  kind: 'pulse';
  center: { x: number; y: number };
  radius: number;                     // grid units
  startTime: number;
  duration: number;
}

type Entity = CaptureFlare | SeedProjectile | FloatingText | BlobPulse;
```

New entity types are added by extending the union and adding a case to the draw function. No base class, no inheritance, no registration.

### Lifecycle

Entities live in a flat array. Each frame, compute progress `t` (0→1), remove expired entries, draw the rest:

```ts
const entities: Entity[] = [];

function updateAndDrawEntities(ctx, now: number) {
  for (let i = entities.length - 1; i >= 0; i--) {
    const e = entities[i];
    const t = (now - e.startTime) / e.duration;

    if (t >= 1) {
      entities.splice(i, 1);
      continue;
    }

    drawEntity(ctx, e, t);
  }
}
```

Reverse iteration so splicing doesn't skip entries. Simple, no allocations per frame.

### Drawing

A switch on `kind`, each case issuing a few canvas calls:

```ts
function drawEntity(ctx, entity: Entity, t: number) {
  switch (entity.kind) {
    case 'capture': {
      // expanding ring that fades out
      const alpha = 1 - t;
      const radius = t * cellDevicePx * 0.8;
      ctx.globalAlpha = alpha;
      ctx.beginPath();
      ctx.arc(centerX, centerY, radius, 0, Math.PI * 2);
      ctx.strokeStyle = entity.color;
      ctx.stroke();
      ctx.globalAlpha = 1;
      break;
    }
    case 'projectile': {
      // lerp from→to
      const x = entity.from.x + (entity.to.x - entity.from.x) * t;
      const y = entity.from.y + (entity.to.y - entity.from.y) * t;
      // draw circle at (x, y) in grid coords, converted to pixels
      break;
    }
    // ...etc
  }
}
```

### Triggering effects

Effects are triggered by detecting state changes during the tile redraw pass. The renderer maintains a shadow map of the previous tick's relevant state (e.g., tile owner). On each store-driven redraw, compare previous and current to classify the change:

```ts
const prevOwner = new Map<CoordKey, PlayerSeat | null>();

// inside drawTiles loop, before drawing each tile:
const prev = prevOwner.get(key);
if (prev != null && tile.owner != null && prev !== tile.owner) {
  startEffect(key, 'capture', tile.owner);
}
prevOwner.set(key, tile.owner);
```

This is renderer-side change classification — it works for anything deducible from before/after state (capture, initial growth, loss). Effects that require knowing *why* a change happened (which blob caused a capture, spread vs. special ability) would need upstream support from the store or protocol. That's a separate decision and shouldn't be needed for v1 effects.

## Frame loop

The frame function composes the two layers:

```ts
const tileBuffer = new OffscreenCanvas(backingSize, backingSize);
const tileCtx = tileBuffer.getContext('2d')!;

let dirty = false;
let rafId: number | null = null;

store.subscribe(() => {
  dirty = true;
  scheduleFrame();
});

function scheduleFrame() {
  if (rafId === null) {
    rafId = requestAnimationFrame(frame);
  }
}

function frame(now: number) {
  rafId = null;

  // tile layer: redraw to buffer only on store changes
  if (dirty) {
    drawTiles(tileCtx, store, theme);
    dirty = false;
  }

  // composite: blit tile buffer, then draw entities on top
  ctx.drawImage(tileBuffer, 0, 0);
  updateAndDrawEntities(ctx, now);

  // keep loop alive while entities are animating
  if (entities.length > 0) {
    rafId = requestAnimationFrame(frame);
  }
}
```

When no entities are active and the store hasn't changed, no rAF is scheduled — zero CPU. When entities are in flight, the loop runs at 60fps but only redraws the entity layer; the tile buffer is blitted without re-rendering. When the store changes, the tile buffer is rebuilt and entities draw on top.

## Why not a canvas framework

Canvas libraries (PixiJS, Konva, Fabric) model the world as a scene graph of visual primitives — sprites with position, tint, alpha, texture. The library owns the objects, tracks dirty state, and manages the render loop. This creates an impedance mismatch with the existing architecture where the store is the single source of truth and the renderer is a stateless projection.

Using a framework for the tile layer means maintaining two parallel representations of 14k+ tiles — the store's tile cache and the library's sprite tree — synchronized every tick. That's overhead with no payoff when the visual mapping is "owner → color, exclusion → alpha overlay."

The entity layer is closer to what frameworks are designed for, but at the current scale (a handful of concurrent effects with simple draw calls), the framework abstraction costs more than it saves.

## Transition path to a framework

If the entity layer grows complex enough — dozens of simultaneous sprites with textures, rotation, blend modes, particle systems — the seam between tile and entity layers is where a framework would slot in.

The transition would look like:

1. Bring in PixiJS (or similar) for the entity layer only.
2. Keep the tile layer as raw canvas, either on a separate canvas underneath or rendered to a `PIXI.Graphics` / `PIXI.Sprite` from an offscreen buffer.
3. Entity data structures stay the same (plain objects with `kind`). Instead of calling `ctx.arc(...)` in `drawEntity`, create/update Pixi display objects from the entity data.
4. Store, tile renderer, entity lifecycle logic, frame scheduling — all unchanged.

**What preserves this option now:**
- Entities as plain data, not class instances with draw methods.
- Tile drawing and entity drawing as separate function calls, not interleaved.
- Positions in game-space coordinates, converted to pixels only at draw time.
- Store completely decoupled from rendering.

These are all good code organization regardless of whether a framework ever arrives.
