# Board-store port + blob-wars POC

Session summary: ported the per-tile board-store architecture from `generals-v2` into `emdan` and scaffolded blob-wars's store layer on top of it.

## Goal

Re-use the architecture behind generals-v2's board rendering for new grid-based games. Daniel and a friend may build several such games (blob-wars first). The architecture's distinguishing feature — per-tile cache + per-tile subscriptions so a single tile change doesn't re-render the whole grid — is the actual value being ported.

**Non-goal: building a "GridGameKit" library upfront.** With N=0 actual games, the right abstraction isn't knowable yet. Defer extraction until a second consumer exists; in the meantime, design game #1 so that extraction is a mechanical refactor rather than a redesign.

## Source material

The architecture is lifted from `generals-v2/apps/frontend/src/domains/games/board-store/`. Three layers:

1. **Generic vanilla store** (`lib/create-store.ts`) — derive→onChange ordering guarantee, version counter for `useSyncExternalStore`, mutate-in-place actions. ~100 lines, no React, no game refs. Genuinely portable.
2. **Per-tile orchestrator** (`board-store.ts`) — wraps the generic store with a tile cache + per-tile subscriptions. The `runPipeline` function iterates coords, computes fresh `TileData` per cell, diffs against cache, notifies only changed tiles' subscribers.
3. **React adapters** (`hooks.ts`) — `useTileData(store, coord)` and `useBoardState(store)` via `useSyncExternalStore`. ~28 lines.

Game-specific filling (rewritten for blob-wars): `types.ts`, `tile-data.ts` (equality), `tile-derived-state.ts` (per-tile derivation), `derived.ts` (global derived).

## Key architectural decisions

### Don't pre-extract a shared library
Reasoning: until game #2 exists, you'd be guessing at coord shape (square vs hex), grid topology (fixed vs dynamic), update model (action vs animation). Wrong-shape extraction is harder to undo than to defer. The architecture already cleanly separates skeleton (~150 lines) from game-specific filling (~200 lines) — extraction will be mechanical at the right time.

### Cache key format: `string` ("x,y"), aliased as `CoordKey`
Switching to numeric (`y * width + x`) is a one-line change to the alias. Hex coords would need a different scheme entirely — that's a future-game decision.

### Iteration wrapped in local one-liner
`runPipeline` calls `iterateCoords(state, cb)` rather than inlining the for-loop. Costs nothing now; lets a future extraction promote `iterateCoords` to a config param without touching `runPipeline`'s body. This is the seam that's most prone to leaking state-shape coupling — wrapping it preempts that.

### Growth direction is per-tile, not per-blob
A tile in a blob renders its own growth indicator based on its specific neighbor situation. Means a single blob can show positive growth on one border and negative on another. Closer to what's visually useful than a single blob-wide value.

### Cooldown semantics
`lastPlacementTick = currentTick` at the moment of placement; `tick` increments after. Result: place at tick T, can place again at tick T+5 (with `PLACEMENT_COOLDOWN = 5`).

### Re-entrancy guard skipped
The TODO in `create-store.ts` is theoretical — React's `useSyncExternalStore` queues renders rather than synchronously dispatching. The only way to hit the bug is a non-React subscriber that directly calls an action. Don't write those.

### `useBoardState` uses version-as-snapshot
The hook reads `store.version` (a number) for tearing detection but returns `store.state` directly for actual data. Works because actions are synchronous and `mutate` mutates state in place (no new object identity to diff). Fine for game-style updates; revisit if React Suspense/transitions enter the picture.

## Game design decisions for blob-wars

These resolve ambiguities in the original blob-wars pitch. All are POC interpretations — easy to revise.

| Question | Decision |
|---|---|
| What is a "blob"? | A connected region of friendly tiles (seeds + grown). Not a single tile. |
| Strength | Count of seeds in the blob's connected region. Grown tiles don't count. |
| Turn structure | Simultaneous-ish (option B): both players input placements (or skip) into `pendingPlacements`, then `commitTurn()` resolves both + ticks growth. No `currentPlayerIndex`. |
| Placement rule | Any empty tile. No adjacency-to-friendly requirement. |
| Cooldown | 5 ticks between placements per player. `PLACEMENT_COOLDOWN` constant in `types.ts`. |
| "Push back" | Weaker blob loses one border tile per tick to stronger neighbor. Symmetric with positive growth. |

### State shape (defined in `types.ts`)

- **Source state** (`BlobWarsSourceState`): `width`, `height`, `tiles[y][x]`, `tick`, `players`, `pendingPlacements`, `status`, `winner`
- **UI state** (`UIState`): `hoveredCoord`, `activeInputPlayerId` (hot-seat)
- **Derived state** (`DerivedState`): `blobs: Map<BlobId, Blob>`, `tileToBlobId: Map<CoordKey, BlobId>`, `ticksUntilPlacementByPlayer: Map<PlayerId, number>`
- **TileData** (per-tile rendering): `coord`, `ownerPlayerId`, `type`, `blobStrength`, `growthDirection`, `isPendingPlacement`, `pendingPlacementByPlayerId`, `isPlaceableForActiveInputPlayer`, `isHovered`

### Derived computation

`deriveBlobWarsState` runs every action via the createStore lifecycle. Two parts:
1. **Connected-component flood fill** over all owned tiles → builds `blobs` and `tileToBlobId`. DFS via stack pop.
2. **Per-player cooldown computation** from `currentTick - lastPlacementTick`.

O(grid size) per action. Fine for POC. Dirty-region tracking can be added later by passing a changed-coords set into `runPipeline` instead of iterating all.

## Seam-visibility moves applied

From the discussion of "make seams visible without cementing them":

| Move | Status |
|---|---|
| External coupling wrapped in local one-liner (`iterateCoords`) | ✅ |
| `CoordKey` type alias for cache key format | ✅ |
| Top-of-file contract comment in `board-store.ts` listing skeleton-vs-filling and assumptions | ✅ |
| No game-specific convenience methods on the wrapper | ✅ |
| Public API not narrowed (wrapper exposes everything from createStore) | ⚠️ Acceptable for POC |
| Contract tests not written | ⚠️ Skipped for speed |

The four architectural assumptions documented in `board-store.ts`:
- tiles addressable by `Coord` with stable serialization
- state changes via discrete actions, not continuous animation
- per-tile rendering is independent (no cross-tile sprite stitching)
- grid size fixed for the lifetime of a session

## Stubs and open work

### Growth resolution in `commitTurn`
Marked TODO in `actions.ts`. Needs game-design input:
- Does a +1 blob always claim one tile per tick, or only sometimes?
- How does a +1 blob pick which border tile to claim?
- What if multiple +1 blobs contend for the same tile?
- Does a -1 blob lose a specific border tile, or pick at random?

### Build tooling
No `package.json`, no `tsconfig.json`, no React installed. Nothing compiles yet. `hooks.ts` will fail to resolve `react` until dependencies are installed. Vite + React + TS is the path of least resistance and matches `generals-v2`.

### UI components
Store has everything needed to render. No `<Board>`, `<Tile>` components exist. Build them using `useTileData` per cell, store passed as prop.

### Contract tests
Two-three quick tests for: `tilesEqual` correctness, per-tile sub firing only on changes, reset behavior. Worth adding before relying on per-tile diff in earnest.

## Decision log: things rejected

- **Pre-extracting a shared `GridGameKit` library** — see "Don't pre-extract" above.
- **Game of Life as the POC instead of blob-wars** — considered as a contrived simplest game. Rejected once Daniel offered blob-wars as a real game #1 to validate against.
- **Strict-alternating turn structure (option A)** — Daniel chose simultaneous-ish (option B). Means no `currentPlayerIndex`, just `pendingPlacements` map + `commitTurn`.
- **Adjacency-required placement rule** — rejected for POC simplicity. Easy to add a constraint later.
- **Branded ID types** — skipped for POC simplicity. `PlayerId` is a plain `string` alias.
