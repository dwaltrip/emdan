# UI perf refactor: follow-up findings

Companion to `4.23-2-ui-perf-refactor.md`. The refactor itself shipped earlier this week; this note records what the verification testing on 2026-04-24 turned up.

The headline target — per-tick render cost on large grids — worked. Most of this note isn't about that; it's about the multi-second "freezes" the first verification run surfaced, which sent us on a long chase that ended in *the freezes weren't real*. They were artifacts of the perf instrumentation itself, compounded by DevTools-attached overhead. This document captures the verification numbers, what the freezes actually were, and the perf-log cleanup that resulted.

## Verification: the refactor's headline targets

On a 120×120 = 14,400-tile grid, simulating phase steady-state, focused window, DevTools closed, perfDebug on (test run `tmp/perf-match-1777078404696-*` — filenames use UTC, so the `2026-04-25` stamps are from 2026-04-24 evening local time):

| metric | p50 | p99 | max |
|---|---|---|---|
| `wsRecv→paint` | 23 ms | 32 ms | 37 ms |
| `commit` (React) | 6 ms | 11 ms | 17 ms |
| `tileRenders / changed` ratio | 2.0× | 2.0× | 2.0× |

97 simulating ticks, 0 over 50ms. Targets in §"Verification criteria" of the original doc met.

Visible-but-unfocused window stays clean too: p99 = 31ms, max ~40ms (excluding the placing→simulating boundary tick, see below). Chrome doesn't meaningfully throttle visible-but-unfocused windows.

## What the original "9-second freezes" actually were

The first verification run (afternoon of 2026-04-24, before the perf-log cleanup) showed apparent multi-second freezes. The console log captured a 9-second `[frame] drift` followed by 9 ticks that all flushed at the same wall-clock timestamp, with React commits running normally during the freeze (so the main thread *wasn't* blocked — only paint was). We chased this through several theories. The actual answer is a stack of measurement artifacts:

1. **Chrome DevTools Elements panel on a 14k-node DOM.** Clicking the Elements tab synchronously walks/serializes the entire DOM. With 14,400 tiles plus their wrappers, this defers the next composite/paint pass for seconds. JS keeps running (commits, message handlers); rAF doesn't fire until paint resumes. The "9 ticks burst at one wall-clock timestamp" pattern is `console.log` being rAF-gated — `paint` is the closing event for the recorder, emitted via `requestAnimationFrame`, so logs queue silently and burst when paint resumes. We now believe one Elements click accounts for most of the multi-second freezes in the original log.

2. **`console.warn`-in-rAF feedback loop.** The frame-drift monitor (`startFrameDriftMonitor` in `perf-log.ts`) was calling `console.warn` on every drift > 10ms. In DevTools, `console.warn` is not free; emitting one extends the next frame, which trips the threshold again, which emits another warning. Once started, the loop sustains itself and amplifies any underlying jank. Same pattern for the longtask observer.

3. **DevTools console buffer growth.** Chrome retains console output even when DevTools is closed; the buffer is exposed when DevTools opens. Volume in the original test was high enough (per-tick `console.log` + per-frame-drift `console.warn` + ambient `console.log` in `MatchDetails`) to add real overhead via that buffer.

4. **React `<Profiler>` always-on.** Wrapping `<Board>` in `<Profiler>` with an `onRender` callback adds non-trivial commit-time overhead in dev builds. Magnified by attached DevTools (see "Aside" below).

None of these were app bugs. They were instrumentation/dev-tools costs that compounded and looked like an app problem.

## Perf-log changes

`ui/src/lib/perf-log.ts` was reworked along three axes: stop being a noise source itself, persist structured data offline so we don't depend on DevTools console scrollback, and disappear entirely when nobody asked for it.

- **No more rAF-driven `console.warn`.** Frame-drift and longtask events now write only to the in-memory record buffer. The per-tick `console.log` (1Hz) stays — it's low-volume and useful for live eyeballing.
- **Structured record buffer.** Each tick, longtask, frame drift, and focus/blur/visibility event is captured as a typed `PerfRecord`. Tick records carry `phase`, `matchId`, `seat`, `visibilityState`, and `hasFocus` so analysis can slice by context. Buffer capped at 1000 records, warns once on overflow.
- **JSON dump, not copy-paste.** The buffer is serialized to a downloadable JSON file automatically on `matchEnded` and `beforeunload`, plus `window.__perfDump()` from DevTools as an escape hatch. Filename is `perf-<matchId>-<seat>-<isoStamp>.json` so two side-by-side clients produce distinguishable artifacts.
- **All gated on `?perfDebug`.** Without the URL flag, every entry point (`perfLog.event`, `timed`, `rAFEvent`, `bumpTileRender`) is a no-op, observers don't install, and the `<Profiler>` wrapper around `<Board>` is omitted entirely. Default dev has zero instrumentation overhead. With `?perfDebug` on, a one-line `[perfLog] enabled` log on startup confirms it.

Analysis script: `tmp-scripts/analyze-perf-dump.ts` — quantiles, top drifts, simulating outliers, focus/blur during simulating, side-by-side comparison. Run as `pnpm tsx tmp-scripts/analyze-perf-dump.ts <p1.json> [p2.json]`.

## Aside: dev-mode measurement traps

Some numbers worth knowing for future investigations.

- **DevTools open ~3× the Profiler-measured commit time.** Same code, same grid, same focus state: placing-phase commits go from ~140ms (DevTools closed) to ~340ms (DevTools open). React DevTools' profiling hook attaches additional measurement when the panel is open. Close DevTools when measuring placing.
- **Elements panel is a foot-gun on 14k+ DOM.** Will hit the "page is not responding" dialog. Same root cause that motivates canvas as the §"Non-goals" escape hatch — adds a *DX* argument on top of the per-tick render argument.
- **Per-tick `console.log` is rAF-gated.** Because `perfLog.rAFEvent('paint', tick)` is the closing event, `formatTickLine` only fires inside an rAF callback. If rAF is delayed (paint paused, tab backgrounded, Elements walk pending), tick logs queue silently and burst on resume. Useful diagnostic signal: a wall-clock burst of identical timestamps means rAF was paused for that interval.

## Known remaining costs (not freezes)

Both anticipated by the original design doc; both deferred.

1. **Placing-phase commits scale with grid size.** Each placement during placing flips `canPlant` for the receiving client. `canPlant` is passed as a prop to every Tile, so the flip invalidates `React.memo` for all 14,400 tiles. ~140ms commit per placement on a 120×120 grid. Noticeable but not freeze-class.

2. **Placing→simulating boundary tick.** Same canPlant-flip cost shows up as a one-shot 350ms commit on whichever player held canPlant=true at the moment simulating started. Visible as the lone outlier in player 2's simulating quantiles in every test run.

Both have the same fix if we ever want it: stop passing `canPlant` as a Tile prop; have Tile read it via `useSyncExternalStore`. Then `React.memo` survives the flip and only the actually-changed tile re-renders. ~half-day of work.

## Future signal

If multi-second freezes ever return:

- Check `tileRenders` for the freeze tick: if it equals `2 × full grid` the canPlant-prop issue is the cause and the Tile prop fix above is the answer.
- Check whether `console.log` ticks burst at a single wall-clock timestamp: yes ⇒ rAF was paused; rule out Elements panel before chasing app code.
- Run with DevTools fully closed and `?perfDebug` first to get an uncontaminated baseline; compare.
