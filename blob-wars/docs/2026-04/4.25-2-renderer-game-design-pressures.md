# Renderer architecture: game design pressures

How potential game design changes stress the renderer and the store, and what that means for architectural choices. This is the "why" behind decisions in the hybrid renderer doc — the evolving requirements that make canvas the right bet and inform how the renderer should be structured.

## Larger boards

**Impact: kills SVG, makes dirty-set optimization load-bearing.**

At 120×120 (14k tiles), full-board redraw is ~3–5ms. B1 (redraw everything on dirty) is fine. At 200×200 (40k tiles), roughly 10–15ms — still within budget but the margin is thinner, especially if the tick rate increases. At 500×500 (250k tiles), full redraw is too expensive per frame; the renderer needs to know which tiles changed and only redraw those (B3 pattern: `runPipeline` exposes a dirty-coord set).

SVG with 40k+ DOM nodes is unusable — Elements panel, memory footprint, and compositing all break down. This alone eliminates the G2 (imperative SVG) option if larger boards are on the table.

The store's `Map<CoordKey, TileData>` with string keys may also start to matter at 250k tiles. A flat typed array indexed by `y * width + x` is more cache-friendly and avoids string allocation. This is a mechanical optimization, not an architectural change — the store contract (`getTileData`, `subscribe`, `version`) stays the same.

## Mid-simulation placement

**Impact: undermines the cheap DOM fix, adds per-tile interaction state to the render path.**

Currently `canPlant` is a phase-level boolean — placing or simulating, never both. The exploration doc's "cheap DOM fix" (board-level CSS attribute gating hover) relies on this: one boolean, one attribute, no per-tile re-render.

If players can place seeds during simulation, the placeable set varies per tile and per tick. Which tiles are valid placement targets depends on exclusion zones, ownership, and possibly proximity rules that change as blobs grow. The DOM fix breaks — you can't gate 14k tiles' hover affordance with one CSS attribute when each tile's placeability is independent.

Canvas handles this naturally. The hover handler checks `store.getTileData(hoveredCoord)` to decide whether to show a placement affordance. No per-tile prop, no fan-out, no CSS class toggling. The cost is one tile lookup per pointer move, which is negligible.

This also affects the entity layer: if placement happens during simulation, you might want a "seed landing" animation that overlaps with ongoing spread animations. The two-layer architecture handles this cleanly — the tile layer shows the new seed on the next store update, and the entity layer plays the landing effect concurrently.

## Faster tick rate

**Impact: tightens the frame budget, changes the animation calculus, may push toward delta-based state updates.**

At 1Hz, the renderer has 1000ms between ticks. At 4Hz, 250ms. At 10Hz, 100ms. The tile redraw itself stays fast (~3–5ms at current board size), but the total budget for store update → pipeline → redraw → entity effects shrinks.

At higher tick rates, the rAF loop needs to coalesce multiple ticks that arrive within a single frame (16ms at 60fps). If two ticks arrive between frames, only the final state matters for the tile layer — draw once with the latest data. Entity effects need to handle this too: two rapid captures of the same tile should either queue both effects or let the newer one supersede.

Higher tick rates also change the visual motivation for animation. At 1Hz, tile state changes look choppy — tweening between states adds visual smoothness. At 4Hz+, the choppiness is less noticeable and tweening adds less value. At intermediate rates (2–3Hz) choppiness is arguably worst — fast enough to notice, slow enough to look jerky.

At the store level, faster ticks may push toward sending deltas rather than full snapshots. Sending a complete board state 10 times per second is more wire traffic and more pipeline work than sending only the tiles that changed. This would change `applySnapshot` to `applyDelta`, but the renderer contract stays the same — the store still exposes `getTileData` and `subscribe`, the renderer still redraws on dirty.

## Modest animations

**Impact: makes the frame loop continuous during effects, motivates the tile buffer cache, validates the two-layer split.**

Any animation means the rAF loop runs at 60fps while effects are in flight. Without the offscreen tile buffer, this means redrawing 14k tiles every frame just to provide a background for a few animated effects. The cached tile buffer (described in the hybrid renderer doc) solves this — blit the cached image, draw entities on top.

Snapshot-driven tweens (color fade on owner change, hover ring fade-in) are purely renderer-side. The renderer detects a state change via its previous-state shadow map, kicks off a tween, and interpolates on each frame. No store or protocol changes needed.

Event-driven effects (capture flash that differs from growth fade, particle burst that depends on *which blob* caused the capture) need causal information the renderer can't infer from before/after state. Two escalation levels:

1. **Renderer-side classification.** Compare previous and current `TileData` to infer the change kind (empty→owned, owned[A]→owned[B], owned→empty). Works for most visual effects. Free — no upstream changes.

2. **Store/protocol surfaces events.** `runPipeline` or the wire protocol exposes a per-tile event kind alongside the dirty set. Needed only for effects that depend on information not in `TileData` (source direction, ability type). Real cost — touches store internals or wire format.

Start with renderer-side classification. Promote to protocol-level events only when a specific visual effect demands it.

## Improved hover effects

**Impact: purely additive in canvas, exercises the entity layer for non-expiring visuals.**

Exclusion-radius preview on hover (show the 7-radius halo where future seeds would be excluded) is a natural fit for the entity layer, though it's a slightly different lifecycle — it exists as long as the cursor is over a placeable tile, not for a fixed duration. This suggests a small extension to the entity model: entities with `duration: Infinity` (or a separate `persistent` flag) that are removed explicitly rather than by expiration.

Growth-direction arrows on hover would work similarly — computed from tile adjacency data on pointer move, rendered as entities, removed on pointer leave.

These effects scale with the visual complexity of the effect, not with board size. Canvas handles them as additional draw calls in the entity pass. In SVG, they'd require creating/removing child elements on hover, which works but gets awkward for anything needing per-frame updates (e.g., a pulsing glow).

## Compound pressure

No single change above forces a particular architecture. But stacking two or more of them narrows the options quickly:

- **Larger boards + faster tick rate** → full-redraw becomes expensive, dirty-set (B3) is needed, SVG is eliminated.
- **Mid-sim placement + animations** → per-tile interaction state changes every tick while effects are in flight, the two-layer split is load-bearing.
- **Faster tick rate + animations** → frame budget is tight, tile buffer cache is essential, coalescing multiple ticks per frame is necessary.
- **All of the above** → canvas is the only viable renderer, the tile/entity split is structurally necessary, and the store may need delta-based updates.

This is the argument for starting with canvas (B1) and the two-layer architecture now, even though the current game is simple enough for SVG or even the current DOM renderer. The architecture should accommodate the design space you're exploring, not just today's mechanics.

## What this means for the store

Most renderer changes don't touch the store. The store's contract (`getTileData`, `subscribe`, `version`) is sufficient for all of the above. The potential store-level changes are:

- **Dirty-coord set from `runPipeline`** (B3) — mild contract addition, ~10 lines, triggered by profiling on larger boards.
- **Delta-based updates** — changes `applySnapshot` but not the renderer-facing API. Triggered by faster tick rates and wire bandwidth concerns.
- **Typed array backing** — changes the internal data structure but not the contract. Triggered by boards large enough that Map overhead matters.
- **Event/causality channel** — adds a `tileEvents` field to snapshots or a new store API. Triggered by visual effects that need to know *why* a tile changed, not just *what* changed.

None of these should be built speculatively. Each has a concrete trigger — a measured cost, a specific visual effect, a game design commitment. The current store is simple and correct; that's the right starting point for a game whose mechanics are still being explored.
