# Canvas renderer: Phase 1 design sketch

## TL;DR

Replace the DOM-based `<Board>` with a canvas renderer in `ui/src/board-canvas/`. Atomic clean refactor — no dual renderer, no feature flag, no perf-bridge with DOM. Phase 1 ships visual parity with today's board plus a hover ring and disabled-cursor affordance. Animation, the offscreen tile buffer, the exclusion-radius preview, and the entity machinery are deferred to Phase 2.

## Architecture

The renderer is structured around a handful of key concepts. Most of the rest of the doc is mechanics; this section is the conceptual frame.

**Store as source of truth, renderer as projection.** The `board-store` already owns all game state and exposes `(subscribe, getTileData, version, state, derived)` as a renderer-agnostic contract. The canvas renderer is one consumer of that contract — it subscribes once, sets a dirty flag on each notify, and projects the current state onto the canvas. No game state lives in the renderer.

**Three semantically distinct render passes** — *conceptually*, not as runtime machinery:

- **Tile pass.** Pure projection of tile state onto the grid. Stateless per frame, redraws on store change.
- **Indicator pass.** State-driven, single-instance, condition-based. Hover ring, disabled cursor; future: exclusion-radius preview, growth-direction arrows.
- **Entity pass.** Time-driven, multi-instance, TTL-based. The obvious future extension when per-instance effects arrive — capture flares, projectiles, particle bursts. Not built in Phase 1; not even a stub. (Phase 2's heartbeat pulse uses a simpler "single-timestamp + derive from store" pattern and doesn't need entity machinery.)

In code, these are just function calls in sequence inside `frame()` — no abstraction layer, no `Pass[]` registry. The split exists to keep our heads straight as we add features, not to drive runtime structure.

**Interactivity flows through a single semantic descriptor.** Given hover position + game state, the renderer computes "what does the hovered tile *mean* right now?" — a small discriminated union with cases like `placeable`, `disabled`, `inert`. The cursor, the hover ring draw call, and the click handler all consume this descriptor independently, so they stay coherent without duplicated conditionals. This is a scoped abstraction — it owns *interactivity decisions*.

**Layered modules: side-effectful vs. pure.** The renderer splits across two flavors:

- **Side-effectful** (DOM-touching, stateful): `board-canvas.tsx` (React), `renderer.ts` (frame loop, pointer handlers, init/destroy), `layout.ts` (canvas sizing, observers).
- **Pure** (no DOM, no state): `draw.ts` (canvas drawing), `theme.ts` (color constants), `coords.ts` (pixel math).

Pure modules are trivially testable, trivially portable to the other game, and can't accidentally couple to React or the DOM. Boundaries between flavors are explicit: `cellDevicePx` flows from layout (side-effectful) through renderer (state holder) into draw/coords (pure consumers) as a parameter, never imported as module state.

**React only at the boundary.** `board-canvas.tsx` is the only React in the module. Everything else is plain JS calling `init(canvas, store, session)` to mount and `destroy()` to clean up. The renderer can be lifted out of React entirely — vanilla demo, Storybook story, test harness — by calling `init`/`destroy` directly. This is what makes it portable to the other game.

**Frame loop: rAF-on-dirty.** Phase 1 has no continuous animation, so the rAF loop is one-shot — scheduled when the dirty flag is set (by store notify, pointer move, or resize), runs once, idle when nothing's pending. Phase 2 will graduate this to "continuous while animation alive" when the first animation lands.

## Goals

- Headroom for game design exploration (larger boards, faster ticks, mid-sim placement).
- Open the door for animation and richer hover effects (deferred to Phase 2).
- Eliminate the DOM cliff at 14k nodes — DevTools panel hangs, Firefox hover lag, CSS-style commit cost.
- Explore and learn about canvas rendering for games — this project doubles as a learning vehicle.
- Validate the store/session/hooks decoupling that explicitly anticipated a renderer swap.
- Build a portable foundation reusable for the other game.

## Non-goals / out-of-scope

Out of scope for the renderer swap initiative (Phase 1 and Phase 2). Each has been considered and ruled out for now; none is on a roadmap.

- Accessibility (no keyboard nav, no ARIA, no screen reader). The 14k-button DOM grid wasn't accessible to begin with; canvas removes the illusion.
- Sprite atlas pre-baked at init.
- OffscreenCanvas + Web Worker for off-main-thread rendering.
- WebGL or PixiJS / canvas frameworks.
- Light/dark theme toggle.
- CSS-var probe for theme colors (replaced by hardcoded constants in `theme.ts`).
- `import.meta.hot.dispose+accept` HMR plumbing (Fast Refresh + `useEffect` cleanup is enough).
- Dual-renderer migration mode (DOM and canvas coexisting behind a flag).
- `subscribeTile` API (deleted; future per-tile React consumers can use `subscribe` + `getTileData` + shallow check at 1–5 components).

## Deferred to Phase 2

- Animation of any kind (no fades, no pulse, no transitions).
- Offscreen tile buffer.
- Continuous rAF / "rAF-while-animation-alive" loop.
- Entity machinery (no entity array, no shadow map, no change classification).
- Exclusion-radius hover preview.
- Possibly: split dirty flags (tile-dirty vs. indicator-dirty) when the buffer arrives.

These are planned for Phase 2 specifically — the architecture in Phase 1 is shaped to accommodate them as a local addition rather than a rewrite.

## Phasing

**Phase 1 (this sketch).** The swap.

- Tile pass + indicator pass.
- Visual parity with today + hover ring + disabled cursor affordance.
- Single dirty flag, one-shot rAF, no continuous animation.
- No offscreen buffer; tile pass draws directly to the main canvas.
- No entity machinery, no shadow map.

**Phase 2 (later).** First animation arrives.

- Offscreen `OffscreenCanvas` tile buffer added; tile pass writes to it, frame blits buffer + draws indicators/animations on top.
- Frame loop graduates to "continuous rAF while animation is alive."
- Possibly split `dirty` into tile-dirty + indicator-dirty.
- First effect candidate: heartbeat pulse around blob-seeds on each tick.
  - Single-timestamp pattern: one `pulseStart: number | null`, set on every store notify, used to compute progress `t` each frame.
  - Derives "which tiles pulse" from current store state (seed tiles) — no per-instance entity array, no shadow map.
- Possibly bundled in the same phase: exclusion-radius hover preview.

Phase 1's frame loop is structured so Phase 2 lands as a local addition — no rewrite of the lifecycle, the indicator pass, or the layout module.

## Module structure

```
ui/src/board-canvas/
  board-canvas.tsx     React wrapper. Only React in this module.
  renderer.ts          init(canvas, store, session) / destroy(). Frame loop. State.
  draw.ts              Pure draw functions: drawAllTiles, drawTile, drawHoverRing.
  theme.ts             Color constants. No DOM reads.
  layout.ts            DOM-side: applyLayout, observers (resize + DPR).
  coords.ts            Pure pixel math: tileToRect, pixelToTile, dprAwareLineWidth.
```

The renderer has no React imports below `board-canvas.tsx`. The lifecycle boundary is `init(canvas, store, session)` returning `destroy()`.

### Per-file responsibilities

**`board-canvas.tsx`** — the React wrapper. ~30 lines. Owns one `<canvas>` ref and one `useEffect` that calls `init(canvas, session.store, session)` on mount and the returned `destroy()` on cleanup. Mounted in `MatchContent` with `key={matchId}` so a new match remounts cleanly. No state, no other hooks, no JSX beyond `<canvas>`. This is the only file in the module that imports React.

**`renderer.ts`** — the imperative core. Exports a single `init(canvas, store, session)` that returns a `destroy()` function. Owns all renderer state in `init`'s closure: `cellDevicePx`, `dirty` flag, `rafId`, `hoverState`, store unsubscribe, observer cleanups. Owns the frame loop (rAF-on-dirty, scheduled by store notifies and pointer events). Attaches pointer handlers (`pointermove`, `pointerleave`, `click`) directly to the canvas. Coordinates the other modules — calls `applyLayout` from `layout.ts`, `drawAllTiles` / `drawHoverRing` from `draw.ts`, `pixelToTile` from `coords.ts`. Holds the canvas-side cursor state (sets `canvas.style.cursor` based on the interaction descriptor). No React imports.

**`draw.ts`** — pure drawing functions. Each function takes a `CanvasRenderingContext2D` plus the data it needs (`store`, `theme`, `cellDevicePx`, etc.) and issues canvas calls. No state, no DOM reads beyond `ctx`, no `getComputedStyle`, no observation of anything. Phase 1 exports: `drawAllTiles(ctx, store, theme, cellDevicePx)`, `drawTile(ctx, x, y, tile, theme, cellDevicePx)`, `drawHoverRing(ctx, hoverCoord, interaction, theme, cellDevicePx)`. Phase 2 grows: `drawPulses` and friends. Trivially testable — pass a mock context, assert the draw calls.

**`theme.ts`** — plain JS color constants. No `getComputedStyle` reads, no CSS-var coupling. Values copied verbatim from current `board.css` at the time of the swap. Owns: `TILE_EMPTY_FILL`, `WALL_FILL`, `WALL_STROKE`, `ORIGIN_SEED_DOT`, `SEAT_COLORS` (per-seat base + hover border + hover glow), `EXCLUSION_TINTS` (per-source tint colors). Loses CSS-as-source-of-truth for theme; gains stable runtime access without DOM probing. Light/dark theming is a future concern that would either re-introduce `getComputedStyle` or grow a small theme-switcher module here.

**`layout.ts`** — the side-effectful "manage the canvas's relationship with the DOM" module. Owns: `applyLayout(canvas, container) → LayoutState` (computes integer device-pixel cell size, sets canvas backing/CSS dimensions, returns the resulting layout numbers), `observeContainer(container, onChange) → cleanup` (ResizeObserver wrapper), `observeDpr(onChange) → cleanup` (`matchMedia` listener with re-arm on each DPR change). Encapsulates everything that knows about the canvas-in-container DOM shape so the rest of the module can ignore it.

**`coords.ts`** — pure pixel math. No DOM, no state, no side effects. `cellDevicePx` flows in as a parameter. Owns: `tileToRect(x, y, cellDevicePx, gutter) → { px, py, size }` (tile draw rect), `tileCenter(x, y, cellDevicePx) → { cx, cy }` (for the seed-dot arc), `pixelToTile(rect, eventX, eventY, gridWidth, gridHeight) → Coord | null` (hit-testing math, with edge-clamping), `dprAwareLineWidth(dpr) → number` (`max(1, round(dpr))` for crisp strokes). The most generically reusable file — likely lifted verbatim into the other game.

## React wrapper (`board-canvas.tsx`)

- One `<canvas>` element with a ref.
- One `useEffect`: calls `init(canvas, session.store, session)`, returns its `destroy()` as cleanup.
- Mounted in `MatchContent` with `key={matchId}` for clean per-match remount.
- Props: `session`, `width`, `height` (grid dimensions).
- No other state, no other hooks.

## Lifecycle: `init(canvas, store, session)`

Strict order:

1. **Apply layout.** Compute `cellDevicePx`, set canvas backing/CSS size. Stored in init closure.
2. **Synchronous first paint.** Walk every coord; `drawTile` each. Cache is already populated by `applySnapshot`.
3. **Subscribe to store.** On notify: `dirty = true`, schedule one-shot rAF.
4. **Attach pointer handlers** to canvas: `pointermove`, `pointerleave`, `click`.
5. **Observe container size + DPR changes.** Each fires `applyLayout` + sets `dirty = true`.
6. **Return `destroy()`.**

### The `applySnapshot`-is-sync contract

Step 2's synchronous read works because `applySnapshot` runs synchronously in the WebSocket handler, populates `tileCache` via `runPipeline`, and only then notifies subscribers (including `useMatchId`, which triggers the React render that mounts us). By the time `init` runs, `tileCache` reflects the current match.

If `applySnapshot` ever becomes async (worker, microtask batch, etc.), step 2 needs to handle empty cache (paint on first notify, or await populated state). **This contract goes in a comment at the top of `renderer.ts`.**

## `destroy()` responsibilities

Explicit checklist (mirrored as a comment in `renderer.ts`):

- `cancelAnimationFrame(rafId)` and null the id.
- Call store unsubscribe.
- Remove all pointer listeners on the canvas.
- Disconnect ResizeObserver.
- Remove `matchMedia` change listener (and don't re-arm).
- Clear hover state.

## Renderer state — closure-scoped

All state lives in `init`'s closure — no module-level mutable state. Each `init` call (e.g., sequential remounts via `key={matchId}`) gets fresh state.

```ts
function init(canvas, store, session) {
  // closure-scoped state
  let cellDevicePx = 0;
  let dirty = true;
  let rafId: number | null = null;
  const hoverState: { coord: Coord | null } = { coord: null };
  // ... unsubs, observer cleanup fns, etc.

  // ... init steps ...

  return function destroy() { /* ... */ };
}
```

## Frame loop

- Single `dirty` flag (Phase 1 only — Phase 2 may split into tile-dirty + indicator-dirty when the buffer arrives).
- One-shot rAF: scheduled only when dirty. Idle when nothing's pending.
- `frame()` body:

```ts
function frame() {
  rafId = null;
  if (!dirty) return;

  const interaction = describeInteraction(hoverState.coord, store, session);
  canvas.style.cursor = interaction.kind === 'placeable' ? 'pointer' : 'default';

  drawAllTiles(ctx, store, theme, cellDevicePx);
  drawHoverRing(ctx, hoverState.coord, interaction, theme, cellDevicePx);

  dirty = false;
}
```

That's the entire frame body in Phase 1. No continuous rAF.

The full-board redraw on every dirty frame is the simple v1 choice — at 14,400 tiles it's well within budget and trivially correct. Dirty-coord tracking (renderer redraws only changed tiles, sourcing the dirty set from `runPipeline`) is a measurable optimization for later: cheap to add when it earns its keep, premature today.

## Tile pass (`draw.ts`)

`drawAllTiles(ctx, store, theme, cellDevicePx)` — nested loop over every coord, calls `drawTile` for each.

`drawTile(ctx, x, y, tile, theme, cellDevicePx)`:

- Compute the tile rect via `coords.tileToRect(x, y, cellDevicePx, gutter)`.
- **Empty tile:** `fillRect` with `theme.TILE_EMPTY_FILL`.
- **Wall:** `fillRect` with `theme.WALL_FILL`, `strokeRect` with `theme.WALL_STROKE`.
- **Owned tile (origin = spread):** `fillRect` with `theme.SEAT_COLORS[tile.owner].base`.
- **Owned tile (origin = seed):** above + centered `arc` filled `theme.ORIGIN_SEED_DOT` (~30% of cell).
- **Exclusion overlay:** if `tile.insideExclusion`, `fillRect` with `theme.EXCLUSION_TINTS[tile.exclusionSource]` over the base.

All math in device pixels. No `ctx.scale(dpr, ...)` anywhere.

**Gutter size:** `Math.round(dpr)` device pixels. Matches today's CSS `gap: 1px` visual fidelity on retina (DPR 2 → 2 device px, DPR 1 → 1 device px); on fractional DPR (1.5) gives 2 device px — still crisp. The gutter is computed once at layout time and flows from the renderer into `tileToRect` as a parameter.

## Indicator state + outputs

The hover ring + cursor are two consumers of one semantic interaction descriptor.

```ts
type TileInteraction =
  | { kind: 'none' }                              // no hover
  | { kind: 'inert' }                             // hovered, no placement available (phase or opponent)
  | { kind: 'placeable'; seat: PlayerSeat }
  | { kind: 'disabled'; reason: 'wall' | 'owned' | 'excluded' };

function describeInteraction(hoverCoord, store, session): TileInteraction {
  if (!hoverCoord) return { kind: 'none' };
  if (!session.canPlant()) return { kind: 'inert' };       // top-level phase gate
  const tile = store.getTileData(hoverCoord);
  if (tile.terrain === 'wall')   return { kind: 'disabled', reason: 'wall' };
  if (tile.owner !== null)       return { kind: 'disabled', reason: 'owned' };
  if (tile.insideExclusion)      return { kind: 'disabled', reason: 'excluded' };
  return { kind: 'placeable', seat: session.currentUser.seat };
}
```

### Outputs

- **Cursor** (renderer-side, side effect): `interaction.kind === 'placeable' ? 'pointer' : 'default'`.
- **Hover ring** (`drawHoverRing` in draw.ts, pure): if `kind === 'placeable'`, draw seat-tinted `strokeRect` + alpha-blended inset fill at `hoverCoord`. Otherwise no-op.
- **Click handler** (renderer): re-evaluate `describeInteraction` for the clicked coord; if `placeable`, call `session.plant(x, y)`. (`session.plant` short-circuits on stale state too — defense in depth.)

### Why the discriminated union

A `clickable: boolean` works for today but doesn't scale. The union forces the *taxonomy* of interactions and makes new consumers exhaustive by construction. `'inert'` and `'disabled'` are semantically distinct — "no placement is happening" vs. "placement is happening but not here" — and future UX is likely to want to branch on that.

## Layout (`layout.ts`)

DOM-touching, side-effectful. Manages the canvas's relationship with the DOM.

- `applyLayout(canvas, container, gridWidth, gridHeight) → LayoutState`:
  - `dpr = window.devicePixelRatio || 1`.
  - Pick the largest integer cell that fits both axes: `cellDevicePx = min(floor(container.clientWidth * dpr / gridWidth), floor(container.clientHeight * dpr / gridHeight))`.
  - `backingWidth = cellDevicePx * gridWidth`, `backingHeight = cellDevicePx * gridHeight`. Sets `canvas.width` / `canvas.height` (attributes).
  - `cssWidth = backingWidth / dpr`, `cssHeight = backingHeight / dpr`. Sets `canvas.style.width` / `canvas.style.height`.
  - `gutter = Math.round(dpr)` device px (returned in `LayoutState` for downstream use).
- `observeContainer(container, onChange) → () => void`: ResizeObserver wrapper; returns disconnect.
- `observeDpr(onChange) → () => void`: `matchMedia('(resolution: ${dpr}dppx)')` with re-arm on each fire; returns cleanup.

**Container styling note.** Today's `.board` uses `aspect-ratio: 1` because `GRID_WIDTH === GRID_HEIGHT`. For non-square grids the container's CSS `aspect-ratio` should match the grid (`GRID_WIDTH / GRID_HEIGHT`) — otherwise the canvas underfills one axis and you get visible gutter outside the canvas. The renderer computes layout correctly either way; the CSS just affects whether unused space appears.

## Coords (`coords.ts`)

Pure math. No DOM, no state. `cellDevicePx` flows through as a parameter.

- `tileToRect(x, y, cellDevicePx, gutter) → { px, py, size }` — tile draw rect.
- `tileCenter(x, y, cellDevicePx) → { cx, cy }` — for the seed-dot arc.
- `pixelToTile(rect, eventX, eventY, gridWidth, gridHeight) → Coord` — hit-testing math. Clamps to the last valid index on the right/bottom edge (e.g., `eventX === rect.right` exactly), so it never returns out-of-bounds. Callers can assume in-bounds because pointer events outside the canvas are filtered by `pointerleave`.
- `dprAwareLineWidth(dpr) → number` — `max(1, round(dpr))`.

`coords.ts` is the genuinely portable bit — any uniform-grid canvas renderer wants exactly these helpers.

## Theme (`theme.ts`)

Plain JS constants. No `getComputedStyle`, no DOM reads.

```ts
export const WALL_STROKE = '#111827';

  base: string;
  hoverBorder: string;
  hoverGlow: string;

export const EXCLUSION_TINTS: Record<ExclusionSource, string> = { /* ... */ };
```

Values copied verbatim from current `board.css` at the time of the swap.

## Pointer / click handling

All in `init`, attached directly to the `<canvas>`.

- `pointermove`: `hoverState.coord = pixelToTile(canvas.getBoundingClientRect(), e.clientX, e.clientY, ...)`; `dirty = true`; `scheduleFrame()`.
- `pointerleave`: `hoverState.coord = null`; `dirty = true`; `scheduleFrame()`.
- `click`: compute coord, evaluate `describeInteraction`, call `session.plant` if `placeable`.

`getBoundingClientRect` is read fresh on each pointer event — keeps math correct across any later layout shift.

## Perf instrumentation

Three changes in the swap commit:

- New `canvasDraw` event: `perfLog.timed('canvasDraw', tick, () => drawAllTiles(...))`. Scoped to the tile pass — the dominant per-tick cost.
- `paint` event emission moves from store (`board-store.ts:143`'s `rAFEvent('paint', tick)`) to renderer (emit at end of `frame()` after redraw completes).
- `perf-log.ts` updates: add `canvasDraw` field to `TickRecord` and `formatTickLine`; remove `commitMs`, `commitBase`, `pipeEnd→commit`, `commit→paint` (these came from `BoardProfiler`'s `reactCommit`, which goes away).

Key signals to watch post-swap: `canvasDraw` duration and `wsRecv → paint`.

  - The `destroy()` responsibilities checklist (mirror of the list above).
- **In `board-store.ts` near where `subscribeTile` lived:** short comment noting that future per-tile React consumers can use `subscribe` + `getTileData` + shallow check at 1–5 components, and that a dedicated per-tile sub is only worth re-introducing if many components subscribe per-tile (unlikely with canvas).

## Risks to watch during implementation

- **`applySnapshot`-is-sync contract being silently broken** by a future refactor. Symptom: blank canvas until the first store notify (frame to seconds depending on the async boundary). Mitigation: contract comment in `renderer.ts`. *NOTE/TODO in the code:* also handle the "cache is legitimately empty" case (initial connection, between matches) — likely a no-op draw of an empty grid is fine, but defer verification until the renderer is running.
- **HMR cleanup leaks.** Incomplete `destroy()` leaves stale rAF callbacks holding old store closures. Symptom: gradual jank after several edits. Mitigation: explicit responsibilities checklist, sanity-check whenever `init` grows a new resource.
- **DPR re-arm forgotten.** `matchMedia` listener is parameterized by current DPR; forgetting to re-register at the new DPR after a change means subsequent zooms don't relayout. Symptom: first browser zoom relayouts correctly, second doesn't. Mitigation: `observeDpr` re-arms internally on each fire.
- **Resize / DPR change blanks the canvas momentarily.** Setting `canvas.width = backingSize` clears canvas state. Phase 1 accepts a one-frame flash (rare events, hard to perceive). Worth flagging for awareness.

## Success criteria

Phase 1 is done when:

- **Visual parity.** The canvas board renders very similarly to today's DOM board across the visual cases (empty, wall, owned-spread, owned-seed, exclusion overlays, hover ring, disabled cursor). Side-by-side comparison via jumping commits in git.
- **No major perf regressions** in the UI or DevTools. `wsRecv → paint` should be no worse than today; the DevTools Elements panel should no longer hang on board inspection; the Firefox hover lag observed today should be gone.
- **Match remount works cleanly.** New match → fresh canvas, no stale state, no flash beyond a single frame.

These are the bars. Anything beyond — animation, exclusion preview, sprite atlas — is Phase 2 or later.
