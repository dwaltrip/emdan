# Canvas renderer: Phase 2 prep 

Updated post-Phase 1. Architecture framings now reflect what we measured, learned, and discussed; detailed implementation design (concrete frame loop changes, buffer plumbing, pulse parameters) still lands during Phase 2 itself. This doc captures the conceptual framings we've settled on and the deliverables we're targeting.

## Architecture (Phase 1 lessons applied)

Phase 2 doesn't restructure Phase 1 — it adds. Three framings make the additions easier to reason about.

### Function-of-input categories

Each thing the renderer draws falls into one of three categories, distinguished by what it's a function of:

| Category      | Signature        | Examples                          |
|---------------|------------------|-----------------------------------|
| Tile pass     | `(state)`        | tiles, walls, exclusion fill      |
| Pure effects  | `(state, input)` | hover ring, exclusion preview     |
| Time effects  | `(state, t)`     | heartbeat pulse                   |

These supersede Phase 1's three-pass framing (tile / indicator / entity). The load-bearing distinction is *time*: pure effects render the present moment as a function of (state, input) — no temporal dimension. Time effects additionally depend on how much time has elapsed since a reference point, which breaks the purity. The tile pass is also a pure projection in this sense — just the special case that's big enough to warrant caching (see below). "Indicator" from the Phase 1 doc conflated binary state-driven affordances with the broader UI concern; "pure effects" is the sharper category.

**Entities remain a future fourth category** for time-driven effects whose set of in-flight instances can't be derived from the store — capture flares fired on newly-captured tiles, projectiles. Pulse is time-driven *without* entities: one global `pulseStart` timestamp suffices because "which tiles pulse" is derived from store state (seed tiles). Entities become necessary when per-instance state with TTL-based lifecycles is unavoidable. Phase 2 doesn't need them.

### The buffer is a layering primitive

Phase 1 measured full-board redraws at <5ms — the buffer is not perf-required for the first animation. Its purpose is structural: it physically separates the static layer (tiles) from the dynamic layer (pure + time effects).

```
buffer (static)  ──► rebuilt only on store dirty
                  ──► blitted to main canvas every frame
main canvas      ──► blit + drawPureEffects + drawTimeEffects each frame
```

Without the buffer, "redraw a hover ring without redrawing 14k tiles" needs dirty-coord tracking (deferred from Phase 1). The buffer makes the static/dynamic split a thing in code, not just a concept. Perf is a secondary win; layering is the driver — and as either side grows (richer tile work, richer animation work), the win compounds.

Note: the buffer caches *only* the tile pass. Pure effects and time effects both draw fresh on top of the blitted buffer each frame. Hover changing on pointer move doesn't invalidate the buffer.

### Three slots of frame input

`frame()` reads from three slots on each invocation:

| Slot          | Lifetime                              | Lives in              | Examples                                    |
|---------------|---------------------------------------|-----------------------|---------------------------------------------|
| Game state    | long-lived; snapshot-replaced on tick | store (`board-store`) | tiles, players, phase, current turn         |
| Input state   | ephemeral; mutated by event handlers  | renderer's closure    | `hoverState` today; keyboard, drag tomorrow |
| Time progress | continuous; mutates every frame alive | renderer's closure    | `pulseStart` (when set)                     |

Three lifetimes, three different mutation triggers — but uniform treatment in `frame()`: each is read at the top, drawn from on each frame. No "animation system," no "tick manager."

**Input state is a form of state — just renderer-local.** It's a presentation concern (where is the mouse? which keys are held?), not a game concern, so it stays out of the store. The slot today has one entry (`hoverState.coord`); plausible future entries include modifier keys (`keyState.shift` for range select), drag origin (`dragState.from`), or pointer dwell time (`hoverDwell.ms`). Each new entry is another nullable field in `init`'s closure, mutated by an event handler, read by draw fns. No "input system" abstraction — the slot is a category, not a runtime thing.

The renderer's state grows accordingly: one nullable timestamp for time progress; one or more fields for input state. The existing module structure absorbs all of this.

### `scheduleFrame` / `requestRender` separation is load-bearing

Phase 1 split the rAF coalescer (`scheduleFrame`) from the public state-changed wrapper (`requestRender`) for naming clarity. It turns out to be the mechanism that unlocks continuous-rAF:

```ts
function frame() {
  // ...
  if (animating) scheduleFrame();   // schedule next frame WITHOUT marking dirty
}
```

`scheduleFrame()` from inside `frame()` self-perpetuates the loop without claiming "something changed." Folding the two functions in Phase 1 would have required either lying with a fake dirty bit at the end of `frame()` or restructuring the loop. The separation cost nothing in Phase 1 and unlocks continuous-rAF as a one-line addition.

## What Phase 2 brings

- **First animation: heartbeat pulse around blob-seeds on each tick.** A time effect using the single-timestamp pattern, not entity machinery — one `pulseStart: number | null` set on store notify; progress `t` computed each frame. Derives "which tiles pulse" from current store state (no per-instance entity array, no shadow map).

- **Offscreen `OffscreenCanvas` tile buffer.** The static/dynamic layering primitive (see Architecture above). Tile pass writes to the buffer on store dirty; frame body blits buffer to main canvas, then draws pure effects + time effects on top. Phase 1's <5ms full-redraw baseline confirms this isn't a perf forcing function — it's a cleanliness win that helps perf as a side effect.

- **Continuous rAF while animation is alive.** Phase 1 is rAF-on-dirty (idle when nothing's pending). Phase 2 graduates to "stays alive while a pulse / future animation is in flight" via `if (animating) scheduleFrame()` at the end of `frame()` — see Architecture.

- **Possibly: split dirty flags.** Tile-dirty (store changed → rebuild buffer) vs. indicator-dirty (hover changed → re-composite without rebuilding buffer). With the buffer in place this becomes the natural shape; without it, single dirty is fine. Decide during implementation.

- **Possibly bundled: exclusion-radius hover preview.** The visible-feature win — a circular halo on hovered placeable tiles during placing phase. As a pure effect (binary, hover-conditional), it slots in as a single new draw fn alongside `drawHoverRing`. May land in this phase or as a stand-alone follow-up; consider shipping it first since it doesn't depend on the buffer or continuous-rAF work.

## What Phase 1 taught

- **Full-board redraw cost**: <5ms for 14,400 tiles, well under the 16ms frame budget. The buffer is therefore not perf-required for the first animation — see Architecture for the layering rationale that still motivates it.
- **Visual baseline**: established. Hover ring, cursor affordance, exclusion overlays all match the DOM version. Animation can land on top without first re-establishing what "good" looks like.
- **Deferred edge cases**:
  - **Empty-cache first paint**: implicitly handled — `getTileData` falls back to `createDefaultTileData` for cache misses, so the very-first paint draws an empty grid uniformly without special-casing.
  - **Indicator-only changes**: still trigger a full-board redraw on each dirty frame. Phase 1 evidence (<5ms) says this is fine for now; revisit if indicator-only paths get expensive after the buffer lands.

## Out of scope (still, in Phase 2)

- **Entity pass / entity array / shadow map.** Only built when a per-instance effect (capture flare, projectile) needs per-instance state. Pulse uses the single-timestamp pattern instead.
- **`Indicator` runtime abstraction / `Pass[]` registry.** The three-pass framing is conceptual scaffolding, not code structure. Per-indicator file extraction is allowed if N=2 evidence (exclusion preview) calls for it; a runtime "indicator system" is not.
- **Per-tile dirty-coord tracking.** Phase 1 deferred this; Phase 2 still defers. Buffer rebuild stays full-board on dirty.
- **Multiple concurrent animations / animation manager.** One pulse at a time, single timestamp. If a second time effect lands, revisit the pattern then; don't pre-build a manager for hypothetical needs.
- Everything else in Phase 1's "Non-goals / out-of-scope" list.

## Open questions (resolved during Phase 2 detailed design)

- **Indicator file-extraction (N=2 question).** If exclusion preview lands as just another draw fn (no own input state), keep `draw.ts` as a flat catalog. If it grows its own input state (debounce timer for hover-stable detection, fade-in animation), pull both hover and exclusion into per-indicator input modules. Decide once we see exclusion preview's actual shape.
- Pulse visual details: duration, color, expansion curve, behavior when ticks arrive faster than the pulse can finish.
- Exclusion preview: snap or fade? Same color as hover ring or distinct?
- Buffer dirty management: one flag for both buffer rebuild + frame composite, or two?
